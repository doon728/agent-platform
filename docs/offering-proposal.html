The Offering

This offering produces a Healthcare AI Agent Accelerator — a curated set of reusable, production-grade artifacts that our delivery teams and developers use to build healthcare AI agents faster. The accelerator is not a product we sell and not a finished deployment. It is a starting point — battle-tested patterns, working container code, infrastructure templates, backend adapters, agent scaffolding, and compliance controls — that compresses the path from "idea" to "agents running in a customer's AWS account" from months into weeks.

How to think about it: The accelerator is to a healthcare AI deployment what a proven framework plus reference architecture is to a software project. It removes the work of inventing the platform from scratch. Every customer deployment tailors it — adding integrations, configuring agents, wiring compliance — but always starts from a working, versioned baseline instead of an empty repo.

Lens C — Customer Business ROI

For the business owner, ROI is measured in specific workflow outcomes. The accelerator targets the highest-value healthcare payer workflows — where multi-step reasoning, integration with claims and care management systems, and compliance controls are all required at once. Two flagship use cases are walked through end to end below, followed by a shorter list of supported workflows.

Flagship 1 — Pre-Call Assessment for Care Management

The problem. A care manager making outreach calls to high-risk members spends 20–40 minutes per call on preparation — pulling claims history, reviewing the care plan, scanning member notes, identifying open gaps. Most of that time is information assembly, not clinical work.

What the agent does. Before the call, the agent assembles a concise assessment for the care manager: member demographics, recent claims and utilization, open care gaps, recent care plan notes, and flags worth discussing. It pulls from claims, the care management platform, the care plan repository, and a shared knowledge base of payer guidelines. A human-in-the-loop gate lets the care manager accept, edit, or reject the summary before it is written back to the case.

ROI levers:

  • Prep time per call reduced (target: from ~30 min to ~5 min)
  • Outreach volume per care manager per day increases
  • Conversations are higher quality because more context is available up front
  • Care plan note quality improves — the agent drafts, the human edits and approves
  • Less rework from missed or incomplete information

Flagship 2 — Prior Authorization Denial Remediation

The problem. Prior authorization denials drive significant cost on both sides. When a PA is denied, the remediation workflow involves reviewing the denial reason, assembling missing clinical documentation, checking policy rules, drafting an appeal, and resubmitting — often done manually across multiple systems and often missing deadlines.

What the agent does. On denial, the agent classifies the denial reason, identifies the documentation gap, pulls the relevant clinical policy criteria, assembles an appeal package with the required evidence, and routes it to a clinical reviewer. A human-in-the-loop gate holds before submission. Status updates flow back to the originating provider and the member.

ROI levers:

  • Appeal turnaround time reduced
  • Appeal approval rate improves — packages are more complete and on-policy
  • Manual effort per denial reduced
  • Leakage from missed appeal deadlines reduced
  • Provider and member satisfaction improves with a faster, cleaner process

Reference Architecture — Request Flow Through Four Containers

The platform is structured as four independent containers, each with a distinct responsibility. A single agent request moves through them in sequence — from entry point, through reasoning, through tool execution, with configuration and governance served by the control plane. This separation is what makes the accelerator deployable per line of business, per AWS account, without duplicating intelligence.

Agent request →
Container 1
Agent Runtime
Thin shell, per LOB, per AWS account
  • Pre-graph — load domain scope, auth/authz, hydrate session
  • In-graph — delegates reasoning to C2 (no LLM logic here)
  • Post-graph — HITL gate evaluation, PHI masking, response delivery
  • Self-registers with C4 on boot using LOB + AWS account ID
ECS / Fargate · VPC · IAM
Container 2

  • RAG retrieval — semantic, keyword, hybrid, graph per agent config

What flows through: C1 receives the request, hydrates session and policy, and calls C2. C2 runs the reasoning graph — invoking C3 whenever a tool call is required and reading/writing memory along the way. C3 executes the tool, returns structured output, and C2 continues reasoning. When the graph completes, C1 applies HITL gates and PHI masking before the response reaches the UI. C4 does not sit in the request path — it serves configuration to C1/C2/C3 at load time and receives traces and metrics after each run.

Why four containers: C1 is thin so it is cheap to deploy per line of business per AWS account — each LOB gets its own data boundary and blast radius without duplicating reasoning logic. C2 and C3 run on AWS AgentCore so we inherit managed scaling, native memory, and tool integration. C4 is shared across the practice — one control plane manages all agents across all customer deployments. Tool Gateway vs MCP: the Tool Gateway is the execution engine (internal control, per-agent access policy, schema validation). MCP is the interoperability interface — an optional facade for third-party clients. They are not competing; they serve different purposes at different layers.

Customer Delivery Engagement — How the Accelerator Is Applied

A typical customer delivery engagement runs 12–16 weeks across 5 phases. The accelerator is the starting point — delivery teams fork an engagement copy from the master accelerator repo, tailor it to the customer's systems and workflows, and deploy it into the customer's AWS account. Timeline and team composition flex based on number of agents, integration complexity, and customer environment. Structure below is indicative — final scope, staffing, and pricing are determined per engagement.

Four Modes of Accelerator Use

Not every customer will accept every part of the accelerator into their environment. That is expected — healthcare payers have strict policies on what runs in their AWS accounts, what code they accept from outside, and which open-source dependencies they approve. The accelerator is structured so that it delivers value across a spectrum — from full deployment down to reference-only use — without losing its core benefit to the delivery team.

Mode What Gets Deployed to Customer What the Accelerator Still Provides
1. Full deployment The entire platform (all four containers, IaC, agents, admin UI) runs inside the customer's AWS account Fastest path; every pattern applied directly; full platform handed over at end of engagement
2. Partial / hybrid Some components accepted (for example, agent runtime and templates), others rebuilt to the customer's stack standards Delivery team still skips months of platform plumbing; customer retains control over their stack choices; accelerator fills the gaps
3. Reference only Nothing deployed; customer builds their own version from scratch Our architecture, patterns, and reasoning strategies are the blueprint their team follows; our working code is the reference they compare against
4. Enablement only No deployment, no code sharing Delivery team uses the accelerator to design, demo, and train faster; customer sees a working agent during design phase; no team starts from whiteboard
The key point: the value of building this accelerator does not depend on any single customer accepting full deployment. Every engagement captures some subset of the value — and in the worst case (mode 4), the practice still skips the whiteboard phase and delivers faster.

How the Engagement Begins

Every engagement opens with a focused discovery motion. The goal in the first two weeks is to agree on what we are building, why it matters, and how we will measure success.

  • Pick two anchor workflows. One member-facing (e.g., pre-call assessment, care gap outreach) and one back-office (e.g., prior authorization remediation, claims triage). Each must be painful, frequent, and measurable.
  • Define success metrics up front. Cycle-time reduction, manual-effort reduction, appeal approval rate, cost-per-case, audit-exception rate, care manager throughput — whichever apply to the chosen workflows.
  • Map system-of-record boundaries. Claims core, care management platform, prior-auth system, member database, data warehouse, IAM, logging.
  • Identify tool candidates. Read and write actions the agent needs exposed through the Tool Gateway to automate the workflow end to end.
  • Confirm compliance posture. HIPAA controls, BAA coverage, data-residency expectations, audit-trail requirements.

The output of discovery is a signed-off workflow map, success metrics, and an AWS deployment plan — enough to commit the build phase scope.

If they require their own penetration test on top (many payers do), they start from a known-clean baseline. Approval cycles typically compress from weeks or months to days.

How the Engagement Team Works in the Repo

The delivery team does not start from a blank repo. At kickoff, the Platform Team creates an engagement repo — a fork of the master accelerator repo, dedicated to this customer. All assigned developers clone this single shared repo and work in it together for the duration of the engagement.

  • Engagement repo is access-controlled — only the assigned team has visibility
  • Forking to personal namespaces is disabled; all work happens on branches of the shared repo
  • main is branch-protected; changes go through pull requests with required reviews
  • Infra changes in the iac/ folder get their own reviewer requirements via CODEOWNERS
  • When the practice ships a new accelerator release (v1.1, v1.2, …), the engagement architect can pull upstream improvements at a sprint boundary

For the full repo governance model — including how the master accelerator repo is maintained and how individual developers access it for exploration — see Section 9.

Pattern Handoff at Engagement End

At the end of the engagement, the customer receives more than code. They inherit the entire Git governance model — the repo structure, branch protection rules, CI/CD templates, CODEOWNERS, exploration sandbox pattern, and the update path for pulling future releases of the accelerator. The customer leaves the engagement with a running agent platform and a way to operate it sustainably.

What the Customer Inherits

  • Master copy of the engagement repo in their own Git organization — their versioned gold copy
  • Branch protection and PR workflow already configured for their own change governance
  • Exploration sandbox pattern — their developers can POC new agents safely, off the master
  • Upgrade path — they can pull future releases of the accelerator on their own cadence
  • CI/CD templates they own and can extend for their pipelines

Why the Handoff Model Matters

  • The customer is not left with a black-box deployment — they know how to operate it
  • Creates a natural continuing relationship — they pull our accelerator upgrades as we release them
  • Reduces the customer objection of "what happens after you leave?"
  • Turns accelerator release cadence into a recurring customer touchpoint

Deployment Topology Options

Production deployment supports three topologies depending on customer preference — chosen during Phase 1 based on data residency, existing AWS estate, and LOB structure.

Per-LOB Multi-Account (Default)

  • C1 Agent Runtime deployed once per line of business, each in its own AWS account
  • C2 and C3 run on shared AgentCore — accessed via PrivateLink
  • C4 Control Plane is a single instance — registers all LOB deployments
  • Blast radius contained to a single LOB; data never crosses LOB boundary
  • Preferred for large health systems with multiple business units

Single-Account & Air-Gapped Variants

  • Single-account — all four containers in one AWS account. Simpler for smaller deployments or single-LOB customers.
  • Fully customer-hosted — C2 and C3 deployed as containers in the customer account instead of AgentCore. For strict data residency or air-gapped environments.
  • Both variants packaged as CDK stacks — identical agent code, only the IaC target changes
HIPAA compliance is built into every stage: KMS encryption at rest, TLS 1.2+ in transit, PHI masking in C1 post-graph, CloudTrail audit logging, IAM least-privilege per service, VPC-only data plane, AWS Config rules for drift detection. All AWS services used are HIPAA-eligible under BAA. Compliance documentation is delivered as part of the engagement, not added after the fact.
Section 9

Repo Governance & Developer Onboarding

The accelerator is distributed through a three-tier repository model. One master repo owned by the practice, one engagement repo per customer, and individual exploration repos for developers who want to learn or prototype. Each tier has a different governance posture — deliberately.

For a walkthrough of what exists today as a working prototype — the patterns proven, the reference agent running end-to-end — see Section 12 at the end of this document.

Repo Type Access Purpose
Master / Template Platform team writes; practice reads Source of truth. Versioned releases (v1.0, v1.1, …). Marked as template repo on the Git platform.
Engagement Assigned engagement team only (10–50 developers) Customer delivery. Branch-protected, forking disabled, PR-enforced. Handed to customer at end.
Exploration Individual developer POCs, learning, experimentation. Self-service via "Use this template." No governance.

Who Enforces Access

  • Platform Team (3–5 people) — owns the master repo, controls write access, publishes releases
  • Practice PMO / Engagement Lead — requests engagement repo at kickoff, defines team, owns access lifecycle
  • Git platform permissions — Teams / Groups enforce access; branch protection enforces PR workflow
  • Developer self-service — exploration repos via "Use this template," no approval needed

How Master Updates Propagate

  • Platform team tags a new release (e.g. v1.1) and publishes release notes
  • Engagement architect decides when to pull — typically at sprint / phase boundary, not mid-sprint
  • Merge happens via git fetch upstream && git merge upstream/v1.1 → PR into engagement main
  • Engagements may stay on their starting version for stability / compliance freeze
Why this model works: The master repo is a gold copy — no one can accidentally break it. Engagement repos are access-controlled team workspaces — 50 developers on a customer engagement share one repo, not 50 independent copies. Exploration repos are disposable sandboxes — developers learn the platform without needing engagement approval. Same source, three different creation and governance paths.

Infrastructure as Code Gets Its Own Review Discipline

Platform code, agent code, and Infrastructure as Code all live in the same repository — one clone gives a team everything it needs to stand up an environment. But IaC changes (VPC, IAM, Bedrock policies, security groups, KMS keys) carry a different risk profile than application code, so the iac/ folder is governed differently inside the same repo:

  • iac/ has its own CODEOWNERS file — infrastructure changes require review by designated infra/SRE owners, not just any engagement engineer
  • Its own branch protection rules applied on top of the repo-wide protections — for example, two approvals required for production-facing changes
  • Dedicated CI pipeline for IaC: plan on PR, apply on merge after approval, with drift detection
  • Customer-specific IaC values (AWS account IDs, VPC CIDRs, DNS names, resource tags) live in engagement-specific tfvars / CDK context — same repo, different files, clearly separated from reusable modules

Single repository, but with review discipline that matches the risk profile of each folder.

Three Consumption Flows

The same master accelerator repo is consumed three different ways. Each flow below answers one simple question: "What does this actor actually do, in order?"

Flow 1 — Individual Developer (Sandbox / POC)

  1. Goes to master accelerator repo URL in the practice's Git
  2. Clicks "Use this template" → new sandbox repo in their own namespace
  3. git clone + docker-compose up — platform running locally in ~5 minutes
  4. Uses Agent Factory UI on localhost to scaffold, configure, and test agents
  5. (Optional) deploys to practice playground AgentCore — namespace-scoped, budget-capped
  6. Pulls v1.x updates when they want; sandbox repos are disposable

Flow 2 — Customer Delivery Engagement

  1. Engagement kicks off — PMO requests an engagement repo from Platform Team
  2. Platform Team forks master into engagements/customer-x-agents, adds the assigned team, enables branch protection, disables personal forks
  3. All assigned developers clone the same engagement repo — one shared workspace
  4. Team tailors platform-core/, shared-infra/, templates; adds customer-specific agents under agents/<domain>/
  5. CI/CD builds and deploys — dev → staging (practice) → prod (customer AWS account)
  6. At end of engagement, codebase is handed to customer with full IP transfer

Flow 3 — Accelerator Version Upgrade

  1. Platform Team releases v1.1 — new backends, agent templates, compliance patterns, bug fixes
  2. Release notes announced to practice and all engagement architects
  3. Each engagement architect decides when to pull — typically at a sprint or phase boundary, not mid-sprint
  4. git fetch upstream && git merge upstream/v1.1 → PR into engagement main → CI passes → deployed
  5. Customers who have received handoff pull the same release on their own cadence
Why three flows, one master: Sandbox exploration is self-service. Customer delivery is access-controlled. Version upgrade is a standard Git merge. All three operate against the same versioned master repo — no side channels, no special tooling. When we ship a new version, every engagement and every developer can pull it with one merge. A zip-based download cannot offer upgrades once extracted — the artifact is frozen. Distributing through Git keeps every engagement and every developer on a living, improvable asset.
Section 10

LLM Keys, Models & Cost Control

LLM access is structured around two principles: no static API keys on developer laptops for production workloads, and cost attributed to the engagement, agent, and developer that incurred it. The accelerator is Bedrock-native for production, with a controlled OpenAI proxy available for dev exploration only.

Primary — AWS Bedrock (Production)

  • No static keys — ECS/Fargate tasks assume IAM roles, call Bedrock directly
  • HIPAA-eligible — Claude models accessed via VPC endpoint
  • Cost isolation — per-engagement AWS account = natural cost boundary, tagged in AWS Cost Explorer
  • Auth audit — every Bedrock call logged to CloudTrail

Secondary — OpenAI Proxy (Dev / Exploration)

  • Practice-run proxy (LiteLLM or similar) — developers never see the raw OpenAI key
  • Scoped proxy tokens — per-developer, short-lived, revocable
  • Daily spend caps enforced at proxy — developer can't exceed their quota
  • Non-PHI only — proxy enforced on exploration data, blocked from engagement repos

Cost Control — Four Layers

Layer Mechanism Owner
Account AWS Budgets + SNS alerts per engagement account — monthly cap with 50% / 80% / 100% alerts Engagement Lead
Model Bedrock per-model token quota — Opus limited, Sonnet standard, Haiku high-volume Platform Team
Agent Per-agent cost metrics in Control Plane — dashboard in Agent Factory UI, traces include token count + cost per invocation Engagement Architect
Emergency Kill switch — engagement lead disables a specific agent from Agent Factory UI without redeploy Engagement Lead

Available Models (chosen per-agent in agent.yaml)

Model Use Provider / Access Relative Cost
Claude Opus 4.7 Complex reasoning — multi-hop, reflection, strategic planning agents Bedrock $$$$ (reserved for hard tasks)
Claude Sonnet 4.6 Default workhorse — chat, triage, workflow agents Bedrock $$ (standard)
Claude Haiku 4.5 Fast / cheap — classification, extraction, high-volume monitoring Bedrock $ (high-volume safe)
GPT-4o / 4o-mini Dev exploration only, non-PHI prototyping OpenAI via practice proxy $ – $$ (scoped quotas)
Model choice is configuration, not code: Each agent specifies its model in agent.yaml. Swapping from Sonnet to Haiku — or from Bedrock to the OpenAI proxy — is a config change, not a redeploy. This is what allows cost optimization after an agent is live: start on Sonnet, observe traces, move predictable steps to Haiku, keep the hard reasoning step on Opus.
Section 11

Technical Limitations & Design Guardrails

The architecture sits on top of cloud services (AWS AgentCore, Amazon Bedrock), a graph orchestration framework (LangGraph), large language models, and a regulated industry (HIPAA). Each comes with real limits the accelerator must design around. This section is not a list of weaknesses — it is the set of constraints we have intentionally designed around, and the guardrails we use to do so.

Limitation Impact If Ignored How the Design Mitigates It
AgentCore Runtime timeout — 15-minute synchronous invocation limit Long-running tools (bulk claims scans, batch summarization, external-system calls that queue) would time out mid-execution and fail the agent run Any Tool Gateway tool that may exceed 2 min runs async: tool returns a job ID immediately, the agent polls a status endpoint, and the workflow resumes on completion. Short tool calls stay synchronous for latency.
AgentCore Gateway / MCP timeout — 5-minute invocation limit on MCP tools/call Agents using the optional MCP facade hit a tighter ceiling than direct Tool Gateway calls MCP is reserved for short, discovery-style tool calls. Long-running work is routed through the direct Tool Gateway async pattern. Same async job-ID mechanism.
MCP schema constraint — JSON Schema $ref / $defs not supported by AgentCore Gateway target sync Tools registered with referenced schemas fail to sync and become invisible to MCP clients Schemas are fully resolved at registration time inside the Tool Gateway before exposure. No runtime reference lookup.
Bedrock model availability by region Not every Claude model is in every AWS region; a deployment could miss the model an agent is configured for Region selection is part of Phase 1 discovery. Per-agent agent.yaml declares a primary model plus a fallback list. Config loader validates availability at deployment.
LLM context window limits Memory, RAG results, prompt scaffolding, and tool schemas all compete for finite context; long sessions blow through budget silently Context engineering trims and prioritizes before the LLM call: summarization of older episodic memory, configurable top-K for RAG, token budgets per section. Summary memory is explicitly compressive.
LLM non-determinism Same input can produce different outputs; evaluation frameworks that assume exact match are unreliable Evaluation uses semantic matching (LLM-as-judge, structural checks, policy rubrics) — not string equality. Temperature and seed are controlled per agent in agent.yaml.
HIPAA PHI boundary — no PHI may leave the customer's VPC or reach external logs A single PHI leak in a trace, log line, or LLM prompt constitutes a compliance incident A PHI masking layer intercepts on the way out of Container 1 (responses) and on the way into Container 2 (LLM prompts). Bedrock is called through VPC endpoints only. Traces, logs, and audit entries apply the same mask before emission.
LangGraph state serialization — graph state must be JSON-serializable to cross the Container 1 ↔ Container 2 HTTP boundary Pydantic objects, callables, open connections, or tool instances in state would break the runtime split Graph state is constrained to primitives only. Rich objects are rebuilt on each side from IDs. Tool clients and LLM clients are created per-request, not held in state.
HITL pause/resume lives in Container 1 only — graph state cannot round-trip to Container 2 during approval windows Approvals can take hours or days; graph state must survive container restarts A durable HITL store in Container 1 (RDS / DynamoDB) persists the full session snapshot on every gate. Resume on approval restores state exactly; Container 2 is stateless across the gate.
Multi-tenant isolation — shared services (C2 platform services, C3 tool gateway) serve multiple customers or multiple lines of business Data or configuration from one tenant visible to another is an immediate compliance failure Per-request tenant context is required on every inbound call. Memory scopes, tool registry entries, and configuration are all keyed by tenant. No shared global state can be written across tenants.
Tool action serialization — write-class tools (note creation, status updates, external submissions) cannot be parallelized Two writes firing concurrently from the same agent can create conflicting state in downstream systems The Tool Gateway tags tools as read or write. Write tools serialize even when routing matches multiple; read tools run in parallel.
LLM cost per invocation — complex multi-step reasoning can accumulate significant cost per request Without guardrails, a single agent going wrong (reasoning loop, retries) can spike spend Per-agent cost tracking, per-tenant budgets, and a kill switch in the Control Plane. Model selection in agent.yaml allows moving predictable steps to Haiku without redeploy. See Section 10 for full cost-control mechanics.

Each item above is a real constraint, not a risk we are hoping to avoid. The accelerator's architecture exists the way it does because these constraints shaped it.

Section 12

Current Prototype — What's Built Today

Over the past several months, we have built a working prototype that demonstrates the architecture end-to-end. The prototype is not the final accelerator this offering will produce — it is the evidence that every core pattern we have designed works, every boundary holds, and every abstraction has been exercised against a real healthcare agent running real workflows. What follows is a plain-language tour of what the prototype proves today and how the platform is envisioned.

Config-Driven by Design

The platform is built so that an agent's behavior is declared, not coded. Every choice that defines how an agent thinks and acts — which reasoning strategy it follows, which memory types it uses, which tools it can call, which prompts it loads, which LLM model it runs on, how it retrieves knowledge — lives in YAML configuration. This is not a convenience; it is the architectural spine. It is what lets an operator tune, govern, and evolve agents through a UI rather than through code, and it is what lets us ship behavior changes without redeploying services.

Four Containers, One Coordinated System

The prototype runs as four independent containers that a developer can bring up on a laptop with a single command. Each container has a clear, narrow responsibility: the agent runtime acts as the thin entry point, the orchestration engine runs the reasoning graph, the tool gateway executes tools, and the control plane serves configuration, governance, and observability. The four are wired together by well-defined interfaces, not by shared state, so each can evolve independently and scale differently in production. The same four containers run locally under docker-compose and in production on AWS AgentCore and ECS — the substrate changes, the boundaries do not.

Shared Core Services

A lot of what agents need is not agent-specific — memory, retrieval, human-in-the-loop approval, observability, tool invocation. Rather than rebuilding these in every agent, the prototype places them in a shared core library that every agent links against. Memory comes as four coordinated types — short-term session state, episodic case history, semantic member facts, and running summary — behind a single interface, with swappable storage backends. Retrieval is multi-dimensional: the platform supports different retrieval strategies, different retrieval patterns, and different stages in the agent flow where retrieval runs. HITL is built with an adapter pattern so approval surfaces can plug in over time. Observability wraps every reasoning step, tool call, and memory operation. All of this is available to every agent by default.

Agents Organized as Type × Reasoning Strategy

The prototype demonstrates what we believe is the right mental model for building agents: every agent is a combination of an agent type and a reasoning strategy. Agent types describe what kind of work the agent does — conversational chat, structured summarization, classification and triage, long-running workflow, monitoring, multi-agent coordination. Reasoning strategies describe how the agent thinks — a simple linear plan, a ReAct loop, plan-then-execute, multi-hop reasoning, reflection. These combine: a chat agent using the simple strategy behaves differently from a chat agent using ReAct, and the framework is envisioned to support every sensible pairing. The prototype ships with two fully wired combinations as proof that the pattern works; the full matrix of combinations is part of what this offering will produce.

Shared Infrastructure for Every Agent

Beyond the core library, the prototype includes shared infrastructure that every agent depends on. The tool gateway is the central enforcement point for secure tool execution — schema validation, per-agent access policy, audit logging, and an optional MCP facade for third-party interoperability. The prompt management service stores prompts with full lifecycle — draft through active through deprecated — so operators can version and roll back without redeploys. These are deployable services, not libraries; every agent in a deployment uses the same instances.

A Template Offering for New Agents

When a new agent is needed — whether for a new healthcare workflow inside a customer engagement or for a developer building a proof of concept — the prototype does not start from a blank file. The platform ships with a set of templates that capture the patterns above in pre-wired form. An operator or developer picks the template that matches the target agent type and reasoning strategy, adjusts the configuration for the workflow, and gets a working agent. This is how we envision new agents being built across every engagement: through configuration, not greenfield code. The Agent Factory UI is the surface that drives this — an operator scaffolds, configures, and governs agents through a UI, and the platform writes the right templates and YAMLs behind the scenes.

Capability-Level UI for End Users

Agents do not exist in isolation — a care manager, a nurse, a claims reviewer works on a whole domain, not a single agent. The prototype demonstrates a capability-level UI pattern: a domain (for example, care management) gets its own unified interface that spans the agents within it. A nurse uses one console to navigate members, cases, and assessments; multiple agents power different panels inside that console. This is how we envision the end-user experience: capability-first, with agents composed underneath. The prototype includes a working capability UI for care management as evidence of the pattern.

Governance and Compliance in the Platform Design

Healthcare's compliance posture shapes the platform's design from the start. PHI masking runs before any response reaches a UI. HITL gates can be placed at any step an agent takes. Every reasoning step, tool call, and memory operation is traced. Prompts are versioned and lifecycle-managed. Tool access is filtered per-agent. Access control and authorization run before reasoning begins. The prototype exercises all of this together in a single agent flow — evidence that the platform's compliance posture is coherent, consistent, and practical for regulated work.

What the Prototype Proves

Taken together, the prototype shows that the architecture holds under a realistic workload. A fully wired reference agent — pre-call assessment for care management — runs end-to-end using every pattern above: the four containers, YAML-driven configuration, the agent-type-by-reasoning-strategy composition, multi-type memory, multi-dimensional retrieval, the tool gateway, prompt management, HITL gates, observability, and a capability-level nurse-facing console. Every abstraction has been exercised against realistic data. Every boundary has been tested. What this offering will produce moves the prototype from "it works" to "it ships" — hardened backends, complete Infrastructure as Code, production CI/CD, and the full catalogue of agent-type and reasoning-strategy combinations the prototype has only sampled.
