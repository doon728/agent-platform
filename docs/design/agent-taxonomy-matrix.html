Section 1

Overview — what this matrix is and how to read it

Enterprise agent deployments cluster into a small number of architectural categories, each with characteristic runtime patterns, reasoning needs, RAG applicability, HITL fit, memory shape, and tool invocation style. A single "agent platform" rarely covers all of them — real systems pick a subset and optimize for it.

This matrix does three things:

  • Taxonomy: categorizes all common enterprise agent patterns (request-response, conversational, long-running autonomous, event-driven, batch, voice/video, multi-agent, specialized).
  • Capability map: shows which reasoning strategies, RAG dimensions, HITL patterns, memory scopes, and tool patterns apply to each category.
  • AEA coverage scorecard: calls out what AEA covers today, what it covers with extensions, and what sits outside its scope.

The categorization synthesizes public industry taxonomies (Anthropic's five patterns, LangGraph design patterns, Confluent's event-driven multi-agent patterns, Bain's three-layer platform model), then maps them onto real enterprise use cases in healthcare payer, government IT, and business applications.

How to read this document. Sections 2–3 establish vocabulary. Section 4 is the heart — one dense page per category. Section 5 is the cross-cutting matrix. Section 6 is the honest scorecard of what AEA does and doesn't cover. Sections 7–8 are concrete use-case mappings. Section 9 is the build plan to close gaps.
Section 2

Industry reference taxonomies

Rather than inventing categories, AEA's taxonomy draws from four widely-cited industry references. Each captures a different slice.

2.1 Anthropic — "Building Effective Agents" (five workflow patterns)

Anthropic's 2024 paper (updated in 2026) distinguishes workflows (predefined paths orchestrating LLMs and tools) from agents (dynamically directed systems). The five canonical workflow patterns:

PatternWhat it doesWhen to use
Prompt chainingSequential LLM calls; each step processes prior output.Deterministic pipelines where steps are known.
RoutingClassify input and direct to specialized handler.Triage, intake, classification.
ParallelizationSectioning (independent subtasks) or voting (multiple runs for consensus).Speedup or reliability via redundancy.
Orchestrator-workersCentral LLM dynamically delegates to worker LLMs.Complex tasks with unpredictable subtasks.
Evaluator-optimizerIterative loop between a generator and an evaluator until quality bar is met.High-quality outputs where critique improves drafts.

Anthropic's principle: start at the lowest complexity that works; add more only when measurable improvement justifies it.

2.2 LangGraph — six design patterns (orchestrator-worker, ReAct, fan-out/fan-in, reflection, supervisor, hierarchical)

LangGraph models agents as typed-state directed graphs with conditional edges. Its six canonical design patterns overlap with Anthropic's but emphasize execution control:

  • ReAct — interleave reasoning (think) and action (tool call) in a loop.
  • Orchestrator-worker — planner decomposes; workers execute sub-tasks.
  • Fan-out / fan-in — one node triggers N parallel nodes, converges back.
  • Reflection — self-critique loop for quality improvement.
  • Supervisor — one agent routes between a pool of specialists.
  • Hierarchical — nested supervisors (multi-level delegation).

LangGraph adds first-class checkpointing and interrupt points for HITL — making it the closest mainstream framework to enterprise-grade production requirements.

2.3 Confluent — four event-driven multi-agent patterns

Confluent's 2026 architecture blueprint identifies four patterns when agents coordinate over event streams (Kafka / EventBridge):

  • Orchestrator-worker (event-driven) — orchestrator emits task events; workers consume and emit results.
  • Hierarchical agent — parent monitors a topic and spawns ephemeral child agents per event.
  • Blackboard — all agents share a single event log that acts as shared memory.
  • Market-based — agents bid on opportunity events; a coordinator assigns work.

2.4 Bain — three-layer agentic platform model

Bain's 2026 enterprise framework separates an agentic platform into three layers — useful for positioning AEA inside customer architectures:

  1. Experience layer — front-end/chat/voice interfaces, HITL review consoles, operator dashboards.
  2. Agent layer — reasoning engine, tool invocation, memory, governance — what most people mean by "the agent."
  3. Data & tool layer — knowledge bases, vector stores, MCP tool servers, external APIs, source systems.

2.5 Other emerging vocabulary (2026)

  • Guardrail agents — specialized agents monitoring other agents and blocking high-risk actions in real time (Berkeley California Management Review, March 2026).
  • Propose-commit pattern — separation of "propose an action" (stored durably, reviewed) from "commit the action" (idempotent execution).
  • Agentic RAG — specialized retrieval agents handling decomposition, validation, synthesis in parallel.
  • AgeMem / memory-as-tool — treat memory operations (store, retrieve, update, summarize, discard) as tools the agent calls.
  • Continue-as-new (Temporal) — long-running workflow pattern that restarts graph state to avoid event-history bloat.

Combining Anthropic's behavioral patterns with deployment realities (latency, concurrency, state durability, modality), enterprise agents cluster into eight top-level categories:

Important boundary — agents vs. business workflow engines. Throughout this document, references to "durable agent session state" (AgentCore FS Persistence, Temporal, Step Functions) refer to persisting the agent's own reasoning state — plan, tool results, pending HITL approval, memory — across pauses and failures. This is not a business workflow engine and does not replace the customer's Camunda / Pega / IBM BPM / ServiceNow. Those remain the customer's property. The agent participates inside a business workflow step (the WF engine calls the agent), but AEA never replaces the BPM platform. Agent types like workflow_agent refer to agent-internal multi-step plans, not business workflows.
Category vs. agent type vs. reasoning strategy vs. behavior — the distinction.
  • Category = runtime / architectural shape (how it's deployed and invoked).
  • Agent type = behavioral role within a category (chat, summary, classifier, workflow, etc.).
  • Reasoning strategy = the control-loop inside the agent (simple, ReAct, plan-execute, multi-hop, reflection).
  • Behavior = compound term describing what the agent does internally. Composed of: reasoning strategy + tool-use pattern + memory interaction + HITL interaction. Example phrases: "single-shot LLM call", "multi-step reasoning with tools", "iterative refinement with critic", "streaming turn-taking with barge-in."
One category contains many agent types; each agent type can use multiple reasoning strategies; each combination produces a distinct behavior profile. A chat_agent (type) in the Conversational (category) running ReAct (strategy) with 5 tools and session memory has a different behavior than the same chat_agent running plan-execute with 0 tools and no memory.
Section 4

Category-by-category deep dive

C1 Request-Response

Defining traits: Synchronous HTTP-style invocation. Agent receives input, produces output, returns. Latency budget seconds. Stateless or minimal session state.

Typical behavior profile: single-shot LLM call (or short ReAct loop) → structured output. Minimal tools. No memory. No HITL (except low-confidence escalation).

Orchestration types: Reasoning (C2-style) only. No runtime orchestration, no multi-agent.

Agent types in this category

  • classifier_agent — intake triage, routing decisions (e.g., "is this an appeal, PA, or claim dispute?"). Structured output.
  • retrieval_agent — pure RAG response, no tool loop. "What's the PA policy for knee MRI?"
  • summary_agent — summarize a document, case, or member record. Single-pass.
  • transform_agent — reformat / extract / redact / translate (pre-defined I/O contract).

Reasoning strategies applicable

  • simple (default) — single planner → execute → respond.
  • ReAct — only if multi-step tool use is needed.
  • Not applicable: plan-execute, multi-hop, reflection (overkill for single-shot).

RAG applicability

  • Dim 1 (stage) — pre-graph retrieval or planner-tool retrieval both common.
  • Dim 2 (source) — usually single-source (policy docs, clinical guidelines).
  • Dim 3 (extraction) — simple top-k or query decomposition for harder questions.

HITL applicability

Usually no HITL — too fast-turn. Exception: classifier with low confidence routes to human review queue (propose-commit on classification decision).

Memory applicability

  • Short-term: usually off (stateless).
  • Episodic / semantic / summary: typically off.

Tools applicability

Light — maybe 0–3 tool calls. API / MCP. No stateful sessions.

AEA architecture fit

✅ Covered today. Pre-graph → simple/ReAct → post-graph fits cleanly. summary_agent already exists; classifier_agent is a small addition (new overlay, new prompt pattern).

C2 Conversational

Defining traits: Multi-turn dialog. Session state matters. User and agent exchange messages over time. HITL may be inline (agent asks user). Latency target 1–3 seconds per turn.

Typical behavior profile: multi-step reasoning with tools → agentic RAG → memory read/write → inline HITL confirmations → streamed response. Session-scoped throughout.

Orchestration types: Reasoning (C2-style) + lightweight session manager (persisting session state between turns). No runtime workflow engine, no multi-agent.

Agent types in this category

  • chat_agent — general-purpose conversational assistant with tools + RAG.
  • copilot_agent — domain-focused assistant alongside a primary workflow (e.g., nurse copilot during call).
  • workflow_agent (interactive) — guides user through a multi-step process (intake, assessment, form-fill).
  • advisor_agent — decision-support copilot for high-judgement work (clinical Q&A with policy context).

Reasoning strategies applicable

  • simple — per-turn Q&A with tools.
  • ReAct — common default for tool-rich chat.
  • plan-execute — for multi-step user requests ("do X, then Y, then Z").
  • reflection — when quality matters more than speed (clinical draft review).

RAG applicability

  • All three dimensions active.
  • Dim 2 source selection often per-turn (what corpus this question needs).
  • Agentic RAG pattern (query decomposition → parallel retrieval → synthesis) fits here.

HITL applicability

First-class. Inline user confirmations ("did I get this right?"), gated tool calls (draft email before send), escalation to supervisor.

Memory applicability

  • Short-term: always on (session history).
  • Episodic: common (remember prior sessions with same member).
  • Semantic: common (learned facts about member, ongoing treatment plan).
  • Summary: common (rolling summaries for long sessions).

Tools applicability

Rich — API, MCP, stateful tool sessions (ongoing searches, drafts).

AEA architecture fit

✅ Covered today. This is the current reference architecture (pre-call-assessment). chat_agent overlay with simple/ReAct strategies is the primary runnable path.

C3 Long-running Autonomous

Defining traits: Runs minutes to days. Durable state. Checkpoint-resume. May pause indefinitely on HITL gate. Needs workflow-engine-like semantics.

Typical behavior profile: plan → execute phased steps with checkpoints → pause on HITL gate → resume → iterative refinement with critic → propose-commit on each non-reversible action. Heavy memory + summarization across phases.

Orchestration types: Durable agent session state (AgentCore Runtime Filesystem Persistence / Temporal-as-session-store / Step Functions-as-session-store — stores the agent's own reasoning state across pauses and failures) + reasoning (C2-style per phase). Multi-agent optional if sub-agents participate.

What this is NOT: a business workflow engine (Camunda / Pega / IBM BPM / ServiceNow). Those remain the customer's property and are untouched. The agent may be invoked from a business workflow step, but the agent doesn't replace the BPM platform. Durable session state here is purely about the agent's own reasoning state.

Agent types in this category

  • adjudication_agent — appeals, complex claims, PA with clinical review.
  • investigation_agent — fraud, waste, abuse cases with evidence collection over time.
  • outreach_agent — population-health campaigns with multi-contact sequences.
  • recertification_agent — eligibility/benefits renewals spanning weeks.
  • research_agent — longer-horizon information gathering (hours).

Reasoning strategies applicable

  • plan-execute — plan spans across durable steps.
  • multi-hop — evidence chains over external sources.
  • reflection — quality gates between phases.
  • evaluator-optimizer — iterative refinement of drafts.
  • Needs new interrupt-resume loop as first-class strategy.

RAG applicability

  • Multi-source, multi-stage retrieval common.
  • Dim 3 extraction varies per step (iterative retrieval, hypothesis-driven).

HITL applicability

Heavy. Propose-commit pattern for every non-reversible action. Durable pause for hours/days. Audit trail mandatory.

Memory applicability

  • All four scopes active.
  • Summary scope critical — long contexts must be compacted.
  • Checkpoint = durable snapshot of state + memory at each step.

Tools applicability

Extensive. External APIs, source systems, document extraction, potentially browser automation. Idempotency keys required on every tool call.

AEA architecture fit

⚠ Partial — needs extension. Current graph supports per-invocation runs. Needs:

  • Durable agent session state — checkpoint/resume mechanism (AgentCore Runtime's Filesystem Persistence, Temporal, or Step Functions used as session store). This stores the agent's own reasoning state, not a business workflow.
  • HITL service already exists — needs hardening for long-pause scenarios.
  • Continue-as-new pattern for event-history bloat.

C4 Event-Driven / Streaming

Defining traits: Agents subscribe to event topics (Kafka, EventBridge, FHIR subscriptions). Reactive. Often high concurrency. Agents may emit events for downstream consumption.

Typical behavior profile: per-event classify / enrich / score → emit decision event. Short-lived per invocation. Stateless per event (state lives in blackboard or downstream). Low-confidence routes to async review queue.

Orchestration types: Runtime (Kafka / EventBridge / SQS / FHIR subscriptions — event routing + delivery) + reasoning (C2-style per event). Multi-agent optional — Confluent's blackboard, orchestrator-worker, or market patterns may apply.

Agent types in this category

  • fraud_detection_agent — scores claim-stream events in real time.
  • intake_triage_agent — event-driven version of classifier (new document lands → agent picks up).
  • alert_agent — monitors event streams for clinical or operational thresholds.
  • change_reaction_agent — reacts to FHIR subscriptions, claim status changes, eligibility updates.

Reasoning strategies applicable

  • simple — most common; per-event classification / scoring.
  • ReAct — when event needs enrichment via tool calls.
  • Confluent's patterns apply: orchestrator-worker, hierarchical, blackboard, market-based.

RAG applicability

  • Usually enrichment-style (fetch related context for the event).
  • Dim 1 pre-graph retrieval most common.

HITL applicability

Queue-based, not inline. Agent routes low-confidence decisions to review queue; humans process asynchronously.

Memory applicability

  • Short-term: usually off (per-event stateless).
  • Episodic / semantic: key-scoped (per member / per provider).
  • Blackboard pattern: shared event log as memory.

Tools applicability

Enrichment tools (look up member, fetch claim history). Emit events as a tool output.

AEA architecture fit

⚠ Partial — needs extension. Pre-graph becomes an event-consumer adapter (Kafka / EventBridge / SQS → invoke agent). Agent graph largely unchanged. Requires:

  • Event-adapter pre-graph nodes (new).
  • Concurrency model (parallel agent instances per event batch).
  • Dead-letter + retry semantics.
  • Emit-as-output side (agent output fans back to an event bus).

C5 Batch / High-Throughput

Defining traits: Large windowed workloads — thousands to millions of units processed in a single job. Parallelization-first. Idempotent, retry-safe. Not session-based.

Typical behavior profile: fan-out units across workers → per-unit single-shot LLM call (or short per-unit ReAct if complex) → shared context cache → fan-in results → exceptions to human queue. No inline HITL. For simple classification/extraction, LangGraph is skipped entirely; for complex per-unit reasoning, a LangGraph run executes inside each worker.

Orchestration types: Runtime (Step Functions / Spark / Ray / Glue / Airflow — batch scheduler + fan-out/fan-in) + optional reasoning (C2-style per unit — used only when per-unit reasoning is complex; simple per-unit skips C2 entirely) + multi-agent fan-out/fan-in worker pool.

Agent types in this category

  • bulk_adjudication_agent — nightly claim auto-adjudication at scale.
  • bulk_extraction_agent — mass document extraction (IDP pipelines).
  • bulk_classification_agent — mass triage / categorization jobs.
  • recertification_sweep — bulk eligibility renewal checks.

Reasoning strategies applicable

  • simple — dominant (same template applied per unit).
  • Halo-style batch query optimization — DAG-level plan with cache reuse and GPU placement.
  • Fan-out / fan-in pattern for parallelization.

RAG applicability

  • Shared context (policy docs) pre-loaded and reused across batch.
  • Per-unit retrieval minimized for efficiency.

HITL applicability

Exception queue only. Units that fail confidence threshold route to human review; batch continues.

Memory applicability

  • Short-term: per-unit (transient).
  • Episodic / semantic: shared across batch (cached for reuse).

Tools applicability

Bulk-optimized tools (batch lookups, vectorized inference). Not per-unit API calls.

AEA architecture fit

⚠ Partial — needs extension. Current graph is per-invocation. Needs:

  • Batch orchestration layer (Step Functions, Spark, Ray, Glue).
  • Scatter-gather pre-graph / post-graph hooks.
  • Caching layer for shared context.
  • Cost-model-aware scheduling (Halo-style).

C6 Real-Time Voice / Video

Defining traits: Sub-500ms end-to-end latency. Streaming audio/video. Interruption (barge-in) handling. Continuous duplex operation.

Typical behavior profile: streaming turn-taking with barge-in → S2S or cascade STT→LLM→TTS → cached context only (no inline RAG) → cascade-fallback on tool calls → post-call summarization. Supervisor whisper for HITL.

Orchestration types: Media-server runtime (Pipecat / LiveKit / AgentCore Voice — manages streaming audio + interruption). Not C2-style reasoning — the voice loop is its own control architecture. No multi-agent typically (voice agent + supervisor whisper = 1.5 agents).

Agent types in this category

  • voice_agent — contact center inbound/outbound.
  • telehealth_intake_agent — voice triage before clinician join.
  • real_time_translator_agent — simultaneous translation.
  • video_observer_agent — video analysis with real-time reporting.

Reasoning strategies applicable

  • Standard reasoning strategies don't fit — latency budget is too tight.
  • Needs streaming-turn-taking strategy (new).
  • Hybrid: speech-to-speech (S2S) for low-latency small talk; cascade fallback when RAG or tools are needed.

RAG applicability

  • Pre-loaded / cached context only — no inline retrieval during speech.
  • Async RAG on tool-fallback path.

HITL applicability

Supervisor listen-in + barge-in. Real-time whispered suggestions to live agent.

Memory applicability

  • Short-term: rolling audio buffer.
  • Summary: post-call.

Tools applicability

Tool calls interrupt speech flow — used sparingly. Often cascade-fallback: S2S for conversation, switch to cascade pipeline when a tool is needed.

AEA architecture fit

❌ Doesn't fit cleanly — different architecture. Would require voice-specific runtime (Pipecat, LiveKit, or AgentCore Voice). AEA can contribute tools, RAG, memory, HITL services as a backend for the voice pipeline — but the voice loop itself is a separate product.

C7 Multi-Agent Collaborative

Defining traits: Two or more agents coordinate. Can be hierarchical (supervisor + workers), peer-to-peer (agents as equals), or debate-based (structured disagreement with consensus).

Typical behavior profile: orchestrator plans and delegates (or peers negotiate / debate / bid) → each sub-agent runs its own reasoning strategy → shared blackboard memory → guardrail agent monitors for high-risk actions → consensus or supervisor commit → final human sign-off on high-stakes outputs.

Orchestration types: Multi-agent (hierarchical / peer-to-peer / debate / market / blackboard — the primary orchestration here) + reasoning (C2-style per sub-agent) + optional runtime (Temporal / Step Functions when the collaboration runs long).

Agent types in this category

  • workflow_agent — hierarchical orchestrator with sub-agents (common).
  • panel_agent — peer agents (clinical, legal, policy) review same case; consensus required.
  • debate_agent — adversarial review (advocate vs. reviewer).
  • market_agent — agents bid on work; coordinator assigns.
  • guardrail_agent — monitors other agents, blocks high-risk actions.

Reasoning strategies applicable

  • Each sub-agent has its own reasoning strategy (mix-and-match).
  • Coordination pattern is a meta-strategy: orchestrator-worker, debate, consensus, market.
  • Needs debate / vote / consensus pattern (new).

RAG applicability

  • Each sub-agent may have its own corpus / retrieval strategy.
  • Blackboard pattern: shared context written/read by multiple agents.

HITL applicability

Guardrail agents provide autonomous oversight; human review still layered for high-stakes outcomes.

Memory applicability

  • Shared memory (blackboard) common.
  • Per-agent private memory for specialized reasoning.

Tools applicability

Each agent has its own tool allow-list. Cross-agent communication via message protocol (A2A).

AEA architecture fit

⚠ Partial — planned. workflow_agent (hierarchical) is in roadmap. Peer / debate / market patterns require new orchestrator types.

C8 Specialized / Outlier

Defining traits: Fall outside the pre-graph → reasoning → post-graph architecture. Usually need a different primary runtime or special-purpose tools.

Typical behavior profile: specialized control loop (perceive-act-observe for browser / proof-search for formal verification / open-ended multi-hop for research) → domain-specific tools → pre-approval gates on non-reversible actions → validation step after each action.

Orchestration types: Specialized runtime (browser-use / Claude Computer Use / proof engine / research harness — manages the perceive-act-observe or proof-search loop). Not C2-style reasoning. Multi-agent optional (e.g., a browser-use agent called by a supervising agent as a tool).

Agent types in this category

  • browser_use_agent — navigates web UIs, clicks, fills forms (Epic, legacy Medicaid portals, provider portals).
  • computer_use_agent — desktop automation (Claude Computer Use, OpenAI Operator).
  • formal_verification_agent — proof-carrying reasoning over medical policies or legal texts.
  • open_research_agent — open-ended literature / evidence synthesis (hours of exploration).
  • federated_agent — agents across organizations with isolated state (payer ↔ provider ↔ HIE).

Reasoning strategies applicable

  • Browser-use: perceive-act-observe loop (accessibility tree + screenshot).
  • Formal verification: proof-search (not LLM-native).
  • Open research: multi-hop + reflection + continue-as-new.

AEA architecture fit

❌ Doesn't fit cleanly — separate architecture. AEA can integrate with these as tools/services (e.g., expose a browser-use agent as a tool), but the primary runtime is not AEA's graph.

Section 5

Master capability matrix

Cross-cutting view — all eight categories, all capability dimensions, side by side.

Orchestration — unifying definition + three levels.

Orchestration = coordinating work across multiple steps or actors to produce a result. The definition does not change by category — only the scope of what's being coordinated changes. Three scope levels exist:

  • Reasoning orchestration (C2-style) — micro scope, inside one agent. Coordinates the agent's internal steps (planner → executor → responder). This is what C2 does today via LangGraph.
  • Runtime orchestration — macro scope, above the agent. Coordinates when, how, where, and how often agents run. Temporal, Step Functions, Kafka, EventBridge, Spark/Ray.
  • Multi-agent orchestration — meta scope, between agents. Coordinates multiple agents cooperating. Hierarchical, peer-to-peer, debate, market, blackboard.

Which levels apply per category:

Section 9

Control plane — runtime-agnostic across all categories

A central, often-missed point: the control plane is the most durable and portable part of AEA. Even when an agent runs on a runtime outside AEA's pre-graph → reasoning → post-graph model (voice on Pipecat, batch on Step Functions, long-running on Temporal, multi-agent on peer-to-peer orchestrators, browser-use on Claude Computer Use), the scaffolding, configuration, prompt management, governance, and fleet management still flow through one UI.

9.1 The category-agnostic design

Every agent — regardless of category — has the same control-plane needs:

  • Template — a reference pattern for the agent type (chat, classifier, voice, batch, etc.).
  • Overlay — per-instance YAML + prompts + config, layered on the template.
  • UI-driven configuration — non-developers (ops, compliance, clinical SMEs) tune agents without code.
  • Agent registry — fleet view, ownership, lifecycle, status.
  • Prompt management — versioning, evaluation, A/B, rollback.
  • Governance & policy packs — allow/deny, PHI handling, HIPAA audit patterns, FedRAMP policies.
  • Deployment wiring — IaC templates per runtime (AgentCore, Fargate, Pipecat, Temporal, Step Functions, Mosaic).
  • Observability — trace aggregation across heterogeneous runtimes.

These needs don't change when the runtime underneath changes. That's why the control plane is runtime-agnostic: it abstracts over runtimes, not inside one.

9.3 Why this is strategically important

The durable differentiation. Runtimes come and go. AgentCore Managed Harness shipped this month. Databricks Mosaic AI is in market. Pipecat, LiveKit, Temporal, Step Functions, future open-source entrants will keep emerging. Enterprises can't pick one and freeze — they need to operate agents across whatever runtime fits each use case.

A unified, category-agnostic control plane — with healthcare domain packs, prompt management, governance, fleet registry, and evaluation — is what survives all of that. The control plane is where AEA's IP compounds; the runtimes are commodity layers underneath.

  • Same operator experience for a nurse copilot (LangGraph), a contact-center voice agent (Pipecat), a nightly claims batch (Step Functions), and a long-running appeals adjudication (Temporal).
  • Customer-deployable. The control plane runs in the customer's environment, governing agents they operate.
  • Platform-agnostic framing: "AEA is a category-agnostic agent control plane with healthcare domain packs, runnable over AgentCore, Mosaic, Pipecat, Temporal, Step Functions, or any future agent runtime."

9.8 Dynamic routing — classifier in front of the agent fleet

Problem: reasoning strategy in AEA is fixed per-agent overlay (not switchable mid-run by design — keeps observability and evaluation deterministic). For a fleet of N chatbots tuned for different query shapes, we need a way to pick the right agent per query.

Pattern: a lightweight router service sits between the UI and the agent fleet. It classifies the query, picks the best-match pre-deployed agent, forwards the invocation, and streams the response back. Aligns with Anthropic's canonical "routing" pattern.

User types in Chat UI
      │
      ▼
┌────────────────────────────────────────────┐
│  Router service (new, thin)                │
│  • Reads Agent Registry (C4)               │
│  • Calls classifier LLM (Haiku — fast)     │
│  • Returns: pick agent X, confidence Y     │
└──────┬─────────────────────────────────────┘
       │  (forwards to chosen agent's /invocations)
       ▼
┌──────┴────┬──────────┬───────────┐
▼           ▼          ▼           ▼
chat_simple chat_react chat_plan  chat_reflection
(each is a separately-deployed C1, strategy fixed)
       │
       ▼
  Response streams back to UI

Why this fits AEA's 4-container model without architectural change

  • Agents unchanged — pre-graph → reasoning (C2) → post-graph intact per agent.
  • C2 orchestration unchanged — each agent still runs its own LangGraph loop with its configured strategy.
  • C3 Tool Gateway unchanged.
  • C4 Control Plane unchanged — Agent Registry already exists; router just consumes its data.
  • Additive — new thin service, not a structural container change.

Agent manifest extension (routing hints)

routing_hints:
  query_patterns: ["simple lookup", "single fact", "status of X"]
  strategy_profile: simple
  confidence_floor: 0.7
  example_queries:
    - "Status of ticket JIRA-123?"
    - "What is the owner of project X?"

The classifier LLM sees the incoming query + all agents' routing hints → returns best-match agent ID + confidence. Below-threshold → fall back to a default agent or escalate to HITL.

When the router is needed vs. not

  • Needed: fleet with multiple agents per use case (e.g., chatbot platform with simple, ReAct, plan-execute, reflection variants for same domain).
  • Not needed: single-agent deployments — UI calls agent directly.
  • Not needed: when the UI already knows which agent to call (e.g., dedicated status bot).
Architectural principle. Reasoning strategy stays fixed per-agent overlay (deterministic, observable, evaluable). Dynamic adaptation happens at the fleet level via routing, not inside a single agent. This is Anthropic's "routing" pattern applied to AEA — adaptive where it belongs, deterministic where it should be.
Effort: ~1–2 weeks. Router service (Lambda or Fargate) ~1 week; manifest extension + Agent Registry UI field ~3 days; Capability UI POST redirect ~2 days. No changes to C1/C2/C3 internals. Sits in C4 control plane area as an Agent Registry consumer.
Section 10

Implementation roadmap for full coverage

To move AEA from ~80% coverage to ~95%, in order of impact × effort:

Phase 1 — Tier-1 extensions (core v1.0)

  • classifier_agent overlay (C1) — ~1 week. Unlocks intake triage and routing use cases.
  • workflow_agent type + durable agent session state (C3) — 4–6 weeks. Build the workflow_agent type and wire durable session state via AgentCore Runtime Filesystem Persistence (just shipped), Temporal used as a session store, or Step Functions session state. This is agent session persistence, not a business workflow engine. Unlocks appeals, PA, outreach, investigations.
  • Event-adapter pre-graph (C4) — 3–4 weeks. Kafka / EventBridge / SQS / FHIR-subscription ingress. Unlocks real-time fraud, intake triage, change reactions.
  • Hierarchical workflow_agent (C7) — 6–8 weeks. Orchestrator + worker sub-graphs. Unlocks complex multi-step cases.

Phase 2 — Tier-2 extensions (v1.5)

  • Batch orchestration layer (C5) — 4–6 weeks. Step Functions / Spark / Ray integration + scatter-gather hooks + shared-context cache.
  • Peer / debate / market multi-agent patterns (C7) — 12+ weeks. New orchestrator types, A2A protocol, blackboard memory scope.
  • Guardrail agents — 4 weeks. First-class agent type that monitors others and blocks high-risk actions.

Phase 3 — Integrations (not platform work)

  • Voice pipeline integration (C6) — partner with Pipecat / LiveKit / AgentCore Voice. AEA exposes tools, RAG, memory, HITL services as backends; voice loop runs in partner runtime.
  • Browser-use / computer-use as MCP tools (C8) — wrap Claude Computer Use, browser-use CLI, or equivalent as callable tools.
  • Formal verification as MCP tool (C8) — wrap proof engines for medical policy / legal reasoning.

Phase 4 — Government-readiness

  • FedRAMP / IL4–6 / GovCloud deployment profile.
  • Air-gapped option (on-prem / classified).
  • Section 508 accessibility pass on capability UIs.
  • FISMA / FedRAMP policy packs.
Total roadmap envelope to reach ~95% coverage: 6–9 months of platform engineering (Phases 1–2), plus integration partnerships (Phase 3) and a Gov v2.0 track (Phase 4). 80% coverage is already shipping-ready today.
