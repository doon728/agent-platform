Section 1

Scope and conventions

This document covers only the agent types actually built in code today:

  • chat_agent_simple — interactive chat agent with simple reasoning strategy. Used by care-management/pre-call-assessment.
  • summarization_agent_simple — deterministic single-pass summarizer. Used by care-management/pre-call-assessment for case/member summaries.

Other agent types (chat_agent_react, chat_agent_plan_execute, chat_agent_reflection, chat_agent_multi_hop, classifier_agent, workflow_agent) are on the roadmap but not yet implemented and are not covered here.

Three levels per agent type

Each agent type has three diagrams, each at a different level of detail:

LevelAudienceShows
Level 1 — High-level architectureArchitects, executivesWhere the agent sits in the 8-category taxonomy + which containers it engages + the call path.
Level 2 — Lifecycle flowEngineers, operatorsPre-graph / in-graph / post-graph timeline. What runs at each phase.
Level 3 — Config overlayOperators, adminsSame lifecycle, annotated with which YAML files drive each step.

Symbols used

  • C1 Agent Runtime — per-agent thin shell (loads overlay config, owns HITL state, forwards to C2).
  • C2 Platform Services — shared reasoning + memory + tools router.
  • C3 Tool Policy Gateway — tool serving + policy enforcement.
  • RAG services/rag — standalone retrieval service (HTTP).
  • Cat = Category (1 of 8 from agent-taxonomy-matrix).

LLM call count for one turn

StepLLM call?Where
Memory readNo—
Pre-graph RAGYes (embedding)services/rag (one embedding generation per query)
PlannerYes (chat completion)C2 — Bedrock or OpenAI
search_kb toolYes (embedding)services/rag (only if search_kb is the chosen tool)
ResponderYes (chat completion)C2 — Bedrock or OpenAI
Semantic extractionYes if enabled (chat completion)C2 platform-core/memory/semantic_engine.py

Typical turn: 2 chat-completion LLM calls (planner + responder) + 1–2 embedding calls. Extractor adds 1 more if enabled.
