Section 1

Overview — accelerator repo model

The accelerator is a single Git repository that contains both the template sources and the generated agent instances. The Agent Factory UI reads template sources under templates/ and scaffolds instances under agents/<capability>/<agent-name>/. Runtime services live in services/ and platform-services/. A shared Python library lives in packages/platform-core/, imported by both Container 1 (agent runtime shell) and Container 2 (platform services).

The mental model is four logical containers that map to physical services at runtime:

ContainerPurposeLives in (code)
C1 Agent Runtime ShellPer-agent thin entry point. FastAPI app that owns session state, HITL approval store, graph orchestration. Delegates all reasoning to C2.templates/agent-runtime-shell/ + overlay-specific files from templates/overlay-templates/overlays/<overlay>/. Generated into agents/<capability>/<agent>/services/agent-runtime/.
C2 Platform ServicesShared tenant service. All reasoning happens here — planner, executor, responder, memory, RAG, tools, observability, summarize, config.services/platform-services/ (canonical) and templates/platform-services/ (template source).
C3 Tool GatewayShared per-tenant tool registry + MCP-style serving. Tools registered with schemas; RAG KB lives here; allow/deny per agent enforced.services/tool-policy-gateway/.
C4 Control Plane (distributed today)Currently split across multiple services — Agent Factory UI, Support API, Prompt Management. To be consolidated.services/agent-factory-ui/, services/agent-factory-support-api/, services/prompt-management/, platform-store/usecase_contracts.json.
Shared platform-core libraryPython library consumed by C1 and C2. Contains reasoning building blocks, memory, HITL, RAG, tools, auth, context, prompts.packages/platform-core/src/platform_core/.
Current state vs target state. This document describes the prototype as it runs today — HTTP between containers, file-backed memory, local Docker, no AgentCore managed services integrated. Target state (AgentCore Runtime hosting, AgentCore Memory, AgentCore Gateway, AgentCore Observability) is documented separately and is on the backlog for the v1.0 build.

templates/agent-runtime-shell/ — Generic C1 code

The FastAPI app that acts as the agent's entry point. Referenced by every generated agent's C1.

agent-runtime-shell/
├── Dockerfile                 ← Container image build
└── src/
    ├── platform/
    │   ├── app.py             ← FastAPI app: /health, /invoke, /approve, /traces
    │   └── langgraph_runner.py  ← Orchestrates pre-graph → reasoning → post-graph
    ├── clients/               ← HTTP clients to call C2 (Platform Services)
    │   ├── base.py            ← Shared HTTP base
    │   ├── config_client.py   ← GET /config/agent, /config/memory, /config/domain
    │   ├── memory_client.py   ← POST /memory/read, /memory/write
    │   ├── rag_client.py      ← POST /rag/retrieve
    │   ├── strategy_client.py ← POST /reasoning/run
    │   └── tools_client.py    ← POST /tools/invoke (delegated to C2 which calls C3)
    └── graph/
        └── build_graph.py     ← Default graph builder (overrideable per overlay)

Key file: src/platform/app.py

  • FastAPI application — the C1 entry point.
  • Loads platform_core.hitl.approval_store — local SQLite-backed HITL state.
  • Authenticates requests via platform_core.auth.authenticate_request.
  • Enforces tenant isolation via platform_core.authorization.enforce_tenant_isolation.
  • Builds per-request context via platform_core.context.build_context.
  • Exposes routes: /health, /config/scopes, /config-flags, /traces, /traces/latest, /invoke (plus HITL approval endpoints).

Key file: src/platform/langgraph_runner.py

  • LangGraphRunner class — orchestrates the 3-phase flow for every invocation.
  • Pre-graph: load agent config from C2, build memory policy state, build scope metadata from domain.yaml, fetch pre-graph RAG (if enabled), initialize HITL context.
  • In-graph: delegate to C2 reasoning service (strategy_client.run). All LLM calls, tool invocations, planner decisions happen in C2.
  • Post-graph: evaluate HITL gates, write memory (via memory_client), emit observability traces.
  • Zero business logic in C1 — only orchestration and HTTP calls.

Key folder: src/clients/

Thin HTTP clients that call C2 endpoints. Each client targets one C2 router:

  • config_client.py → C2 /config (get_agent_config, get_memory_config, get_domain_config)
  • memory_client.py → C2 /memory (read/write across 4 scopes)
  • rag_client.py → C2 /rag (retrieve chunks per agent's RAG config)
  • strategy_client.py → C2 /reasoning (run the reasoning strategy — simple, ReAct, plan-execute, etc.)
  • tools_client.py → C2 /tools (C2 proxies to C3 Tool Gateway)

What an overlay carries

  • overlay.yaml — declares agent_type, required components (planner, responder, executor), feature flags (memory, observability, prompt_versioning), required prompts, entrypoint (orchestration_graph path), and metadata about the RAG dimension 2 pattern used.
  • overlay.yaml + reasoning.yaml + rag.yaml + hitl.yaml + tools/tools.yaml — the runtime config: agent.type, planner_mode, reasoning.strategy, tools (mode + allowed list), retrieval (pre_graph RAG + planner_tool RAG, each with Dim 1 strategy / Dim 3 pattern / top_k / threshold).
  • memory.yaml — per-scope read/write policies for the 4 memory scopes (short_term, episodic, summary, semantic).
  • prompts/prompts.yaml — seed prompts for planner / responder / summarizer, used when Prompt Management has no overriding version.
  • agents/ — Python modules for the overlay's planner, executor, responder (or summarizer), plus reasoning strategy implementations.
  • graph/state.py — the LangGraph state schema (TypedDict) passed between nodes.
  • orchestration/build_graph.py — the LangGraph topology builder. Wires together the planner, executor, responder into a StateGraph with the right edges.

What C2 does

  • Reasoning (core): runs the reasoning strategy for every agent. Loads planner / executor / responder per overlay; invokes LLM; handles tool calls via C3 client; returns final response.
  • Memory: provides read/write endpoints across the 4 scopes. Backend adapter chosen by config (MEMORY_BACKEND=file in local dev).
  • RAG: handles pre-graph retrieval and planner-tool retrieval. Calls C3's KB endpoints.
  • Config: serves agent config, memory config, domain.yaml to C1 at invocation time.
  • Tools: proxies tool invocations from C1 → C2 → C3 Tool Gateway. C2 does per-agent allow/deny enforcement.
  • Observability: collects trace events from C1 + emits its own spans; exposes trace retrieval endpoint.
  • Summarize: separate endpoint for summarization agents (batched context → summary output).

What C3 does

  • Tool registry: holds tool definitions (name, input schema, output schema, handler reference). Loaded at startup from db_registry (Postgres) or bootstrap scripts.
  • Tool invocation: receives invocation envelopes from C2; validates input schema; calls the tool handler; returns envelope response.
  • KB / RAG backend: ingestion (rag/ingest.py) + retrieval (rag/retriever.py). Serves search_kb tool.
  • Data store: Postgres for structured data (members, claims, cases) and tool definitions. Synthetic data generator for local dev.
  • Admin UI (separate service): React app at tool-admin-ui/ to view / manage registered tools and KB content.
  • OpenAPI spec: openapi.yaml documents the tool invocation contract. MCP serving is a planned evolution from today's HTTP invocation.

Representative tool handlers (in src/tools/registry.py)

  • search_kb — KB retrieval with query
  • get_member — fetch a member record
  • get_member_summary — precomputed member summary
  • get_case_summary — case summary
  • get_assessment_summary, get_assessment_tasks — assessment data
  • write_case_note — write-back to case notes

platform-services (top-level) — Prompt Management

Separate microservice dedicated to prompt lifecycle. Distinct from services/platform-services/ (C2).

platform-services/
└── prompt-management/
    ├── app/
    │   ├── main.py              ← FastAPI app (REST endpoints)
    │   ├── db.py                ← Database connection
    │   ├── models.py            ← SQLAlchemy ORM models (Prompt, Version, Evaluation)
    │   ├── schemas.py           ← Pydantic request/response schemas
    │   └── store.py             ← Prompt CRUD + lifecycle transitions
    ├── data/
    │   ├── prompts.json         ← Seed prompts
    │   └── evals.json           ← Seed evaluations
    ├── seed_prompts.py          ← Seed data loader
    └── sql/                     ← SQL schema / migrations

What it does

  • Per-agent versioned prompts — store prompt text with version metadata.
  • Lifecycle: draft → active → retired. Only one active version per prompt at a time.
  • Update without redeploy: C2's prompt client fetches latest active prompts at invocation time. Rolling a new prompt = promote new version to active; no container redeploy.
  • Evaluation records — store eval outcomes alongside prompts.
  • Fallback: if no active prompt is in Prompt Management, C2 falls back to prompt-defaults.yaml packaged in the overlay.

Section 9

platform-store — use-case contracts

platform-store/
└── usecase_contracts.json

Central registry of known use-case contracts. A use-case contract defines what an agent is expected to do — inputs, outputs, prerequisites, success criteria. Consumed by:

  • Agent Factory UI — when scaffolding a new agent, pick a use-case contract as the starting point.
  • Support API — exposes contract list to the UI.
  • platform-core usecase_config_loader.py — loads contracts at runtime for use-case-aware agents.
Today this is a flat JSON file. Future state: Postgres-backed contract store, UI-driven CRUD, versioned contracts.

Section 12

Architecture considerations & tradeoffs

This section documents the design intent behind the four-container split, the production runtime options per container, and the key tradeoffs. It also calls out a terminology collision with AgentCore that can confuse readers.

Terminology note — "Tool Gateway" collision. Two different things are both called "Tool Gateway" in this document space:
  • AEA Tool Gateway (C3) — our abstraction layer: a shared per-tenant service that registers tools, enforces allow/deny per agent, serves RAG KB, and optionally federates to downstream tool runtimes.
  • AgentCore Gateway — AWS managed service that exposes tools to agents via MCP (Model Context Protocol). AWS docs sometimes refer to this as a "tool gateway."
Throughout this doc: AEA Tool Gateway = our C3 code. AgentCore Gateway = the AWS managed MCP service. They are not the same layer — our C3 can sync-register its tools with AgentCore Gateway (or be bypassed entirely, see C3 options below).

12.1 Why four containers? (design intent)

The accelerator is deliberately split into four logical containers, each with a different responsibility and a different runtime-fit profile:

ContainerWhy it exists (design intent)
C1 Agent Runtime ShellPer-agent entry point. Owns session state, HITL approval store, graph orchestration for that specific agent. Kept intentionally thin so multiple agents can share the heavy platform layer below.
C2 Platform ServicesShared multi-tenant backend that every agent calls — reasoning engine, memory, RAG, tools, observability, config, summarize. Centralized so one platform upgrade benefits all agents; changes to prompts/memory/RAG don't require rebuilding every agent.
C3 AEA Tool GatewayShared tool + RAG registry. Abstraction layer between agents and underlying tool runtimes (MCP servers, AgentCore Gateway, or direct APIs). Provides allow/deny, per-agent tool scoping, and single point of policy enforcement.
C4 Control PlaneOperator console + admin APIs. Agent Factory UI, prompt management, use-case contracts. Not in the runtime path — used to scaffold, configure, govern, and observe agents.

12.2 Production runtime options per container

C1 Agent Runtime Shell

OptionProsCons
AgentCore Runtime (recommended)Purpose-built for agent hosting (per-session, auto-scale, identity, memory integration). Strengthens AWS co-sell. Native fit for "one agent, one deployment."Requires conforming to AgentCore Runtime's packaging + invocation model. Less control over runtime environment.

12.3 C3 integration patterns with AgentCore Gateway (three options)

C3 (AEA Tool Gateway) and AgentCore Gateway (AWS managed MCP server) overlap in purpose — both expose tools to agents. Three integration patterns exist:

PatternHow it worksProsCons
A. C3 + AgentCore Gateway (sync)Tools registered in C3 also sync-register with AgentCore Gateway. Agents discover tools via MCP from AgentCore Gateway; C3 remains as the single source of truth and policy enforcement layer.AWS-native tool surface (MCP). C3 keeps policy/allow-deny authority. Best of both.Two registries to keep in sync. More moving parts.
B. Tools registered directly to AgentCore Gateway (skip C3)Tools register directly with AgentCore Gateway's MCP server. C3 is eliminated or reduced to a thin policy shim.Simpler — one tool registry. Fully AWS-managed. Strongest AWS co-sell.Lose C3's per-agent allow/deny, RAG KB integration, and abstraction over non-MCP tools. Harder to swap tool runtimes later.
C. C3-only (no AgentCore Gateway)C3 serves tools directly to agents; AgentCore Gateway not used.Full control. No dependency on AgentCore Gateway.Agents don't get AgentCore-native tool surface. Weaker AWS co-sell. We duplicate what AgentCore Gateway already does.

Recommendation: Pattern A for v1.0. Keeps C3 as policy + abstraction layer, uses AgentCore Gateway as the MCP serving plane. Pattern B is a future simplification once AgentCore Gateway covers all our enforcement needs.

12.4 Key tradeoff — why C1/C2 are split (and why it's uncomfortable with AgentCore Runtime)

The biggest architectural question is whether the C1/C2 split is correct, or whether they should be combined into one agent-per-container model that fits AgentCore Runtime cleanly.

AspectCombined C1+C2 (per-agent)Split C1/C2 (current AEA)
Deployment modelOne container per agent — self-contained bundle.One C1 per agent + one shared C2 for all agents.
AgentCore Runtime fitClean fit. Each agent = one AgentCore Runtime deployment.Partial. C1 fits AgentCore Runtime; C2 does not (shared multi-tenant service).
Platform upgradesEvery agent must be rebuilt + redeployed.Upgrade C2 once → all agents benefit.
Code duplicationEach agent ships full reasoning + memory + RAG stack.DRY — shared reasoning engine, memory layer, RAG.
Prompt / config mgmtPer-agent. Duplicated across agents.Centralized in C2. Single source of truth.
Network hops per reasoning stepZero — all in-process.One extra hop per step (C1 → C2).
AWS co-sell narrativeStrongest — "each agent is an AgentCore Runtime agent."More nuanced — "AgentCore Runtime hosts agents (C1); Fargate hosts the shared platform (C2)."
Accelerator valueWeaker — each agent is independent.Stronger — shared platform is the reusable asset.
The architectural tension named plainly. C2 contains LangGraph reasoning code (services/platform-services/src/services/reasoning/strategies/). On the surface this looks like an agent runtime and raises the question: "shouldn't that be on AgentCore Runtime?" It is not — because AgentCore Runtime is per-agent-session by design, while C2 is a shared multi-tenant service that many agents call concurrently. Same code (LangGraph), different runtime semantics (dedicated per session vs. shared always-on). That's why C2 is Fargate, not AgentCore Runtime.

AEA's deliberate choice: split C1/C2 to make the platform reusable across many agents. That's what makes this an accelerator — a shared platform N agents plug into — rather than N independent agents. The cost is that C2 doesn't fit AgentCore Runtime's per-agent model; the benefit is that one platform upgrade benefits all agents and keeps prompt/memory/policy management centralized.

12.5 What AgentCore services we use (and don't)

AgentCore is a collection of managed primitives, not a single monolithic service. AEA uses some primitives and not others:

AgentCore serviceAEA usageWhere it plugs in
AgentCore RuntimeC1 (per-agent hosting) — v1.0 targetHosts each agent's C1 entry point. Not used for C2.
AgentCore MemoryC2 memory service backs onto it (v1.0 target)Long-term memory storage. C2 provides the API; AgentCore Memory stores durably.
AgentCore GatewayC3 federates to it (Pattern A, §12.3) — v1.0 targetMCP tool-serving layer. C3 registers tools with it.
AgentCore IdentityAuth/OAuth for tool access — v1.0 targetUsed by C3 + agents for tool-call authorization.
AgentCore ObservabilityTraces from C1/C2/C3 — v1.0 targetUnified tracing/metrics for agent workloads.
AgentCore Code InterpreterOptional — use case dependentSandboxed code execution for data/compute-heavy agent tasks.
AgentCore BrowserOptional — use case dependentBrowser automation tool for agents that need to navigate UIs.

Summary: AEA uses AgentCore as a set of managed primitives (Memory, Gateway, Identity, Observability) + Runtime for C1. C2/C3/C4 shared services run on Fargate because they are multi-tenant backends, not per-agent runtimes.

12.6 Open questions for v1.0 build

  • C4 consolidation: Stay distributed (UI + Support API + Prompt Mgmt + contracts store separate) or consolidate into a single Control Plane service?
  • C3 integration pattern: Lock in Pattern A (C3 + AgentCore Gateway sync), or accept Pattern B (direct MCP, skip C3)?
  • C2 split further: Keep C2 as one service, or split into reasoning-engine (Fargate) + CRUD services (Lambda)?
  • Per-tenant vs. per-agent C2 deployment: One shared C2 per tenant, or one C2 per capability/agent-family?
  • AgentCore Runtime packaging for C1: Does each agent become its own AgentCore Runtime deployment, or one runtime serves multiple overlays?

Phase-by-phase breakdown

Pre-graph (in C1)

  1. Config load. config_client fetches agent.yaml, memory.yaml, and domain.yaml from C2's /config endpoints. Overlay-specific config is served by C2 based on AGENT_TYPE env var.
  2. Memory policy state. _build_memory_policy_state combines memory.yaml read/write policies with any per-request overrides from ctx.memory_policy_override.
  3. Scope metadata. _build_scope_metadata extracts scope IDs (member_id, case_id, etc.) from the request context based on domain.yaml's scopes definition.
  4. Pre-graph RAG. If agent.yaml has retrieval.pre_graph.enabled = true, rag_client.retrieve calls C2 with the query and config; C2 orchestrates (hits C3's search_kb tool); chunks are attached to the graph state.
  5. HITL init. If any HITL gates apply, context is marked; approval_store is checked for pending approvals on this thread.

In-graph (delegated to C2)

  1. Reasoning invocation. strategy_client.run POSTs to C2 /reasoning with overlay config, graph state, and prompt.
  2. C2 reasoning router dispatches based on reasoning.strategy (simple / react / plan-execute) and agent.type. Loads the overlay's planner / executor / responder modules.
  3. LLM calls. platform_core.llm wraps the LLM provider (OpenAI today); C2 invokes it.
  4. Tool calls during reasoning. If the planner decides a tool is needed, or if planner_tool RAG fires mid-graph, C2 calls platform_core.tool_gateway_client which HTTP-calls C3's /invoke endpoint. C3 runs the tool handler, returns envelope.
  5. Response synthesis. Responder module synthesizes the final answer. Returned to C1.

Post-graph (in C1)

  1. HITL evaluation. If the agent's output triggers a gate rule, write to approval_store (SQLite) and return a pending-approval response. Approver hits /approve endpoint later → resume flow.
  2. Memory writes. memory_client POSTs writes for each enabled scope (short_term, episodic, summary, semantic) per memory.yaml policy.
  3. Observability. Trace events (spans, metrics, input/output digests, LLM/tool calls) emitted to C2 /observability.
  4. Response return. Response flows back to the UI.

Config loading on startup

  • Both C1 and C2 load their config/base.yaml + config/<env>.yaml + environment-variable overrides.
  • C2 loads manifests and overlays at startup from the mounted /app/overlays volume (mounted from agents/<cap>/<agent>/overlays).
  • C2 loads domain.yaml from the mounted volume.
  • C3 loads tool definitions from db_registry (Postgres) on startup (bootstrap_all.sh seeds if empty).

Section 15

How templates generate an instance

When an operator / developer creates a new agent via Agent Factory UI (or the Support API), the scaffolding process composes files from template sources into agents/<capability>/<agent-name>/.

Scaffolding sequence

  1. Pick inputs: capability (e.g., care-management), agent name (e.g., pre-call-assessment), overlay (e.g., chat_agent_simple), use-case contract (optional).
  2. Copy common shell: templates/overlay-templates/common/services/agent-runtime/ → agents/<cap>/<agent>/services/agent-runtime/.
  3. Copy runtime-shell source: files from templates/agent-runtime-shell/src/ become the Python source inside the new agent's services/agent-runtime/src/.
  4. Copy overlay files: templates/overlay-templates/overlays/<overlay>/ → agents/<cap>/<agent>/overlays/<overlay>/ (manifest, config, agents, graph, orchestration).
  5. Copy docker-compose + infra stubs: templates/overlay-templates/common/docker-compose.yml + infra/ → agent root.
  6. Copy domain.yaml: capabilities/<cap>/domain.yaml → agent root (scoped per-agent).
  7. Copy per-agent UI (if applicable): templates/agent-ui-template/<ui-variant>/ → services/chat-ui/ or services/summary-ui/.
  8. Register in Agent Registry: Support API writes registry entry (agent_id, LOB namespace, overlay, paths).
  9. Result: a runnable agent. Developer runs ./start-runtime.sh or the compose file to bring it up locally.

What's static vs generated at scaffold time

  • Static (copied as-is): agent-runtime-shell source, overlay source, Dockerfile, docker-compose, infra stubs.
  • Generated / substituted: environment variables in docker-compose (agent name, capability name, use-case), config paths, capability-specific domain.yaml reference.
  • Added via UI later: custom prompts via Prompt Management, tool allow-list edits, memory policy adjustments (edit agent.yaml / memory.yaml in the scaffolded repo; restart or rebuild).

Section 17

Known gaps and current-state notes

What works today

  • C1 ↔ C2 ↔ C3 HTTP flow end-to-end (local Docker).
  • Two overlays implemented: chat_agent_simple, summarization_agent_simple.
  • One reference agent running: pre-call-assessment.
  • Capability UI for Care Management.
  • Agent Factory UI (React + TypeScript) — Workspaces, AgentRegistry, ConfigLab, PipelineBuilder, PromptLifecycle, TestResults pages.
  • Support API — basic endpoints for scaffolding and registry.
  • Prompt Management service — versioned prompts with lifecycle.
  • Tool Gateway — multiple tool handlers + KB retrieval + Postgres-backed synthetic payer data.
  • platform-core shared library installed into C1 and C2.
  • HITL durable state via SQLite in C1.
  • File-backed memory with 4-scope policies.
  • RAG across 3 dimensions (stage × strategy × pattern) configurable per agent.

What's partial / in progress

  • Runtime split — C1/C2 are split by intent but some overlay logic still lives close to the agent runtime; further tightening underway.
  • Reasoning strategies — simple and react implemented; plan_execute stubbed; multi_hop, reflection on backlog.
  • Memory backends — file works; pgvector / DynamoDB / Redis adapters scaffolded but not wired end-to-end.
  • HITL — durable state is present; full approver UI + notification paths are minimal.
  • Observability — trace emission works; domain enrichment + dashboards minimal.
  • Agent Registry — data model exists via Support API; cross-account sync not wired.

What's not in the prototype

  • AgentCore integration — Runtime, Memory, Gateway, Identity, Observability not consumed yet. All AWS managed-service wiring is v1.0 build work.
  • MCP protocol — Tool Gateway serves via HTTP + OpenAPI today, not MCP. MCP facade / AgentCore Gateway sync is backlog.
  • Bedrock — LLM goes to OpenAI today; Bedrock wiring is backlog.
  • Bedrock Knowledge Bases — KB is our own Postgres + retrieval code today.
  • PHI masking as enforcement architecture — designed, not yet built as pre-response interceptor.
  • Full IaC (Terraform / CDK / Helm) — stubs only; v1.0 workstream.
  • Unified CI/CD across services — minimal per-service CI only.
  • Production-hardened security — local dev has AUTH_MODE=OPTIONAL; SAST/DAST/pen-test pipeline is v1.0 work.
  • Exception handling / retries / circuit breakers — minimal today; backlog item #1.
  • Event-driven agents — not wired; backlog item #2.
  • Guardrails service — designed, not built; backlog item #6.
  • Connect integration — not wired; v1.0 work.
  • IDP via Bedrock Data Automation — not wired; v1.0 work.
  • Reference agent templates beyond chat_agent_simple / summarization_agent_simple — PA Denial Remediation, Pre-Call Assessment (full scenario content), Claims Triage — v1.0 work.
  • Pre-built payer tool library — the generic tool handlers in Tool Gateway are a starting point; full library of FHIR R4, entity-level payer APIs, InterQual, MCG, Availity, Change Healthcare, EHR FHIR wrappers is v1.0 work.

Open architectural questions

  • C4 consolidation: Agent Factory UI, Support API, Prompt Management, usecase contracts store are separate today. Should they consolidate into a single Control Plane service, or stay separate?
  • C3 MCP serving path: add MCP facade inside C3 (Pattern 3) or sync to AgentCore Gateway (Pattern 1)? Covered in design.md §8.
  • Agent Registry granularity: our Registry (Agent ID + LOB + prompts + RAG + tools + domain.yaml) vs AgentCore native Registry (AWS-level agent catalog) — sync or coexist?
  • Memory backend default for v1.0: AgentCore Memory primary, with pgvector for semantic — or keep file for first deployments and switch later?
