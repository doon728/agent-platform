Architecture Proposal

Enterprise Chatbot Platform — Reusable, Context-Aware, Cross-Source

One AWS-native platform; many self-serve chatbots. Trained on ServiceNow, Jira, Confluence — extensible to any dataset. Goes beyond generic Copilot.

Section 1

Use case & business context

What Tom's team (Joe's org) wants to build

  • A system that pulls project and delivery status from multiple sources — ServiceNow (incidents/changes), Jira (tickets/projects), Confluence (docs/runbooks).
  • Reusable, context-aware chatbots that teams can train on specific data sets without rebuilding each time.
  • Beyond generic Copilot — domain- and dataset-specific answers with tool invocation and multi-source reasoning.
  • Aligned with their pivot toward AWS serverless architecture.

Constraints & context

  • Current vendor rolling off ML resources; architectural guidance but limited hands-on execution.
  • Budget-constrained — cannot backfill vendor resources; need a platform approach, not per-bot custom builds.
  • Existing pilots: OCR/IDP, regression testing, data ingestion/sanitization — the platform must integrate, not replace.
Core problem: hand-building one chatbot per use case scales linearly — every bot reinvents auth, RAG, tool plumbing, memory, and governance. The real need is a reusable platform where each new bot is a config, not a project.

Layer roles

  • Runtime — per-bot agent execution on AgentCore Runtime; thin entry point, handles session + HITL.
  • Platform — shared C2 services (reasoning, memory, RAG, observability) used by every bot.
  • Tools — C3 policy layer + AgentCore Tool Gateway for MCP tool serving (ServiceNow, Jira, Confluence connectors).
  • Control Plane — C4 UI for teams to self-serve new bots, manage prompts, view governance.
  • Orchestration — Step Functions + EventBridge for scheduled / batch / event-triggered workflows.

Section 4

Reasoning strategies — configurable per bot, per query type

Unlike generic Copilot (one fixed reasoning pattern), each bot picks reasoning strategy by use case. Teams can even use multiple strategies inside one bot for different persona.

Query typeStrategyExampleWhy this strategy (in this context)Flow (high-level)
Simple lookup simple "Status of ticket JIRA-123?" One tool call, one LLM response. No loop, no planning. Fastest and cheapest — use for direct fact retrieval. 1 LLM call total. User query → LLM reasons + optionally calls 1 tool → LLM returns response. Done.
Multi-source lookup ReAct "Open tickets and incidents for Project X" 2–5 tool calls needed (Jira + ServiceNow). Agent reasons between calls — "got tickets, now fetch incidents, now summarize." Adaptive: if Jira returns zero, agent can skip ServiceNow. No upfront plan required — too light for plan-execute, too heavy for simple. Reason-Act loop. LLM thinks → picks tool + args → tool runs → LLM observes result → LLM thinks again → picks next tool or ends. Final LLM call synthesizes response. Multiple LLM calls interleaved with tool calls.
Cross-source correlation plan-execute "Why is Project X late?" — plan: fetch Jira + ServiceNow + Confluence → correlate Upfront plan with dependencies. Agent decides: "I need ticket history, change windows, and recent decisions, then correlate them." Planning is worth the overhead because later steps depend on earlier findings. Multiple coordinated tool calls with causal reasoning across results. Planner → Executor → Responder (all LLM). Planner LLM outputs structured plan (ordered steps with dependencies). Executor loop walks the plan — LLM picks tool args per step, tool returns data. Responder LLM synthesizes all step results into final answer. Structure is deterministic (LangGraph); plan content is LLM-generated.
Quality-sensitive output reflection Executive weekly status report — draft, self-critique, refine Output is high-stakes (C-level visibility), so quality > speed. Agent drafts, then a critic loop checks for accuracy, tone, completeness. Loops until quality gate passes. Common for generated artifacts that humans will trust at face value. Generator ↔ Critic loop. Generator LLM drafts output → Critic LLM evaluates (accuracy, tone, completeness). If fails, feedback returns to generator to refine. Iterates until quality gate passes or max loops hit → final response. Both are LLM calls; the loop is deterministic.
Open-ended investigation multi-hop "What's blocking delivery across the portfolio?" — evidence chain across sources No pre-defined answer path. Each hop (tool call + reasoning step) builds on the last — "found project X at risk, now look at its dependencies, now check staffing on those, now cross-reference with Confluence decisions." Termination is heuristic (sufficient evidence / confidence reached). Used for exploratory questions where the answer shape isn't known upfront. Evidence-chain hops. Hop 1: LLM reasons + calls tool → observes. Hop 2: LLM reads hop-1 result, forms new question, calls next tool → observes. Repeat until LLM decides evidence is sufficient. Final LLM synthesizes. Each hop is an LLM call + tool call; termination is LLM-judged.
Key differentiator: Reasoning strategy is a config, not code. A team can switch a bot from simple to plan-execute via overlay YAML as requirements evolve. Generic Copilot (and most single-vendor chatbots) don't expose this lever.
Section 5

Multi-agent — when and how it kicks in

Most daily queries run as single-agent. Multi-agent workflows fire for advanced cases — accuracy and depth matter more than raw speed.

When multi-agent applies

  • Executive weekly status report — specialist sub-agents per source (Jira agent, ServiceNow agent, Confluence agent) → synthesis agent combines → reflection loop for quality.
  • Delivery risk assessment across portfolio — sub-agents analyze different dimensions (schedule, budget, staffing, blockers), synthesize into one risk score.
  • Root cause analysis — investigator sub-agents pull evidence; synthesis agent forms hypothesis; critic agent challenges.

How multi-agent is triggered

  • User query — main chat_agent's planner decides "this needs multi-source deep analysis" → invokes workflow_agent sub-graph. User-transparent.
  • Scheduled — EventBridge Scheduler cron (e.g., Monday 9am exec report) → Step Functions → workflow_agent.
  • Event-driven — project enters "at risk" state in Jira → EventBridge event → workflow_agent runs risk analysis.

Orchestration model — hierarchical, not A2A

  • Orchestrator agent calls sub-agents like tools (function calls, shared graph state) — hierarchical.
  • Sub-agents don't negotiate with each other; they report to the orchestrator.
  • No A2A protocol needed — true peer-to-peer Agent-to-Agent is for debate / market / federated patterns, not this use case.
Recommendation for this client. Phase 1–2: single chat_agent with ReAct + plan-execute. Phase 3+: introduce hierarchical workflow_agent for exec reports and portfolio risk. Same platform, same control plane — multi-agent is a capability layered on as needs grow.
Section 6

Where Step Functions fits (and doesn't)

Where Step Functions fits

  • Scheduled multi-agent jobs — weekly executive report, monthly portfolio review.
  • Long-running agent workflows — delivery risk analysis taking 5–30 minutes across many sources. Step Functions holds state, retries, DLQ.
  • Nightly batch jobs — scan all projects for risk signals, flag changes, push alerts.
  • Durable agent session state — for long-running workflow_agent instances that pause on HITL.
  • Data sanitization pipelines — integrates with their existing OCR/IDP work (ingestion → extraction → PII scrubbing → indexing).
  • Error handling + retry — built-in retry policies, fallback branches, DLQ for orchestrating unreliable external APIs.

Where Step Functions does NOT fit

  • Real-time conversational turns — too heavy, too slow for sub-3-second user replies. Use AgentCore Runtime directly.
  • Per-user chat sessions — overkill; session state lives in AgentCore Memory.
  • Simple single-agent queries — overkill; direct agent invocation is faster and cheaper.
Simple split: AgentCore Runtime = live conversational. Step Functions + EventBridge = scheduled, batch, long-running, event-driven. They complement each other.

6.1 Step Functions concept — DAG vs. state machine

DAG = Directed Acyclic Graph. Workflow is defined as a set of tasks (nodes) connected by ordered edges (arrows) with no loops back. Step Functions calls this a "state machine," but it behaves as a DAG: Task A → Task B → Task C, with parallel branches (fan-out / fan-in), error handlers, retries. Each task has durable state; if a task fails mid-flow, Step Functions resumes from the last checkpoint.

Verified flow summary:
  • Step Functions fires when: scheduled (cron), event-driven (webhooks, EventBridge), multi-agent orchestration needed, long-running analyses, batch scans.
  • Step Functions stays OUT of: live conversational chat turns — that goes direct through AgentCore Runtime for latency.
  • The DAG gives you: parallel execution (fan-out), synchronized joining (fan-in), durable state, retries, error branches, audit trail — things a raw Lambda chain cannot do cleanly.
Section 7

Context-aware: how each bot gets tuned

Every bot is a config overlay (YAML + prompts) on a common template. Teams don't rebuild — they configure.

What's configurable per bot

  • Prompts — system prompt, planner prompt, responder prompt. Versioned, A/B-testable.
  • Tool allow-list — which ServiceNow / Jira / Confluence tools this bot can call.
  • Dataset scope — which Confluence spaces, Jira projects, ServiceNow CIs are in scope.
  • Reasoning strategy — simple / ReAct / plan-execute / reflection / multi-hop.
  • Memory scopes — short-term only vs. persistent episodic/semantic.
  • RAG configuration — which corpora, extraction strategy, top-k, thresholds.
  • HITL gates — inline user confirmation before high-impact actions.
  • Access control — who can use this bot, data isolation rules.

Phased delivery plan (6 months)

Phase 1 — Platform foundation + first bot (weeks 1–6)
  • Deploy platform (C1 on AgentCore Runtime, C2 on Fargate, C3 + AgentCore Tool Gateway, C4 control plane).
  • Stand up ServiceNow, Jira, Confluence MCP connectors with stateful sessions.
  • First bot: project / delivery status chatbot spanning all 3 sources.
  • Reasoning: ReAct for day-to-day, plan-execute for cross-source correlation.
  • Deploy via IaC (CDK) — reproducible.
Phase 2 — Self-serve & second/third bots (weeks 7–12)
  • Agent Factory UI live — teams create new bots via config, not code.
  • Second bot: runbook / operations assistant (Confluence + ServiceNow knowledge).
  • Third bot: regression test copilot (their existing pilot) — bot reads test results, suggests fixes.
  • Prove self-serve works: Tom's team scaffolds bot #3 themselves with minimal external help.
Phase 3 — Multi-agent + scheduled workflows (weeks 13–18)
  • Introduce workflow_agent (hierarchical multi-agent).
  • Weekly executive status report — sub-agents per source + synthesis + reflection. Scheduled via Step Functions + EventBridge.
  • Portfolio risk assessment — batch scan + event-driven triggers on risk-state changes.
  • Data sanitization agent integrated into their OCR/IDP pipeline.
Phase 4 — Governance, scale, expansion (weeks 19–24)
  • Governance dashboards: per-bot performance, prompt version tracking, cost per bot, policy compliance.
  • Evaluation workbench — continuous quality evals, A/B prompt tests.
  • Org-wide rollout — finance, HR, engineering ops each get their own context-aware bot.
  • Handoff: Tom's team operates the platform; we stay in advisory mode.

Section 10

Risk & open questions

Risks

  • Data sensitivity — enterprise project data may include confidential information. Mitigation: per-bot scope + access control + PII/PHI masking.
  • LLM cost at scale — uncapped conversations can burn budget. Mitigation: resource-aware optimization (Haiku for simple, Sonnet for complex, Opus only for high-stakes).
  • Tool rate limits — ServiceNow / Jira / Confluence APIs have limits. Mitigation: stateful MCP caches, exception handling with retries, circuit breakers.
  • Vendor pivot gap — their current vendor is rolling off ML resources. Mitigation: our engagement focuses on knowledge transfer so Tom's team operates the platform post-handoff.

Open questions for the Tom / Joe conversation

  • Which dataset sources are Phase 1 priorities (beyond Jira/ServiceNow/Confluence)?
  • Who are the first 3 bot "tenants" — which teams want their own bot first?
  • Compliance / data residency requirements (HIPAA on internal tooling data? SOC2?)
  • AWS account topology — single account or per-team sub-accounts?
  • Preferred chat UI: Slack, Teams, Web, or all?
  • Budget envelope and approval process — do we need to stage the platform work?
