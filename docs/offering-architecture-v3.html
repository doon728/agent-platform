Reference Architecture — Intelligent Agent Accelerator for Healthcare

AWS Native
Diagram 1 — Overview

How This Architecture Works

Container 1 — Agent Runtime

Thin shell on AWS AgentCore Runtime. One per agent.

Container 1 runs on AWS AgentCore Runtime — each agent is hosted as its own per-agent container with session isolation, managed scaling, and a unique ARN. It handles three stages: pre-graph (load domain scope from Container 4, validate auth/authz, hydrate session), in-graph (delegate all reasoning to Container 2 — no LLM logic here), and post-graph (evaluate HITL gates, deliver response). Access policy (auth/authz) is enforced in pre-graph — authenticating the caller and checking permission for the requested agent and scope. This is distinct from content guardrails, which are content-level safety filters applied inside Container 2 on the LLM call path via Amazon Bedrock Guardrails. Being thin means one agent deployment per LOB per account costs almost nothing in engineering effort. On first boot, Container 1 automatically registers with Container 4 using its LOB namespace and AWS account ID discovered from instance metadata.

Container 2 — Platform Services

All reasoning here. Shared per tenant.

Container 2 runs on ECS/Fargate as a shared tenant service, with LangGraph providing the graph execution and Bedrock providing LLM inference. It is shared across all LOBs within a tenant — one deployment serves every agent for that customer. Reasoning strategies (ReAct, plan-execute, multi-hop, reflection), memory (4 scopes), RAG, and LLM inference all execute here. Isolation is logical, not physical: every request carries an Agent ID and LOB namespace, and Container 2 loads exactly that agent's config — prompts, tools, strategy — from Container 4. A Claims agent and a Care Management agent share the same service but run completely isolated configurations.

MCP Server + Container 3 — Tool Gateway

One gateway per tenant, LOB-namespaced tool catalogs.

Container 3 runs on ECS/Fargate as a shared tenant service — one deployment holds LOB-namespaced tool catalogs: Care Management gets care plan and FHIR tools, Claims gets adjudication tools, Member Services gets enrollment tools. Within each catalog, each agent has an allow/deny list enforced by policy — Claims agents cannot call Care Management tools. The MCP Server is an optional facade over Container 3 for third-party interoperability; it can wrap the managed AWS AgentCore Gateway when present. All calls reach LOB source systems (claims DB, care management platform, member DB) via AWS PrivateLink — no public internet.

Container 4 — Control Plane

ECS/Fargate. One per tenant. Logically isolated per agent and LOB.

Container 4 runs on ECS/Fargate as one deployment per tenant and governs every agent across all LOBs within that tenant. Agent configs are stored as YAMLs keyed by Agent ID + LOB namespace, with metadata in RDS. Care Management prompts are completely separate from Claims prompts — different config, different version history, different HITL rules. Observability aggregates CloudWatch metrics and X-Ray traces from all accounts into one cross-LOB dashboard. One Agent Factory UI, one place to govern everything.

Multi-Account Deployment Model

One control plane. Many thin runtimes.

Container 1 and Container 3 deploy per AWS account — each LOB gets its own account. A LOB with multiple sub-domains (e.g. Care Management covering UM, CM, DM) may have one account per domain — each running its own Container 1 + Container 3 pair. Container 2 and Container 4 are shared and central. IaC (CDK + Terraform) parameterized by LOB namespace and account ID handles deployment. The deployment unit is always: one account ID + one LOB namespace + one Container 1 + one Container 3.

Data Sources & RAG Knowledge Bases

Source systems are external. RAG KBs are per LOB/domain.

Source systems (claims DB, member DB, care management platform, prior-auth system) are the customer's existing operational infrastructure — they live in their own accounts and are deployed outside this platform. Container 3 reaches them via PrivateLink APIs. RAG knowledge bases (clinical guidelines, coding policy docs, benefit summaries) are built per LOB or per clinical domain and stored in the customer's environment — OpenSearch or pgvector. The platform's RAG config is flexible: KB endpoints, retrieval strategy, and chunking are all configurable per agent in the Control Plane. The exact KB topology varies by customer environment.

HITL Flow — Human-in-the-Loop

YAML-driven approval gates. Pause, approve, resume.

HITL gates are defined per agent in YAML — not hardcoded. When a gate condition is met, the flow is: (1) User request arrives at Container 1. (2) C1 delegates to Container 2. (3) C2 signals HITL required. (4) C1 checks the HITL & Policy Service in Container 4 — which gate, which approver role, what SLA. (5) Execution pauses; state is written to durable storage (RDS or DynamoDB). (6) Approver is notified (email/Slack/UI). (7) Approver reviews in the Care Manager UI or approver portal. (8) Approval (or rejection) recorded with audit trail. (9) C1 resumes from durable state; C2 continues execution; response delivered. Each LOB defines its own gates — clinical HITL (nurse approves care plan) is separate from claims HITL (supervisor approves prior auth). Gate definitions live in agent YAML and are versioned in Control Plane.

LOB vs Domain — Deployment Granularity

LOB is the account boundary. Domain is the sub-scope within it.

A Line of Business (LOB) maps to an organizational unit and typically an AWS account boundary — examples: Medicare, Medicaid, Commercial Insurance. A Domain is a functional subdomain within a LOB — examples within Medicare: Care Management (CM), Utilization Management (UM), Disease Management (DM). In practice the deployment unit can be either: a single LOB with all domains sharing one Container 1 + Container 3, or one account per domain when compliance or team boundaries require it. The platform supports both. Agent IDs and LOB namespaces in Container 4 are structured as LOB:domain:agent — so isolation is always enforced at the right granularity regardless of deployment topology. The domain scope is captured in domain.yaml, stored in the Agent Registry, and loaded into Container 1 at startup.

Runtime Split — What Runs Where

Why C1 is on AgentCore and C2 / C3 / C4 are on ECS/Fargate.

AWS AgentCore is a set of managed services — AgentCore Runtime (hosts agents), AgentCore Memory (short-term + long-term memory), AgentCore Identity (auth + credentials), AgentCore Gateway (MCP-native tool server), AgentCore Observability (OpenTelemetry-compatible traces and metrics). The accelerator consumes these managed services, it does not replace them. Container 1 is a thin per-agent shell that fits AgentCore Runtime's model exactly — one agent, one session-isolated microVM, one ARN. Containers 2, 3, and 4 are shared tenant-level services that serve many agents; they need warm state, connection pools, and long-lived server processes, which fit regular ECS/Fargate rather than per-session runtimes. This is the same split AWS itself makes: AgentCore Runtime hosts agents; Memory, Identity, Gateway, Observability are separate managed services, not workloads on AgentCore Runtime.

Container 2 Is a Hybrid

Our code on ECS/Fargate + managed AgentCore services consumed.

Container 2 runs on ECS/Fargate and contains our own code — the LangGraph runner, reasoning strategies (simple, ReAct, plan-execute, multi-hop, reflection), HITL engine, RAG orchestration, and domain-level trace enrichment. Alongside our code, Container 2 consumes AgentCore Memory (managed short-term + long-term stores), AgentCore Identity (managed auth + outbound credentials), AgentCore Observability (managed trace ingestion), Amazon Bedrock (LLM inference), Bedrock Knowledge Bases (RAG retrieval backend), and Bedrock Prompt Management (prompt storage). We build only what AgentCore does not ship — reasoning patterns, HITL, RAG orchestration, and domain observability — and wire the managed services underneath.

MCP Tool Serving — Where It Lives

AgentCore Gateway is the MCP server when used. Our C3 is the fallback.

AgentCore Gateway is a managed AWS service that is itself an MCP server. It converts APIs, AWS Lambda functions, OpenAPI specs, Smithy definitions, and pre-built connectors (Salesforce, Slack, Jira) into MCP-compatible tools that agents consume through MCP endpoints. When a customer uses AgentCore Gateway (the default path), the "MCP Server" in the architecture is AgentCore Gateway — managed by AWS, running on AgentCore, no separate component to build or deploy. The Container 3 Tool Gateway in our design is a fallback for customers who need on-premises tool execution, custom schema enforcement beyond AgentCore Gateway, or non-AWS tool backends. In that case C3 runs on ECS/Fargate with our own implementation and the optional MCP facade on top for third-party interoperability.

Observability — Two Layers

AgentCore Observability is the infrastructure layer. We add domain.

AgentCore Observability (managed, OpenTelemetry-compatible, CloudWatch-backed) provides per-agent traces, spans, metrics, and logs at the infrastructure and agent-execution layer. It sees every session, every invocation, every tool call at the platform level. What it does not see is domain context — which member a conversation is about, which clinical workflow the agent is executing, which HITL gate outcome drove a decision, how a response ties back to the capability UI the nurse is looking at. Container 2 emits that domain-level trace enrichment on top of AgentCore Observability, and Container 4 aggregates it into per-capability dashboards the infrastructure layer on its own cannot produce.

Why this architecture is the right call

Source systems never leave their boundary

Patient, claims, and member data stays in the LOB's AWS account. Container 3 calls source systems via PrivateLink — no data traverses the public internet. Each LOB's audit trail, IAM boundary, and encryption keys are fully independent. Satisfies HIPAA and per-LOB compliance isolation simultaneously.

Logical isolation without physical duplication

Container 2 is shared but every agent runs isolated config. Agent ID + LOB namespace is the isolation key — same as how a multi-tenant SaaS platform isolates customers. No rebuilding the reasoning service per LOB. A new LOB deploys one thin runtime and registers config; the platform applies to it immediately.

One place to govern everything

Prompt updates, tool policy changes, reasoning strategy updates — done once in Container 4 and reflected immediately across every agent and every LOB. CloudWatch + X-Ray feeds one cross-LOB observability dashboard. One audit trail. Central governance is designed into the architecture.
