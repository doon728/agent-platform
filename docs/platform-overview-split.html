Platform

What It Is

A multi-tenant enterprise AI agent platform. Agents are configurable via YAML overlays — no code changes needed to switch reasoning strategies, memory policies, RAG patterns, or HITL rules. The platform is domain-agnostic; the first industry deployment is healthcare / care management.
The core design principle: all business logic lives in Container 2 (shared). Container 1 is a thin shell that owns only auth, HITL state, and HTTP orchestration. This means a new agent is just a new C1 instance pointing at the same C2 — zero duplication of logic.
Deployment

Container Split

Three containers with distinct deployment scopes. C1 is deployed per-agent per-customer. C2 is deployed once per customer, shared across all agents. C3 is deployed once per industry.
C1 — Per Agent · Per Customer

Agent Runtime Shell

FastAPI shell. Zero business logic. Routes every phase to C2 via HTTP.

  • Auth + tenant isolation
  • Context build + scope hydration
  • LangGraph runner (phase orchestrator)
  • HITL approval_store (SQLite, local)
  • Proxy endpoints: /chat, /invocations, /summarize, /debug/memory
  • HITL endpoints: /hitl/pending, /hitl/decide, /hitl/status
C2 — Shared · Per Customer

Platform Services

All business logic. Config, memory, RAG, reasoning, tools, summarize, observability.

  • /config — agent.yaml + domain.yaml merge
  • /memory/read + /memory/write
  • /rag/retrieve — 3-dim retrieval
  • /reasoning/run — full planner→executor→responder
  • /tools/invoke — tool registry dispatch
  • /summarize — scoped summary generation
  • /observability/traces
C3 — Shared · Per Industry

Tool Gateway

Industry-specific tool execution. One instance serves all agents in the industry.

  • Tool registry + /tools/specs
  • /tools/invoke — executes registered tools
  • KB / vector store for RAG
  • Synth data store (dev/test)
  • Tool Admin UI — port 5200
  • Document ingestion + chunk management
Switching environments: the single env var PLATFORM_SERVICES_URL in C1 switches from local (http://platform-services:8080) to VPC or cross-VPC. No code changes.
Request Lifecycle

Pre / In / Post Graph

Every request passes through three phases. C1 orchestrates all three — it calls C2 for each phase and owns only the state transitions between them.
Pre-Graph

Setup — C1 → C2

  • 1. Auth + context build (C1)
  • 2. Config load — agent.yaml, prompts, tool policy (C1 → C2)
  • 3. Memory read — all 4 types (C1 → C2)
  • 4. Pre-graph RAG — silent KB inject (C1 → C2, if enabled)
  • 5. Thread history build from recent_turns (C1)
In-Graph

Reasoning — C2 only

  • 1. Planner (LLM) — decides which tool to call
  • 2. Router — resolves tool input from context
  • 3. Executor — calls tool via C3 gateway
  • 4. HITL gate — returns NEEDS_HITL signal to C1 if required
  • 5. Responder (LLM) — synthesises final answer
Post-Graph

Persist — C1 → C2

  • 1. Short-term write — every turn (C1 → C2)
  • 2. Episodic write — on write-tool success (conditional)
  • 3. Summary write — turn_count / token_threshold (conditional)
  • 4. Response returned to client with memory_trace
HITL signal pattern: C2 executor never submits to the approval store — it returns {"result": "NEEDS_HITL", tool_name, tool_input, risk_level} to C1. C1 owns the approval_store (SQLite) and handles submission, polling, and post-approval tool execution locally.
Memory System

4 Memory Types

Each memory type has independent read and write policies configured in memory.yaml. Reads happen pre-graph. Writes happen post-graph. Each type has a different scope, trigger, and retention purpose.
Built

Short-Term Memory

Last N conversation turns. Written every turn. Read every turn. Scope: thread/conversation. Keeps the agent aware of recent context without reading the full history.

Built

Episodic Memory

Significant clinical events — tool calls, HITL approvals, notes written. Written on write-tool success trigger. Scope: case or assessment. Enables the agent to recall what actions were taken for a specific case.

Built

Summary Memory

LLM-compressed narrative of the conversation. Written on turn_count or token_threshold trigger. Scope: conversation or member. Prevents context window overflow in long sessions.

Built

Semantic Memory

Vector-embedded facts stored for similarity retrieval. Written from agent observations. Read via embedding search. Scope: member or case. Enables cross-session knowledge persistence.

Memory policy overrides: the UI can send memory_policy_override in the request body to toggle individual memory types on/off per message. Used in the Memory tab of the chat panel.
Retrieval Augmented Generation

3-Dimension RAG

RAG is configured per agent in agent.yaml with two independent retrieval stages and three independent configuration dimensions. Each stage and dimension can be set differently.
DimensionOptionsWhere ConfiguredStatus
Dim 1 — Strategysemantic · keyword · hybridagent.yaml retrieval.pre_graph.strategy / planner_tool.strategyBuilt
Dim 2 — Stagepre_graph (silent inject) · planner_tool (LLM decides)agent.yaml retrieval.pre_graph.enabled / planner_tool.enabledBuilt
Dim 3 — Patternnaive · self_corrective · multi_hop · hyde · agenticagent.yaml retrieval.pre_graph.pattern / planner_tool.patternPartial — naive built, others roadmap
Pre-graph RAG runs before the planner — chunks are silently injected into context. The planner never sees the retrieval step. Planner-tool RAG exposes search_kb as a tool the planner LLM calls explicitly when it decides KB retrieval is needed.
Human in the Loop

HITL — Approval Pattern

HITL is risk-level based. Each tool is assigned a risk level in agent.yaml. High/medium risk tools require supervisor approval before execution. The approval flow is entirely owned by C1.

Signal Flow

  • C2 executor detects high-risk tool → returns NEEDS_HITL signal (no state change)
  • C1 receives signal → submits to local approval_store (SQLite)
  • UI polls GET /hitl/status/{approval_id} every 5s
  • Supervisor approves/rejects via POST /hitl/decide
  • On approval: C1 executes tool directly (bypass_hitl=true) + writes episodic memory

Config (agent.yaml)

  • risk.approval_required: true/false — master switch
  • risk.risk_levels — per-tool risk level (high/medium/low)
  • hitl.routing_rules — which risk levels require approval
  • hitl.adapter — internal (SQLite) or external (Pega/Epic, roadmap)
  • hitl.sla.timeout_minutes — approval timeout

Admin Toolchain

Agent Factory & Supporting Services

The admin toolchain scaffolds, configures, monitors, and tests agents — all without touching code directly.

Agent Factory UI — port 5173

  • Create Agent — scaffold form → generates overlay files + registers agent
  • Agent Registry — live flow diagram, config tabs (Memory, RAG, HITL, Tools, Prompts, Routing), workspace controls
  • Prompt Governance — prompt version management
  • Workspaces — start/stop/restart containers
  • Test / Config Lab — edit YAML, run test scenarios, compare results grid
  • Test / Results — all saved test runs, export CSV

Support API — port 8000

  • Scaffold — generate overlay files from templates
  • Registry CRUD — usecase_registry.json
  • Workspace — docker compose up/stop/restart
  • Config Lab API — read/write overlay YAML files
  • Agent config merge — agent.yaml + domain.yaml + memory.yaml

Tool Admin UI — port 5200

  • Tool Registry — add/edit/delete tools, mode, tags, endpoint, schema
  • Knowledge Base — document ingestion, chunk management, delete docs

Capability UI — port 3000

  • care-management: member search, cases, assessments, care plans
  • Inline chat panel — Chat, Summary, Memory, Trace tabs
  • AI Summary panel — scoped summary per member/case/assessment
  • Supervisor view — HITL approval queue
