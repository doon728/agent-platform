About This Document

This is the Customer Delivery Playbook for the Intelligent Agent Accelerator for Healthcare. It describes how delivery teams apply the accelerator to customer engagements — from discovery to handoff. The playbook is itself an artifact this offering produces; every engagement follows it.

For the business case behind the accelerator (opportunity, offering, ROI, build plan, ask), see the Offering Proposal Summary. For platform architecture, per-container detail, deployment model, repo governance, and technical limitations, see the Accelerator Technical Design companion document.

How to read this document: every customer engagement uses the same phased structure and the same discovery motion. Team composition, timelines, and deliverables flex based on scope — the framework does not.
Section 1

Customer Delivery Engagement — How the Accelerator Is Applied

A typical customer delivery engagement runs 12–16 weeks across 5 phases. The accelerator is the starting point — delivery teams fork an engagement copy from the master accelerator repo, tailor it to the customer's systems and workflows, and deploy it into the customer's AWS account. Timeline and team composition flex based on number of agents, integration complexity, and customer environment. Structure below is indicative — final scope, staffing, and pricing are determined per engagement.

Four Modes of Accelerator Use

Not every customer will accept every part of the accelerator into their environment. Healthcare payers have strict policies on what runs in their AWS accounts, what code they accept from outside, and which open-source dependencies they approve. The accelerator is structured so it delivers value across a spectrum — from full deployment down to reference-only use — without losing its core benefit to the delivery team. The mode for a given engagement is agreed during Phase 1 Discovery, after the customer's InfoSec and architecture reviews.

Mode Summary
1. Full deployment Entire accelerator deployed into customer AWS; full ownership transferred at end
2. Partial / hybrid Selected components deployed; others rebuilt or swapped for customer-standard equivalents
3. Reference only Nothing deployed; customer builds their own version using our architecture as blueprint
4. Enablement only No deployment, no code sharing; accelerator used for design, demo, and enablement inside the engagement

Mode 1 — Full Deployment

What the customer receives. The complete accelerator deployed into the customer's AWS account: all four containers (C1 agent runtime, C2 platform services, C3 tool gateway, C4 control plane), the full catalog of six reference agents included in the accelerator, the Agent Factory UI, and all Infrastructure as Code. During the engagement, two to three of the reference agents are selected and tailored to the customer's specific workflows as their production agents; the remaining reference agents are handed over as working examples the customer's AI COE can extend later. At engagement end, the customer receives their engagement repo as a full master copy — platform code, templates, reference agents, IaC — in their own Git organization.

Deployment shape. C1 agent shells deploy on AWS AgentCore Runtime (one per agent). C2 platform services, C3 tool gateway, and C4 control plane deploy on ECS/Fargate within the customer's AWS account. Typically per-LOB multi-account (agent shells per line of business, one shared C2/C3/C4 per tenant), or single-account for smaller customers. All infrastructure deployed via IaC; security evidence package reviewed by customer InfoSec before go-live.

Customer AI COE role post-handoff. The customer's AI COE becomes the equivalent of our Platform Team internally. They own the platform code and templates going forward, can scaffold new agents via the Agent Factory UI on their own, can modify templates to evolve the platform for their internal needs, and can pull upstream releases from the practice (v1.1, v1.2) at their cadence — or freeze on the version they received.

When this mode fits. Customer has AWS-native posture, has an internal AI COE or plans to build one, wants full IP transfer, and InfoSec is comfortable with the security evidence package.

Mode 2 — Partial / Hybrid Deployment

What the customer receives. A subset of the accelerator components, chosen during Phase 1. Common patterns: accept the agent runtime and templates, swap the tool gateway for a customer-standard API gateway; or accept the control plane (Agent Factory UI, observability) but deploy C1 agents inside the customer's existing Kubernetes cluster instead of ECS. Engagement repo handed over with the modifications; customer-standard components are wired in.

Deployment shape. C1 agent shells on AWS AgentCore Runtime, C2/C3/C4 on ECS/Fargate (our defaults); some components may be rebuilt on customer-chosen substrates. Our IaC covers what we deploy; customer IaC teams integrate with their existing stack for the rest.

Customer AI COE role post-handoff. Owns the deployed portion of the accelerator. May also own the custom integration layer where we rebuilt to their standards. Upstream release adoption depends on which components they accepted — fully for accepted ones, selectively for swapped ones.

When this mode fits. Customer has established internal platform standards they cannot or will not displace (their own tool gateway, their own observability stack, their own container orchestration). Accelerator fills specific gaps rather than replacing their full platform.

Mode 3 — Reference Only

What the customer receives. Documentation, architecture diagrams, design patterns, and optionally read-only access to our demo environment. No accelerator code deployed into the customer's environment. Customer's engineering team builds their own equivalent using our reference architecture as a blueprint.

Deployment shape. None in customer environment. Our team may provide design reviews, pattern walk-throughs, and implementation guidance during the engagement; customer does the build.

Customer AI COE role post-handoff. Owns the platform they built. No ongoing upstream relationship with us on platform code.

When this mode fits. Customer has strict prohibitions on accepting outside code (some Medicaid MCOs, some compliance-driven orgs), or has an existing internal platform team that prefers to build everything themselves but wants the architecture patterns de-risked. The accelerator's architecture work compresses their design phase from months to weeks; they spend the engagement fee on guidance, not on deployed code.

Mode 4 — Enablement Only

What the customer receives. Access to our running demo environment (practice-hosted, non-customer data), workshops, design sessions, and strategy guidance. No accelerator code enters the customer's environment in any form; no documentation or architecture content is transferred.

Deployment shape. None. The engagement is scoped as strategy and capability uplift, not as implementation.

Customer AI COE role post-handoff. Customer uses the enablement to make better build-vs-buy decisions and to scope their own internal work.

When this mode fits. Customer is early in their AI strategy, wants senior guidance before committing to a build, or is evaluating whether to use the accelerator in a future engagement. For the delivery team, this mode still uses the accelerator internally for design, demo, and training — no team starts from whiteboard.

The key point: the value of building the accelerator does not depend on any single customer accepting full deployment. Every engagement captures some subset of the value. In the worst case (mode 4), the practice still skips the whiteboard phase, arrives with a credible working demo, and delivers faster than it would without the accelerator.

How the Engagement Begins

Every engagement opens with a focused discovery motion. The goal in the first two weeks is to agree on what we are building, why it matters, and how we will measure success.

  • Pick two anchor workflows. One member-facing (e.g., pre-call assessment, care gap outreach) and one back-office (e.g., prior authorization remediation, claims triage). Each must be painful, frequent, and measurable.
  • Define success metrics up front. Cycle-time reduction, manual-effort reduction, appeal approval rate, cost-per-case, audit-exception rate, care manager throughput — whichever apply to the chosen workflows.
  • Map system-of-record boundaries. Claims core, care management platform, prior-auth system, member database, data warehouse, IAM, logging.
  • Identify tool candidates. Read and write actions the agent needs exposed through the Tool Gateway to automate the workflow end to end.
  • Confirm compliance posture. HIPAA controls, BAA coverage, data-residency expectations, audit-trail requirements.

The output of discovery is a signed-off workflow map, success metrics, and an AWS deployment plan — enough to commit the build phase scope.

Security & Compliance Evidence Comes Bundled

One of the most common customer objections to adopting outside code — especially in healthcare — is "we do not trust code we did not write." The accelerator answers this objection directly by shipping with a complete security and compliance evidence package, produced during the build of each release.

Scans & Tests Run by Practice on Every Release

  • SAST / DAST / SCA — static, dynamic, and dependency analysis automated in CI for every v1.x release
  • SBOM — full software bill of materials generated per release; dependencies vulnerability-scanned against CVE feeds
  • Container image scans — Trivy or equivalent run on every image before tag
  • Secrets & credential scans — pre-commit hooks plus repo-wide scans on every PR
  • Internal penetration test — performed against each major release; findings remediated before release is tagged

Documentation Bundled With the Release

  • HIPAA controls matrix — every control, how the accelerator implements it, where it is enforced in code
  • Architecture security review — threat model, data flow diagrams, encryption, auth, and audit documentation
  • Scan reports — signed, version-tagged, tied to the exact code being deployed
  • BAA-eligibility evidence — every AWS service used is HIPAA-eligible under BAA
  • Pen-test report with remediation trail
What this changes for the engagement: the customer's InfoSec team receives the full evidence package at kickoff. Their review validates work already completed rather than uncovering issues from scratch. If they require their own penetration test on top (many payers do), they start from a known-clean baseline. Approval cycles typically compress from weeks or months to days.

How the Engagement Team Works in the Repo

The delivery team does not start from a blank repo. At kickoff, the Platform Team creates an engagement repo — a fork of the master accelerator repo, dedicated to this customer. All assigned developers clone this single shared repo and work in it together for the duration of the engagement.

  • Engagement repo is access-controlled — only the assigned team has visibility
  • Forking to personal namespaces is disabled; all work happens on branches of the shared repo
  • main is branch-protected; changes go through pull requests with required reviews
  • Infra changes in the iac/ folder get their own reviewer requirements via CODEOWNERS
  • When the practice ships a new accelerator release (v1.1, v1.2, …), the engagement architect can pull upstream improvements at a sprint boundary

For the full repo governance model — including how the master accelerator repo is maintained and how individual developers access it for exploration — see the Accelerator Technical Design companion document.

Pattern Handoff at Engagement End

At the end of the engagement, the customer receives more than code. They inherit the entire Git governance model — the repo structure, branch protection rules, CI/CD templates, CODEOWNERS, exploration sandbox pattern, and the update path for pulling future releases of the accelerator. The customer leaves the engagement with a running agent platform and a way to operate it sustainably.

What the Customer Inherits

  • Master copy of the engagement repo in their own Git organization — their versioned gold copy
  • Branch protection and PR workflow already configured for their own change governance
  • Exploration sandbox pattern — their developers can POC new agents safely, off the master
  • Upgrade path — they can pull future releases of the accelerator on their own cadence
  • CI/CD templates they own and can extend for their pipelines

Why the Handoff Model Matters

  • The customer is not left with a black-box deployment — they know how to operate it
  • Creates a natural continuing relationship — they pull our accelerator upgrades as we release them
  • Reduces the customer objection of "what happens after you leave?"
  • Turns accelerator release cadence into a recurring customer touchpoint
