Section 1

Executive Summary

The Intelligent Agent Accelerator for Healthcare is a pre-built, reusable platform and delivery asset designed for healthcare payer AI engagements. It ships with enterprise-ready pre-built agents for core payer workflows — and the scaffolding to rapidly create new agents tailored to each customer's specific requirements. Compliance controls and a security evidence package come bundled — built once by the practice, improved continuously, and used by every delivery team and developer across the practice.

The ask. Approve this proposal. Delivery engagements shift from bespoke builds to accelerator-based delivery. Customers reach production agents in weeks instead of months. The practice gains a repeatable, sellable healthcare payer AI service.

While the accelerator is built for healthcare payers first, the architecture is domain-agnostic. The same platform and patterns can be adapted to financial services, insurance, and other regulated industries with similar compliance, auditability, and integration requirements — extending the practice's return beyond healthcare.

Section 2

Opportunity

The Intelligent Agent Accelerator for Healthcare solves the bespoke-build problem. Any new agentic AI engagement should not start from a blank repo — rebuilding platform capabilities (orchestration, memory, tool gateways, HITL, compliance controls) before the actual workflow begins. The accelerator provides a pre-built foundation every engagement starts from: architectural patterns proven in a reference implementation, reusable platform components, and pre-built agent templates — creating a repeatable healthcare payer offering for the practice.

Practice

The accelerator gives delivery teams an engineered starting point. Repeatable architectural patterns, reusable platform components, a policy-as-code approach to compliance (HIPAA controls, audit logging, PHI handling) and governance (memory scopes, tool access, HITL gates, prompt lifecycle), and reusable delivery assets compress delivery time and standardize quality across engagements. The practice gains a repeatable healthcare payer AI offering it can position and sell consistently.

Customer IT

Payer engineering teams have AI infrastructure and models available but lack a repeatable approach for building agent systems at scale. The accelerator delivers a reference architecture exercised against a working agent and a reusable platform foundation, so their team focuses on workflow implementation rather than platform construction.

Customer Business

High-value workflows — care management outreach, prior-auth remediation, claims triage — are well-suited for AI automation but remain constrained by manual effort and integration complexity. The accelerator enables AI agents to be built and deployed into these workflows faster, with compliance patterns built into the platform.

Section 3

The Offering

This offering produces the Intelligent Agent Accelerator for Healthcare — a curated set of reusable, enterprise-ready artifacts built once and applied across every customer engagement.

What the Offering Produces

Artifact What It Is
Container platform (C1–C4) Four deployable, secured containers: agent runtime (thin shell, runs on AWS AgentCore Runtime), platform services (LangGraph-based, runs on ECS/Fargate, shared per tenant), tool gateway (runs on ECS/Fargate, shared per tenant), control plane (runs on ECS/Fargate, per tenant). Each container ships with least-privilege IAM, encryption at rest and in transit, audit logging, and an evidence trail tied to the release version. Same code runs locally under docker-compose.
Backend adapters Memory, RAG, and graph backends for AWS-native stores (pgvector, OpenSearch, Bedrock Knowledge Bases, DynamoDB, S3, Neptune) behind a single swappable interface.
Infrastructure as Code Terraform modules and AWS CDK stacks for the full AWS footprint — AgentCore, ECS/Fargate, RDS, ElastiCache, OpenSearch, Secrets Manager, KMS, CloudTrail.
CI/CD pipelines Build, test, eval, security scan, ECR push, and environment promotion templates.
Agent templates Overlay-pattern templates covering the agent-type × reasoning-strategy matrix — seven agent types combined with five reasoning strategies. Operators scaffold new agents by picking the template that matches.
Agent Factory UI Admin web UI to scaffold, configure, govern, and observe agents without writing code.
Authentication & authorization patterns User authentication (SSO / Cognito / OIDC), service-to-service auth across the four containers, tool authentication to external payer systems, agent-level authorization (allow-listed tools, scoped memory access), and per-tenant scope enforcement for multi-LOB deployments. Applied at every layer of the platform.
Compliance patterns PHI masking, HITL approval gates, audit logging, KMS encryption, IAM least-privilege, BAA-ready service selection — reusable modules, not documentation.
Security & compliance evidence package Bundled with every release: SAST / DAST / SCA scan reports, SBOM, container scans, secrets scans, internal penetration test, HIPAA controls matrix, architecture security review. Customer InfoSec reviews become validation, not discovery.
Six pre-built reference agents Six end-to-end working healthcare payer agents, each a different agent type × reasoning strategy, each with its own UI. Demo-ready and engagement-ready.
Delivery Playbook Runbook for how engagement teams apply the accelerator to customer delivery — discovery motion, phase structure, team composition, customer security review process, engagement repo patterns, and handoff procedures. Every engagement team follows the same playbook.
Demo + Demo Script A working end-to-end demo environment plus a presentation script. Used in customer pitches, internal enablement, and RFP responses. Keeps every customer conversation anchored to the same credible working demonstration.

Industry Alignment

The accelerator's core architectural choices match the reference architectures published by the major agentic AI platform vendors:

  • Per-agent runtime + shared tenant services — the model used by AWS Bedrock AgentCore. Each agent deploys as its own containerized runtime with unique ARN and versions; Memory, Gateway, Identity, Observability are shared account services. AgentCore is HIPAA-eligible as of February 2026. (runtime docs, HIPAA compliance)
  • Developer configures; platform is read-only — the model used by AWS AgentCore (Memory, Gateway, Identity, Observability consumed as managed services) and Microsoft Azure AI Foundry Agent Service (agents "defined entirely through configuration — instructions, model selection, and tools"). (AgentCore overview, Foundry Agent Service)
  • Agent-type × reasoning-strategy overlays — builds on Microsoft Foundry's Prompt / Workflow / Hosted agent-type taxonomy and established reasoning-strategy patterns (ReAct, plan-and-execute, multi-hop, reflection) implementable as graph patterns in LangGraph. (Foundry agent types, LangGraph Platform)

Full citation table — including Google Vertex AI Agent Engine, Anthropic Claude Agent SDK, Salesforce Agentforce, and detailed constraints — is in the Accelerator Technical Design companion document.

Built on AgentCore, Not a Replacement for It

The accelerator consumes AWS AgentCore primitives (runtime hosting, memory, gateway, identity, observability) where they fit and wraps them where flexibility is needed. What we plan to build is the healthcare-payer vertical layer on top: pre-built reference agents, the Agent Factory UI with healthcare scaffolding, the HITL engine AgentCore does not ship, the compliance evidence package, and the delivery playbook. A detailed "what we plan to build vs. what AgentCore provides" table is in the Accelerator Technical Design companion document.

Accelerator Differentiation

The architectural patterns are shared with AWS AgentCore. The accelerator's differentiation is what sits on top of them:

  1. Vertical packaging — healthcare payer-specific patterns, workflows, and compliance posture. AgentCore is domain-agnostic; this accelerator ships with payer context baked in.
  2. Delivery-integrated — comes with a Delivery Playbook, demo + demo script, security evidence package, and pre-built reference agents. Not code alone; a full engagement-ready asset.
  3. Pre-built agent matrix — six working agents across agent-type × reasoning-strategy combinations, each with its own UI. Ready to demo, ready to tailor to customer workflows during engagement.
Technical detail: Platform capabilities, reference architecture, per-container breakdown, deployment model, repo governance, LLM keys/cost controls, technical limitations, and current prototype state — see the Accelerator Technical Design companion document.

Lens C — Customer Business (Flagship Use Cases)

Pre-Call Assessment for Care Management

Problem. Care managers making outreach calls spend 20–40 minutes per call on preparation — pulling claims, reviewing care plans, scanning notes, identifying gaps. Most of that time is information assembly, not clinical work.

Agent. Before the call, the agent assembles member history, open gaps, risk flags, and relevant payer guidelines. HITL gate lets the care manager accept, edit, or reject before it is written to the case.

ROI levers: prep time per call reduced · outreach volume per care manager per day increases · care plan note quality improves · rework from missed information declines.

Prior Authorization Denial Remediation

Problem. PA denials require manual review of denial reason, documentation gaps, policy rules, appeal drafting, and resubmission — often across multiple systems, often missing deadlines.

Agent. On denial, the agent classifies the reason, identifies missing documentation, pulls clinical policy criteria, assembles the appeal package, and routes for clinical review. HITL holds before submission.

ROI levers: appeal turnaround time reduced · approval rate improves · manual effort per denial reduced · missed-deadline leakage reduced.

Other Supported Workflows

  • Care gap closure — identify, prioritize, draft plan updates
  • Clinical summarization — structured summaries of visit notes and case histories
  • Risk monitoring — event-driven alerts on threshold breaches
  • Claims triage — extract, validate, route, flag anomalies for review
  • Member engagement — self-service eligibility, benefits verification
Section 5

Accelerator Use

The accelerator serves two audiences: the practice's own developers and architects, and customers during delivery engagements. Each audience uses the accelerator differently; each has its own companion playbook covering the detail.

Internal Use

Practice developers and architects use the accelerator as soon as it is released — not only during customer engagements. Typical internal uses:

  • POCs and exploration — create a sandbox repo from the master template, run the platform locally with docker-compose up, scaffold agents through the Agent Factory UI in minutes
  • Pre-sales demos and RFP responses — use the pre-built reference agents and working demo script to show capability without building from scratch for every conversation
  • Design and prototype work ahead of engagements — validate customer-specific agent designs against the platform before engagement kickoff
  • Real-runtime validation — optional deploy to the practice's playground AgentCore sandbox to test against real AWS managed services (AgentCore Runtime, Memory, Identity, Gateway, Observability, and Bedrock) before customer deployment
  • Platform improvement — developers contribute improvements back to the master repo as pull requests; the Platform Team reviews and ships them in the next v1.x release
Internal detail — Platform Team ownership model, master repo governance, local development workflow, playground AgentCore usage, release cadence, LLM key management for internal use — is in the Internal Playbook companion document.

Customer Use — Four Modes

Not every customer will accept every part of the accelerator into their environment. Healthcare payers have strict policies on what runs in their AWS accounts, what code they accept from outside, and which open-source dependencies they approve. The accelerator is structured so it delivers value across a spectrum — from full deployment down to reference-only use — without losing its benefit to the delivery team.

Mode Deployed to Customer Customer AI COE Post-Handoff
1. Full deployment Entire accelerator (all four containers, IaC, agents, admin UI) in customer AWS; engagement repo handed over as their master copy Owns platform code and templates. Scaffolds new agents independently. Pulls our upstream releases or freezes on the version they received.
2. Partial / hybrid Selected components accepted; others rebuilt or swapped for customer-standard equivalents Owns the deployed portion plus the custom integration layer. Upstream release adoption depends on which components they accepted.
3. Reference only Nothing deployed; customer builds their own version using our architecture as blueprint Owns what they built. No ongoing upstream relationship with us on platform code.
4. Enablement only No deployment, no code sharing; used for design, demo, and enablement No code ownership. Customer uses the enablement to inform their own build-vs-buy decisions.
Why this matters: the value of building this accelerator does not depend on any single customer accepting full deployment. Every engagement captures some subset of the value — and in the worst case (mode 4), the practice still skips the whiteboard phase and delivers faster.
Per-mode detail — what exactly the customer receives, deployment shape, AI COE role, and when each mode fits — is in the Customer Delivery Playbook companion document.
Section 6

Internal Build Plan

Building the accelerator is what this proposal is asking to fund. The build is not starting from zero — a working prototype is already in place. The four-container platform, the core shared library (memory, RAG, HITL, tool gateway, observability, prompt management), one end-to-end reference agent (pre-call assessment for care management), and a capability UI for care management are running today. Major architectural decisions — runtime split, agent-type × reasoning-strategy model, policy-as-code approach, HIPAA posture — are locked and documented. Full detail of what exists today is in the Accelerator Technical Design companion document.

The remaining work is to productize — complete the runtime split, add the missing backend adapters, implement remaining reasoning strategies, expand from one to six reference agents (each with its own UI), add Infrastructure as Code and CI/CD, implement PHI masking as an enforced layer, produce the security evidence package, and author the delivery playbook and demo script.

With a planned team of two full-time offshore developers plus part-time architect, security, SME, technical writer, and team-lead support, the build runs roughly 18–22 weeks across seven phases. Phases overlap. Final scope, team, and duration are subject to sizing during Phase 1.

Build Phases

Phase Duration Deliverable
Phase 1 — Roadmap Weeks 1–2 Product roadmap defined: what goes into v1.0, what is deferred to v1.1 and beyond, feature priorities, sequencing, and dependencies. Sets the scope the rest of the phases execute against.
Phase 2 — Design Finalization Weeks 3–4 Remaining architecture gaps closed: final runtime-split wiring decisions, IaC pattern selection, backend adapter priorities, auth model confirmation, release and governance conventions locked. Most design work leverages what already exists in the prototype.
Phase 3 — Core Platform Completion Weeks 4–12 Runtime split completed. Backend adapters beyond file backend (S3, DynamoDB, pgvector, OpenSearch, Redis) implemented. Remaining reasoning strategies (multi-hop, reflection) added. Auth hardened across the four containers. IaC (Terraform + CDK) and CI/CD pipelines built out.
Phase 4 — Reference Agents Build-Out Weeks 10–17 Expand from one working reference agent to six. Each a different agent-type × reasoning-strategy combination. Each with its own UI. All running end to end against realistic payer data.
Phase 5 — Security & Compliance Evidence Weeks 14–19 PHI masking enforced as a platform layer (not just prototype scope). SAST / DAST / SCA automated in CI. SBOM generated per release. Internal penetration test completed with remediation trail. HIPAA controls matrix documented. BAA-eligibility evidence compiled.
Phase 6 — Delivery Playbook & Demo Weeks 16–21 Delivery playbook written. Demo environment stood up with working demo script. Internal enablement materials produced.
Phase 7 — v1.0 Release & Rollout Weeks 21–22 Accelerator tagged as v1.0. Master repo published with branch protection, CODEOWNERS, release notes. Practice-wide enablement session. First design-partner customer engagement kickoff.

Investment

Direct cost is the team allocation across 18–22 weeks, plus tooling (AWS sandbox, Bedrock access, third-party penetration test engagement, security scanning licenses). Because the core platform is already prototyped, the remaining investment is disproportionately weighted toward productization (backend adapters, IaC, CI/CD), reference agent build-out (one to six), and the compliance + delivery artifact package.

Payback reminder (see Section 4): The build investment is absorbed across subsequent engagements. Each engagement that uses the accelerator begins from a hardened foundation rather than a blank repo.
