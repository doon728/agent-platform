Internal Architecture Document

Enterprise AI Agent Platform
Architecture & Capabilities

An enterprise-grade AI agent platform built for healthcare payers. Agent behavior — tools, memory, HITL, RAG, prompts — is fully configurable per agent via YAML and the Admin UI. Domain integration (wiring your data hierarchy into the context layer) is a one-time engineering step per use case. Built for environments where memory, human oversight, and reliable tool execution are not optional.

Designed to be AWS AgentCore compatible from day one — the orchestration engine swaps to AgentCore without touching Memory, RAG, HITL, or Tool Gateway. Platform capabilities run unchanged.

Section 1

What This Platform Is

Most AI agent frameworks give you an LLM call and a tool loop. This platform gives you the full production stack: memory that persists across sessions, retrieval that grounds responses in your clinical content, human approval gates before high-risk actions execute, a governed tool registry, and full observability into every decision the agent makes — all configurable per agent without touching code.

Core design principle: every capability is built once in the platform layer and configured per agent. Turning on episodic memory, enabling pre-graph RAG, or wiring HITL to Pega requires a config change — not a new deployment. Agents are configurations on top of a shared capability platform.

Reference Use Case — Healthcare Care Management

The platform is demonstrated with a healthcare care management agent serving nurses. The agent helps nurses navigate member records, case history, assessments, and clinical tasks through natural language — while ensuring that any action with clinical risk is routed through a supervisor approval workflow before it executes.

The patterns demonstrated here apply equally to any regulated industry: financial services, insurance, appeals management, prior authorization. Only the domain data and clinical tools change. The platform capabilities are identical.

⚙️
workflow_agent
Multi-step process · fixed graph · triggered by event or schedule
Roadmap

Post-discharge workflow: Step 1 — fetch discharge summary. Step 2 — run risk assessment. Step 3 — if high risk, create urgent care plan; if low risk, schedule follow-up call. Step 4 — send notification. Each step is a LangGraph node. The graph IS the business logic. Unlike chat_agent, the user doesn't drive the flow — the graph does.

Memory — optional per step
Steps that write clinical data can trigger episodic memory writes. Steps that need prior context can read short-term. Memory is not needed for every workflow — only where continuity matters across runs.
HITL — optional per step
High-risk steps in the workflow (e.g. "create care plan") can require approval before the step executes. Lower-risk steps proceed automatically. Same HITL adapter pattern — config-driven per tool.
RAG — optional
Steps that need to reference clinical guidelines or policies can call search_kb. Not typically needed in structured process workflows — data comes from system tools, not KB.
Tool Gateway
Each workflow step calls one or more tools via the same Tool Gateway URL dispatch. No planner LLM needed — the graph determines which tool each step calls. Tools are still validated and routed through the gateway.
🎯
supervisor_agent
Multi-domain orchestrator · routes to specialist sub-agents · one conversation
Roadmap

A healthcare assistant that handles both clinical and billing questions. User asks "what's my copay?" — supervisor routes to billing_agent. User asks "what are the member's medications?" — supervisor routes to clinical_agent. The user experiences one conversation. Underneath, specialist agents each have their own tools, memory, and HITL config.

Memory — shared thread
The supervisor maintains a shared thread_id across all sub-agent handoffs so short-term context is preserved. Each sub-agent can have its own memory scopes — the supervisor orchestrates the conversation but doesn't own the domain memory.
HITL — delegated to sub-agents
Each sub-agent has its own risk_levels and HITL config. The supervisor doesn't handle approvals — it delegates to the sub-agent which owns that domain. Billing_agent approves billing writes; clinical_agent approves clinical writes.
RAG — delegated to sub-agents
Each sub-agent has its own KB access and RAG config. Clinical_agent searches clinical_kb; billing_agent searches formulary_kb. The supervisor routes the message — the sub-agent owns retrieval for its domain.
Tool Gateway
Sub-agents each call their own allowed tool sets through the shared Tool Gateway. The supervisor itself may call lightweight routing tools (e.g. classify intent). All sub-agent tool calls still go through URL dispatch and schema validation.
These are not mutually exclusive — they coexist in one repo. The care management repo today runs both chat_agent and summary_agent under separate overlay folders. They share the same Tool Gateway, the same memory backend, and the same platform runtime. Only the overlay graph differs. A future repo could add workflow_agent for post-discharge automation alongside the existing chat interface.

Platform Architecture

The platform has five layers. Every user message passes through all five in order. The boundaries are strict — the agent runtime never touches the backend directly, tools only execute through the gateway, and memory writes never block the user response.

Figure 1 — Platform layers. Numbered arrows show the sequence of calls for a single user message. Down arrows (①③⑤) show requests; up arrows (②④⑥) show responses. Dashed amber arrow (⑦) = Memory READ before graph execution. Dashed green arrow (⑧) = Memory WRITE after graph execution, async — does not block the response. Both connect Layer 2 to the Memory Store in Layer 4.

Execution Flow — Chat Agent

This execution flow is specific to the chat_agent — the conversational agent that routes user messages to tools via an LLM planner. Every message through a chat agent follows this exact path.

Note — execution model varies by agent type. The platform provides shared capabilities (Memory, RAG, HITL, Tool Gateway, Prompt Governance) but each agent type defines its own execution model — how it uses those capabilities and in what order. The summary_agent uses a fixed parallel tool call pattern with no LLM planner and no HITL. Future agent types (workflow_agent, supervisor_agent) will each have their own distinct execution model. When evaluating or extending the platform, always identify the agent type first — the execution flow follows from that.

Scope Hierarchy & Rollup

Memory scopes form a hierarchy. When a nurse opens a case, episodic memory automatically rolls up from all assessments under it. When they open a member, it rolls up across all cases and assessments.

Figure 3 — Memory scope hierarchy. Child scopes register to parents at write time. Retrieval traverses up the tree automatically.

Runtime Toggle — Per-Message Policy Override

The Memory Panel in the chat UI lets users toggle each memory scope on/off per session. This is a runtime override — it does not change the agent's config file. The override affects both read AND write for that turn.

Current limitation: read and write are toggled together. A future improvement separates them: memory.read.enabled and memory.write.enabled as independent controls. Critical for the Summary Agent, which should read episodic history but must never write to it.

RAG / Retrieval

RAG grounds the LLM in your content — clinical protocols, care management policies, formularies, guidelines. The platform has two RAG stages and three independent configuration dimensions. Changing one dimension does not affect the others.

Three Independent Dimensions

DimensionControlsOptionsToday
Dim 1 — StrategyHow retrieval works inside one KB toolSemantic (vector) · Hybrid · Keyword (BM25) · Graph RAGSemantic only
Dim 2 — KB SelectionWhich knowledge base(s) to querySingle KB · Multi-KB with router · Fan-out + re-rankSingle KB
Dim 3 — PatternHow retrieval integrates into reasoningNaive · Self-Corrective · Multi-Hop · HyDE · AgenticNaive (implicit)

Two RAG Stages

🔄
Stage 1 — Pre-Graph RAG
Ambient enrichment · every turn · silent
1
Triggered by config
If retrieval.pre_graph.enabled: true in agent.yaml, fires automatically on every turn before graph starts.
2
Embed user message
User's raw message → OpenAI Embeddings API → 1536-dim vector.
3
Vector search in KB
pgvector cosine similarity → top-k chunks above threshold.
4
Inject as background
Chunks injected into ctx[rag_context]. Planner + Responder see them silently. User never sees this happen.
When to use: KB has protocols / guidelines that should inform every response regardless of what user asks.
Tune: top_k low (2-3), threshold high (0.5+)
🎯
Stage 2 — Planner Tool RAG
Explicit query · intent-driven · LLM decides
1
Triggered by LLM decision
Planner LLM decides user is asking a direct KB question → selects search_kb as its tool.
2
Executor calls Tool Gateway
Fires POST /tools/invoke with search_kb tool and the user's query as input.
3
Same retrieval process
Embed → pgvector search → threshold filter → return top-k chunks.
4
Chunks become the answer
KB content is the primary response for this turn. Responder formats it for the nurse. User sees KB answer directly.
When to use: always on — search_kb in allowed tool list. LLM decides when.
Tune: top_k higher (5), threshold lower (0.35)
Both stages can be active simultaneously. Pre-graph provides ambient enrichment in the background. Planner tool handles cases where KB content IS the explicit answer. They are independent — each has its own top_k, threshold, and (future) pattern config.

Human-in-the-Loop (HITL)

HITL is not gating a database write — it is gating the agent's judgment. The human approver validates whether the AI's decision to take an action was correct given clinical context. The agent proposes. A human decides. Only then does execution happen.

Normal Path vs HITL Path

1

2
Executor checks risk → low → proceed immediately
3

4
Responder formats result → response returned to nurse
5
Memory written post-graph. Done.

1
Planner selects tool (e.g. write_case_note)
2
Executor checks risk → high → store approval request in HITL Store. Tool Gateway NOT called yet.
3

—
Async gap — nurse can continue working. Supervisor reviews in Admin UI.
4
Supervisor: POST /hitl/decide → approved or rejected
5
Approved: Tool Gateway executes. Rejected: Tool Gateway not called. Either way →
6
Responder (call 2) → final outcome message returned. Episodic memory written either way.

The Tool Gateway is a pure router with no business logic. Every tool is an endpoint URL. Change the URL and you change where execution happens. Nothing else in the platform needs to know.

Section 13 — Architecture Direction

Multi-Agent & Workflow Architecture

The platform today is single-agent per request. The design direction for multi-agent workflows and workflow orchestration is captured here — not yet built, but the decisions are made. The guiding constraint throughout: a new domain or use case must require only new YAML config files, zero platform code changes.

Status: Design documented. Implementation is roadmap. All decisions below are directional — the goal is to validate the architecture before building.

The Core Problem — Single Agent vs Multi-Agent Context

Today a single agent receives a flat context dict ({member_id, case_id, assessment_id}) and resolves memory scopes from it in one shot. Multi-agent breaks this: a supervisor may start with only member_id, spawn a sub-agent that discovers case_id via a tool call, which spawns another that discovers assessment_id. Context grows across agent boundaries. Memory writes must land at the right scope level for each agent. The platform must handle all of this through config, not code.

Design Decision 1 — Context Propagation: Additive Model

Chosen: Additive

Context flows forward through the workflow and can only grow — each step adds fields, never overwrites. A sub-agent that discovers assessment_id enriches the shared context. The next step downstream sees the full enriched context automatically.

Why: Deterministic, auditable, debuggable. Hidden state mutation bugs are impossible by design.

Rejected: Mutable shared state

Any step can read and overwrite any field. More flexible but creates hidden coupling — step 3 silently breaks when step 1 changes a field name. Unacceptable in regulated workflows.

Design Decision 2 — Per-Agent Primary Scope

Each agent declares its primary_scope — the level of the domain hierarchy it operates at — in its own agent.yaml. Memory reads and writes are resolved against that agent's declared scope schema. Same memory store, different scope resolution per agent. No code change needed — only config.

Each agent's scope schema is declared in its own agent.yaml. An agent may read upward in the hierarchy (e.g. assessment agent also reads case-level context) — configured explicitly, not assumed.

New domain, zero code: A claims platform (policy → claim → item) defines three agents with three scope schemas in three agent.yaml files. scope_resolver.py reads each agent's config at runtime. No Python changes.

Design Decision 3 — Workflow Definition: Config-Driven Fixed Graph

For regulated industries, workflow routing must be deterministic — not decided by an LLM at runtime. The supervisor LLM decides what question to ask, not which agent to invoke. Agent routing is defined in config.

Design Decision 4 — HITL Across Agent Boundaries

Approval is tool-scoped, not agent-scoped. Each sub-agent has its own approval queue entry when it calls a high-risk tool. The supervisor waits (LangGraph interrupt) until all pending sub-agent approvals resolve before synthesizing a final response. This is the parallel approvals pattern applied at the workflow level — multi-agent makes it a hard requirement, not an optimization.

The audit log records which agent in the workflow triggered each approval — full traceability across the entire multi-agent execution.

Key differences from chat_agent: No Phase 1 (pre-graph), no LLM planner, no HITL, no episodic memory write, no trace graph in UI. The only memory interaction is reading and writing summary_cache. Triggered by scope (assessment/case/member), not by a user chat message.
