Core Capabilities
Five Platform Capabilities
Every capability is config-driven, adapter-pattern ready, and independent of any specific domain.
Memory System
Four types · Scope-aware read/write · Toggle per session
Built
Short-term — thread turn history. LLM sees last N messages per conversation.
Episodic — clinical events written per turn. Scoped to assessment / case / member.
Semantic — long-term facts about the member. Persists across cases and time.
Summary Cache — AI-generated summaries. 30-min TTL. Served by summary_agent.
Read/Write split — independent toggles. summary_agent reads but never writes. Roadmap
RAG — 3 Independent Dimensions
Strategy · Stage · Pattern — each configured independently
Partial
Dimension 1 — Strategy — what backend and how to retrieve. Vector, keyword, graph, hybrid.
Dimension 2 — Stage — pre-graph (inject before LLM) or planner tool (agent decides when to retrieve).
Dimension 3 — Pattern — naive · self-corrective · multi-hop · HyDE · agentic. Roadmap
Multi-KB routing — fan-out across multiple knowledge bases, merge + re-rank. Roadmap
HITL — Human in the Loop
Agent proposes · Human approves · System of record executes
Built
Risk levels in agent.yaml — high/medium/low per tool. Triggers approval automatically.
Approval queue + Supervisor UI — async. Agent waits. Supervisor approves or rejects with reason.
Memory write after decision — approval and execution written to episodic memory automatically.
Adapter pattern — one-file swap to Pega / ServiceNow / Epic. Agent never touches system of record. Roadmap
Dynamic risk scoring — content-aware classification, not just keyword matching. Roadmap
Tool Gateway
Registry · Lifecycle · Governance · Schema-first
Built
Tool registry — name, description, tags, mode (read/write), primary_arg, URL endpoint.
Dynamic LLM schema — tool descriptions injected into planner context automatically from registry.
Mode-based HITL trigger — write-mode tools auto-trigger HITL based on risk config.
Bucket hierarchy — Industry → LOB → Region. Agent only sees tools scoped to its bucket. Roadmap
Admin UI — add/edit/delete tools, upload KB docs, manage embeddings — no code. Roadmap
Context Scopes
Domain-aware · Hierarchy-driven · Preload strategy
Partial → Roadmap
Today — hardcoded to care management (assessment → case → member). Works in production.
domain.yaml — capability defines all possible scopes, ID fields, hierarchy once. All agents inherit. Roadmap
active_scopes in agent.yaml — each agent declares which scopes it resolves and preload strategy (always / conditional). Roadmap
Generic test UI — fetches /config/scopes, renders ID input fields dynamically. Works for any capability. Roadmap
Guardrails
Input · Output · Tool call — three interception points
Roadmap
Input guardrails — PII detection, prompt injection blocking, out-of-scope request filtering.
Output guardrails — hallucination check, PII redaction, clinical fact validation.
Tool call guardrails — ID validation, scope mismatch blocking, HITL bypass prevention.
Adapter-compatible — plug into Bedrock Guardrails, Azure Content Safety, or AgentCore guardrail layer.

Repo Architecture
New Repo Structure — App vs Agent Shell
Capability owns the domain. App owns the UI. Agent shell is thin, generated, independently deployable.
📁 agent-platform / (root)
├── platform/ ← platform core (never changes) ├── templates/ │ ├── agent-runtime-template/ │ └── capability-ui-template/ ← generic test UI only ├── platform-tools/ ← Agent Factory + Support API ├── care-management/ ← capability folder └── claims/ ← another capability
📁 care-management / (capability)
├── cm-hero-fl-app/ ← permanent app (not generated) │ ├── capability.yaml ← capability definition │ ├── domain.yaml ← ALL scopes defined here │ └── ui/ ← nurse UI (members, cases...) │ └── src/pages/ └── agents/ ← generated agent shells ├── pre-call-assessment/ ← thin shell │ ├── agent.yaml │ ├── docker-compose.yml │ └── services/ui/ ← generic test UI └── post-discharge/ ← another agent shell
⚡ App vs Agent Shell — Key Distinction
App (permanent)
Owns domain.yaml · Owns capability UI · Never generated · Never deleted · Own git history · Deploys to CDN
Agent Shell (generated)
Owns agent.yaml + prompts · Generated by Agent Factory · Independently deployable · Can be deleted & regenerated · Deploys to container
Contract between them
UI sends scope IDs in POST payload to /invocations — that's it. UI and agent deploy independently.
