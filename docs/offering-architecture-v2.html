Reference Architecture — Intelligent Agent Accelerator for Healthcare

AWS Native

How This Architecture Works

Container 1 — Agent Runtime

Thin shell. One per LOB / AWS account.

Container 1 is a lightweight service — the only component that runs inside the LOB's own AWS account, co-located with its data. It runs three stages: pre-graph (load context, validate input, apply access policy), in-graph (hands off to Container 2 for all reasoning — no logic lives here), and post-graph (PII/PHI masking, HITL gate evaluation, response delivery). Being thin means deploying it per LOB or per environment is cheap and risk-free — there is no duplication of intelligence, only a data boundary.

Container 2 — Orchestration Engine

All the intelligence. Shared. Centrally managed.

Container 2 runs on AWS AgentCore with LangGraph. This is where agent reasoning executes — the selected strategy (ReAct, plan-execute, reflection, multi-hop) runs here, calling memory, RAG, and tools. When Container 1 sends a request, it includes an Agent ID and LOB namespace. Container 2 looks up that agent's configuration from Container 4 — its specific prompts, RAG config, tool list, reasoning strategy — and executes with exactly those settings. A Claims extraction agent and a Care Management chat agent share the same engine but run completely different configurations.

MCP Server + Container 3 — Tool Gateway

Per-LOB tool catalog. Per-agent tool selection.

The MCP Server is a shared protocol layer — it receives tool call requests from Container 2 and routes them to Container 3 using the Model Context Protocol, enabling third-party agent interoperability. Container 3 (Tool Gateway) holds a separate tool catalog per LOB: Care Management gets EHR and care plan tools, Claims gets adjudication and audit tools, Member Services gets enrollment and benefit tools. Within each LOB's catalog, each agent has an allow/deny list — a claims extraction agent cannot call a care plan writer. All tool calls route back to LOB data in Container 1's account via AWS PrivateLink.

Container 4 — Control Plane

One governance system. Logically isolated per agent and LOB.

Container 4 is deployed once and governs every agent across all LOBs. It stores each agent's configuration — keyed by Agent ID and LOB namespace — in S3 (YAML configs) and RDS (registry metadata). Care Management prompts, Claims tool policies, and Member Services HITL rules are all stored separately and served on demand to Container 2. The Agent Factory UI lets each LOB team manage only their agents. Observability spans all LOBs in one dashboard with role-based access. When a new LOB deploys Container 1, it self-registers with Container 4 on first boot — no manual wiring required.

Why this architecture is the right call

Data stays in its compliance boundary

Each LOB's patient data, claims records, and member data never leave their AWS account. Container 1 is the only component that touches that data — and it lives inside the LOB's account. Satisfies HIPAA, per-LOB audit isolation, and enterprise security requirements simultaneously. Tool Gateway reaches back via PrivateLink — still no data crosses the public internet.

No intelligence duplicated per LOB

Container 1 is thin — deploying it per LOB costs almost nothing in engineering or operational complexity. All platform intelligence (reasoning, memory, RAG, tool routing) lives in Containers 2 and 3. A new LOB gets a full-featured agent platform by deploying one thin runtime and registering their config in the Control Plane — not rebuilding the engine.

One place to govern everything

Prompt updates, new tool policies, reasoning strategy changes — done once in Container 4, reflected immediately across every agent, every LOB. One audit trail. One dashboard. One scaffold to onboard a new LOB. The architecture makes central governance the default — not an afterthought bolted on after five separate deployments.
