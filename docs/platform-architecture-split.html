Deployment Model

Three Containers, Three Responsibilities

Each container has a distinct deployment scope and responsibility. C1 is per-agent per-customer. C2 is shared across all agents for a customer. C3 is shared across all customers in an industry.

C1 — Per Agent · Per Customer

Agent Runtime Shell

Thin FastAPI shell. Owns auth, context, HITL approval state, and HTTP client orchestration. Zero business logic.

  • Auth + tenant isolation
  • Context build + scope hydration
  • LangGraph runner (pre/in/post phases)
  • HITL approval_store (SQLite)
  • HTTP clients → C2
  • /chat, /invocations, /hitl/*, /summarize, /debug/memory
C2 — Shared · Per Customer

Platform Services

All business logic. Config, memory, RAG, reasoning (planner/executor/responder), tools, observability, summarize.

  • /config — agent + domain config
  • /memory/read + write
  • /rag/retrieve
  • /reasoning/run — full in-graph cycle
  • /tools/invoke
  • /summarize — scoped summary
  • /observability/traces
C3 — Shared · Per Industry

Tool Gateway

Industry-specific tool execution and knowledge base. Deployed once, shared across all agents in the industry.

  • Tool registry + specs
  • /tools/invoke — executes tools
  • KB / vector store
  • Synth data store (dev/test)
  • Tool Admin UI (port 5200)

Memory System

4 Memory Types — Read Pre-Graph · Write Post-Graph

Each type has independent read and write policies, configurable per agent in memory.yaml. All reads happen before the planner. All writes happen after the responder.

Short-Term

Last N conversation turns. Written every turn. Read every turn. Keeps the agent aware of what was just discussed.

Scope: conversation (thread)

Episodic

Significant clinical events — tool calls, HITL approvals, case notes written. Written on write-tool success. Read when case/assessment scope is active.

Scope: case / assessment

Summary

Compressed narrative of the conversation. Written on turn_count or token_threshold trigger. Prevents context window overflow in long sessions.

Scope: conversation / member

Semantic

Vector-embedded facts stored for similarity retrieval. Written from agent observations. Read via embedding search. Enables knowledge that persists across sessions.

Scope: member / case
RAG System

3 Dimensions of Retrieval

Each dimension is independently configurable in agent.yaml. Two retrieval stages (pre-graph and planner-tool) can each have different settings across all three dimensions.

Dimension 1

Strategy

How the vector search query is executed against the knowledge base.

semantic keyword hybrid
Dimension 2

Stage

When retrieval runs relative to the agent reasoning loop.

pre_graph — silent inject planner_tool — LLM decides
Dimension 3

Pattern

How the retrieval query is constructed and refined.

naive self_corrective multi_hop hyde agentic
Roadmap

Next 10 Items to Build — Prioritized

Ordered by production impact and dependency. Pattern: Items 1/2/6 = separate pluggable services. Items 3/5/9/10 = overlay/config only. Items 4/7/8 = platform-core + UI.
# Item Type Why Now
1 Exception Handling & Recovery PLUGGABLE SERVICE Agents silently fail on tool timeout, LLM API error, network failure. Production blocker. Covers all calls — not tool-only.
2 Event-Driven Agents (Event Bus) PLUGGABLE SERVICE Most production agents are event-triggered. Webhook/queue/CDC/cron → AgentEvent → /invocations. Agents don't change.
3 multi_hop + reflection strategies OVERLAY ONLY Completes reasoning strategy set. New strategy files only. Zero platform changes, low risk.
4 Observability UI — Lineage + Metrics UI + PLATFORM-CORE Demo-critical. Trace viewer + quality/latency/token metrics. Reads existing /api/traces. No backend changes.
5 Routing tab in Agent Registry (3f) UI + YAML ONLY Completes Agent Registry. Config-driven hard routes. Small, self-contained change.
6 Guardrails (10m) PLUGGABLE SERVICE Healthcare/enterprise safety requirement. Pre-LLM input + post-LLM output policy check. Config-driven per agent.
7 Context Engineering (7) PLATFORM-CORE Token budgeting + priority truncation in context_builder.py. Needed before real member data hits context limits.
8 Goal Definition & Tracking (14) PLATFORM-CORE Persistent multi-step objectives across turns. New goal_state memory type + goal_tracker node in graph.
9 Resource-Aware Optimization PLATFORM-CORE CONFIG Query classifier + model router. Haiku → Sonnet → Opus by complexity. Direct customer cost impact.
10 summarization_agent_rag overlay OVERLAY ONLY KB-enriched summaries. New capability, zero platform risk. Pre-graph RAG + updated synthesis prompt.
PLUGGABLE SERVICES (1, 2, 6)
Deploy independently. Agents don't change. Exception Handling wraps ALL external calls (not tool-only). Event Bus calls /invocations. Guardrails intercepts LLM calls.
OVERLAY / CONFIG (3, 5, 9, 10)
New files and YAML options only. Zero platform rewrite. Any agent picks up changes by config update.
PLATFORM-CORE + UI (4, 7, 8)
Extend context_builder.py, memory store, Agent Factory UI pages. No new services.

Overlay Constraints

Invalid Agent Type × Strategy Permutations

Reasoning loops only make sense for agents that don't know upfront what they need. Batch and single-pass agents are always one-shot.
Agent Type ❌ Invalid Strategies Reason
summarization_agentreact, multi_hop, reflection, tree_of_thoughtYou know upfront what data to fetch. No discovery loop needed.
extraction_agentreact, plan_execute, multi_hop, reflection, tree_of_thoughtSingle pass: text in → structured fields out. Nothing to iterate over.
triage_agentreact, plan_execute, multi_hop, reflection, tree_of_thoughtClassify and route — always one step.
monitoring_agentmulti_hop, reflection, tree_of_thoughtCheck threshold → alert. react valid if investigation needed, nothing else.
workflow_agentmulti_hop, tree_of_thoughtWorkflow = executing steps, not researching questions or exploring branches.
retrieval_agentreflection, tree_of_thoughtSearch doesn't need self-critique loops or branch exploration.
plan_execute is the universal exception — not a loop, just "plan all fetches upfront, execute in parallel." Valid for any agent type that calls multiple tools. Agent Factory dropdown must filter valid strategies per agent_type — invalid combinations are hidden, not just disabled.
Prompt Alignment

Prompt Strategy Must Match Overlay

prompt-defaults.yaml is per overlay = agent_type content requirements × strategy format requirements. Wrong prompt on wrong strategy = broken agent at runtime.
Strategy Required Prompts Key Format Rule
simpleplanner_system_promptPick one tool + argument. No thought field. CoT instructions wasted here.
reactreact_planner_system_promptMust include Thought: / Action: format. Loop until DONE in Thought.
plan_executeplan_execute_planner_system_promptOutput full ordered plan as JSON list: [{tool, argument}, ...] before executing.
multi_hopdecomposer + sub_planner + synthesizerThree distinct prompts for three LLM roles. All required.
reflectionplanner + responder + reflectorReflector must output {issues_found, critique, suggested_correction}.
tree_of_thoughtbranching_planner + branch_evaluator + synthesizerEvaluator scores branches before pruning; synthesizer selects best path.
Agent Type Prompt Content Requirements
chat_agentConversational tone, memory-aware (prior turns), scope-aware (member/case context)
summarization_agentSynthesis-focused, structured output (key concerns, next steps), no conversational tone
extraction_agentStrict JSON output schema, no CoT, no explanation — just extract
triage_agentFixed classification labels in prompt, route decision output only
monitoring_agentThreshold language, alert vs no-alert decision output
retrieval_agentQuery reformulation instructions, source citation format, groundedness requirement
workflow_agentStep execution instructions, handoff format between steps
UI enforcement (Agent Registry → Prompts tab): detect strategy from agent.yaml → show required prompt fields → warn if missing → block save if any required prompt is empty → warn if simple strategy but CoT format detected in prompt.
Graph Phases

Pre / In / Post Graph — Universal for All Agent Types

Same 3-phase boundary for every valid overlay. What changes is what runs inside each phase, not the phases themselves.
Phase What Always Runs What Varies by Agent Type
PRE Memory read, RAG retrieval (if enabled), context build Batch agents (extraction, triage, monitoring) skip episodic memory read. summarization_agent_rag runs KB retrieval here.
IN Reasoning loop — planner → executor → responder Graph shape changes per strategy: simple = 2 LLM calls, react = loop, plan_execute = plan then parallel execute, multi_hop = decompose → sub-plans → synthesize
POST Memory write, HITL evaluation, response formatting Batch agents skip episodic/semantic write. HITL evaluation only runs for agent types with HITL enabled.

Observability

Trace Content — Per Overlay

Same trace event structure (pre/in/post graph) for every overlay. Payload content differs per agent type and strategy. Observability engine is shared — trace fields are overlay-specific.
Overlay / Strategy What the Trace Captures
simple2 LLM calls (planner + responder), 1 tool call, tool output
reactN loop iterations — each with thought + action + observation. Loop count, tool order.
plan_executeFull plan (ordered tool list), each tool result, final synthesis call
multi_hopSub-questions generated, tool call + sub-answer per sub-question, final synthesis
reflectionInitial answer, reflector critique, retry triggered (y/n), final answer delta
summarization_agentTool outputs fetched, synthesis LLM call, structured output fields
extraction_agentInput text, extracted fields, confidence score per field
triage_agentInput, classification label, confidence score, route decision
multi_agentSupervisor decision, sub-agent invoked, nested sub-agent trace, aggregation
Always emitted regardless of overlay:
pre_graph_complete — memory loaded, RAG retrieved, context built. Duration + token count.
in_graph_step — one event per step (LLM/tool/internal). Duration, input, output.
post_graph_complete — memory written, HITL evaluated, final response. Duration.
Evaluation

Eval Metrics — Per Overlay

Same eval framework and runner for all overlays. Metric set is overlay-specific — what "good" means differs per agent type and strategy. Eval runner reads from the trace to compute strategy-level metrics.
Agent Type / Strategy What to Evaluate Metrics
chat_agentAnswer quality, groundedness, tool selection, memory recallLLM-as-judge, groundedness %, tool accuracy, memory hit rate
summarization_agentCompleteness vs source, factual accuracy, hallucination rateROUGE/BLEU, LLM-as-judge completeness, hallucination flag rate
extraction_agentField accuracy vs ground truth, recall/precision per fieldExact match %, F1 per field, missed field rate
triage_agentClassification accuracy, routing correctnessAccuracy, precision/recall per class, confusion matrix, misroute rate
monitoring_agentAlert correctness, false positive rate, missed threshold rateAlert precision/recall, false positive %, threshold replay accuracy
multi_agentSub-agent selection, aggregation quality, end-to-end answerRouting accuracy, aggregation LLM-as-judge, golden set score
react strategyLoop efficiency — right tools, right order, no unnecessary iterationsLoop count, tool selection accuracy, unnecessary iteration rate
reflection strategyDid reflection catch real errors? Did retry improve score?Grade delta before/after retry, false critique rate, retry trigger rate
plan_execute strategyWas the plan correct upfront? Were all planned tools necessary?Plan accuracy vs optimal, unnecessary tool rate
multi_hop strategySub-question quality, sub-answer accuracy, synthesis coherenceSub-question coverage, sub-answer accuracy, synthesis coherence score
Shared eval framework components: Golden set (ground truth per overlay) · Eval runner (replays golden set against live agent) · Metric calculator (overlay-specific metric set) · Eval report (score breakdown + regression detection) · LLM-as-judge (shared grader used by chat, summarization, workflow, multi_agent)
Observability and eval are linked: eval runner reads from the trace to compute strategy-level metrics. You cannot compute loop count, plan accuracy, or reflection delta without the trace.
