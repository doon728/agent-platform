Reference Architecture — Agentic Enterprise Accelerator for Healthcare

AWS Native
Diagram 1 — Overview

How This Architecture Works

Container 1 — Agent Runtime

Thin shell on AWS AgentCore Runtime. One per agent.

  • Hosted per-agent on AgentCore Runtime — session-isolated microVM, unique ARN, managed scaling; one deployment per LOB per AWS account, self-registers with Container 4 on boot.
  • Three stages: pre-graph loads domain scope and enforces auth/authz; in-graph delegates all reasoning to Container 2; post-graph evaluates HITL gates and returns the response.
  • Access policy (authenticate caller, authorize agent + scope) runs here — separate from content guardrails, which apply on the LLM call path inside Container 2 via Bedrock Guardrails.
Container 2 — Platform Services

All reasoning here. Shared per tenant.

  • Runs on ECS/Fargate as one shared service per tenant — LangGraph executes the graph, Bedrock serves LLM inference; reasoning strategies, 4-scope memory, and RAG all execute here.
  • Isolation is logical, not physical — every request carries Agent ID + LOB namespace, and Container 2 loads exactly that agent's prompts, tools, and strategy from Container 4.
  • A Claims agent and a Care Management agent share the same service but run fully isolated configurations — same reasoning engine, different prompts, tools, and policies.
MCP Server + Container 3 — Tool Gateway

One gateway per tenant, LOB-namespaced tool catalogs.

  • One ECS/Fargate deployment per tenant holds LOB-namespaced catalogs — Care Management gets care plan + FHIR tools, Claims gets adjudication tools, Member Services gets enrollment tools.
  • Each agent has an allow/deny list inside its catalog, enforced by policy — e.g., Claims agents cannot call Care Management tools; schema-validated at the gateway.
  • MCP Server is an optional facade for interoperability — it can wrap AgentCore Gateway where present; all calls to source systems go through AWS PrivateLink, never the public internet.
Container 4 — Control Plane

ECS/Fargate. One per tenant. Logically isolated per agent and LOB.

  • One ECS/Fargate deployment per tenant governs every agent across every LOB — agent YAMLs in S3, registry metadata in RDS, all keyed by Agent ID + LOB namespace.
  • Care Management prompts, tools, and HITL rules are fully separate from Claims — different configs, different versions, different approver roles.
  • One Agent Factory UI and one observability dashboard — CloudWatch + X-Ray feeds from every account aggregate into a single cross-LOB view.
Multi-Account Deployment Model

One control plane. Many thin runtimes.

  • Container 1 and Container 3 deploy per AWS account — each LOB gets its own account; a LOB with multiple sub-domains (UM, CM, DM) can split further into one account per domain.
  • Container 2 and Container 4 are shared and central — all LOBs in a tenant use the same reasoning service and the same control plane.
  • Deployment unit is always: one account ID + one LOB namespace + one Container 1 + one Container 3, provisioned via CDK + Terraform parameterized by namespace.
Data Sources & RAG Knowledge Bases

Source systems are external. RAG KBs are per LOB/domain.

  • Source systems (claims DB, member DB, care management platform, prior-auth) live in the customer's existing accounts — Container 3 reaches them via PrivateLink APIs, never replicated or copied.
  • RAG knowledge bases (clinical guidelines, coding policies, benefit summaries) are built per LOB or per clinical domain inside the customer's environment on OpenSearch or pgvector.
  • KB endpoints, retrieval strategy, and chunking are all configurable per agent in Container 4 — exact KB topology varies by customer environment.
HITL Flow — Human-in-the-Loop

YAML-driven approval gates. Pause, approve, resume.

  • Gates are defined per agent in YAML — not hardcoded; C2 signals HITL, C1 checks Container 4's HITL & Policy Service for gate rules, approver role, and SLA.
  • Execution pauses and durable state is written to RDS/DynamoDB; the approver is notified (email/Slack/UI), reviews in the Care Manager or approver portal, and decision + audit trail is recorded.
  • Each LOB defines its own gates — clinical HITL (nurse approves care plan) is completely separate from claims HITL (supervisor approves prior auth); definitions versioned in Control Plane.
LOB vs Domain — Deployment Granularity

LOB is the account boundary. Domain is the sub-scope within it.

  • LOB = organizational unit, typically an AWS account boundary (Medicare, Medicaid, Commercial); Domain = functional subdomain within it (CM, UM, DM inside Medicare).
  • Deployment is flexible — one LOB account can host all its domains, or compliance/team boundaries can drive one account per domain; the platform supports both.
  • Agent IDs are structured as LOB:domain:agent in Container 4, and domain scope is captured in domain.yaml loaded by Container 1 at startup — isolation enforced regardless of topology.
Runtime Split — What Runs Where

Why C1 is on AgentCore and C2 / C3 / C4 are on ECS/Fargate.

  • AWS AgentCore is a set of managed services — Runtime (hosts agents), Memory, Identity, Gateway (MCP), Observability; the accelerator consumes them, it doesn't replace them.
  • Container 1 fits AgentCore Runtime's model exactly — one agent, one session-isolated microVM, one ARN, thin and per-agent.
  • Containers 2, 3, 4 are shared tenant services — warm state, connection pools, long-lived server processes — which fit regular ECS/Fargate, not per-session runtimes.
Container 2 Is a Hybrid

Our code on ECS/Fargate + managed AgentCore services consumed.

  • On ECS/Fargate we run our own code — LangGraph runner, reasoning strategies (simple, ReAct, plan-execute, multi-hop, reflection), HITL service, RAG orchestration, domain trace enrichment.
  • Underneath we consume managed services — AgentCore Memory (short-term + long-term), AgentCore Identity (auth + outbound credentials), AgentCore Observability (trace ingestion).
  • We also wire in Amazon Bedrock (LLM inference), Bedrock Knowledge Bases (RAG retrieval), and Bedrock Prompt Management — building only what AgentCore doesn't ship.
MCP Tool Serving — Where It Lives

AgentCore Gateway is the MCP server when used. Our C3 is the fallback.

  • AgentCore Gateway is itself a managed MCP server — it converts APIs, Lambda functions, OpenAPI/Smithy specs, and pre-built connectors (Salesforce, Slack, Jira) into MCP-compatible tools.
  • When Gateway is in use (default path), the "MCP Server" in the diagram is Gateway — no separate component to build, AWS manages it.
  • Container 3 is the fallback for on-prem tool execution, custom schema enforcement, or non-AWS backends — runs on ECS/Fargate with our own implementation plus an optional MCP facade.
Observability — Two Layers

AgentCore Observability is the infrastructure layer. We add domain.

  • AgentCore Observability (managed, OpenTelemetry-compatible, CloudWatch-backed) handles the infrastructure layer — per-agent traces, spans, metrics, logs, every session and tool call.
  • What it doesn't see is domain context — which member a conversation is about, which clinical workflow ran, which HITL gate outcome drove a decision, which capability UI the nurse is viewing.
  • Container 2 emits domain trace enrichment on top, and Container 4 aggregates it into per-capability dashboards the infrastructure layer alone cannot produce.

What This Architecture Enables

Source systems never leave their boundary

  • Patient, claims, and member data stays where it lives today — on-prem data centers or the customer's own cloud accounts — Container 3 reaches it via PrivateLink, Direct Connect, or VPN; nothing traverses the public internet.
  • Each LOB's audit trail, IAM boundary, and encryption keys stay independent.
  • Satisfies HIPAA and per-LOB compliance isolation simultaneously.

Logical isolation without physical duplication

  • Container 2 is shared, but every agent runs isolated config — Agent ID + LOB namespace is the isolation key, same as a multi-tenant SaaS platform.
  • No rebuilding the reasoning service per LOB — one reasoning engine serves every agent across every LOB.
  • A new LOB deploys one thin runtime and registers config; the platform applies to it immediately.

One place to govern everything

  • Policy-as-code from Container 4 — prompts (versioned, draft → active → retired), HITL gates, memory scopes, and RAG config across its 3 dimensions (Stage × Source × Extraction) are defined in YAML, versioned, and rolled out centrally — no code changes, no redeploys.
  • Tool Gateway admin in the same place — LOB-namespaced tool catalogs created here, tools onboarded with schema contracts, per-agent allow/deny policies applied, and per-LOB secrets/credentials governed from one console.
  • Observability is agent-level and cross-LOB — per-agent traces, spans, metrics, logs, eval scores, and lineage at AgentCore Observability's infra layer + domain enrichment from Container 2, aggregated in Container 4 into per-agent dashboards and one cross-LOB view; one audit trail, one eval surface.
  • Governance is the architecture, not bolted on — Agent Factory UI writes the same YAMLs the runtime reads; config, registry, tool policies, and observability live in one place by design.
