Section 1

Executive Summary

The Agentic Enterprise Accelerator deploys agentic AI at enterprise scale on Amazon Bedrock AgentCore — pre-built digital buddies for healthcare payer workflows that orchestrate across claims, member, and care management systems, manage state, and adhere to HIPAA-grade governance.

The Agentic Enterprise Accelerator for Healthcare is a pre-built, reusable platform and delivery asset designed for healthcare payer AI engagements. It ships with enterprise-ready pre-built agents for core payer workflows — and the scaffolding to rapidly create new agents tailored to each customer's specific requirements. Compliance controls and a security evidence package come bundled. The accelerator is built once by the practice and maintained as a versioned asset the delivery teams and internal developers can draw on.

The ask. Approve this proposal. Delivery engagements shift from bespoke builds to accelerator-based delivery. Customers reach production agents in weeks instead of months. The practice gains a repeatable, sellable healthcare payer AI service.

The current proposal is for the accelerator. Should leadership choose to pivot toward an Intelligent Agent Platform offered as Agent-as-a-Service, the same architecture supports it — no rework required. The decision is strategic, not technical.

While the accelerator is built for healthcare payers first, the architecture is domain-agnostic. The same platform and patterns can be adapted to financial services, insurance, and other regulated industries with similar compliance, auditability, and integration requirements — extending the practice's return beyond healthcare.

Section 2

Opportunity

The Agentic Enterprise Accelerator for Healthcare solves the bespoke-build problem. Any new agentic AI engagement should not start from a blank repo — rebuilding platform capabilities (orchestration, memory, tool gateways, HITL, compliance controls) before the actual workflow begins. The accelerator provides a pre-built foundation that an agentic AI engagement can draw on: architectural patterns proven in a reference implementation, reusable platform components, and pre-built agent templates — creating a repeatable healthcare payer offering for the practice.

Solution and Sales

Solution and Sales teams gain a repeatable healthcare payer AI offering they can position, price, and sell consistently. Pre-built reference agents and a working demo make pre-sales faster — RFPs go out with working evidence, not slideware. The result is a sellable service line with a known shape, not a one-off build each time.

Delivery

Delivery teams don't start every engagement from scratch. They inherit a working platform, reference agents, and a delivery playbook. Engagements finish in weeks instead of months, staffing is simpler, and every team delivers to the same standard — no more rebuilding the same plumbing on every project.

Customer AI COE / App Teams

Payer AI COE and app teams have AI infrastructure and models available but lack a repeatable approach for building agent systems at scale. The accelerator delivers a reference architecture exercised against a working agent and a reusable platform foundation, so their team focuses on workflow implementation rather than platform construction.

Customer Business

High-value workflows — care management outreach, prior-auth remediation, claims triage — are well-suited for AI automation but remain constrained by manual effort and integration complexity. The accelerator enables AI agents to be built and deployed into these workflows faster, with compliance patterns built into the platform.

Accelerator Differentiation

The architectural patterns are shared with Amazon Bedrock AgentCore. The accelerator's differentiation is what sits on top of them:

  1. Vertical packaging — healthcare payer-specific patterns, workflows, and compliance posture. AgentCore is domain-agnostic; this accelerator ships with payer context baked in.
  2. Delivery-integrated — comes with a Delivery Playbook, demo + demo script, security evidence package, and pre-built reference agents. Not code alone; a full engagement-ready asset.
  3. Three pre-built reference agent templates — PA Denial Remediation, Pre-Call Assessment for Care Management, Claims Triage — each with payer-tuned content. The overlay matrix (agent-type × reasoning-strategy) remains available as a platform capability via the Agent Factory UI.

On PHI masking. Amazon Comprehend Medical and AWS Bedrock Guardrails are the AWS services that detect PHI. The accelerator uses them — doesn't replace them. What the accelerator adds: the PHI check is already wired into the agent flow. When an agent generates a response, the response passes through this check (which calls Amazon's service) before it reaches the user. If PHI is found, it gets masked. Rules for what to mask per agent live in YAML config. The accelerator ships with this pre-built — customer doesn't have to wire it themselves.

Section 5

How the Agentic Enterprise Accelerator Is Adopted

The accelerator serves two audiences: internal engineers and solutions teams, and customers during delivery engagements. Each audience uses the accelerator differently; each has its own companion playbook covering the detail.

The accelerator lives as a master repo owned by the Platform Team. Internal developers clone a sandbox from it for POCs and demos. For a customer engagement, the Platform Team creates an engagement fork that the delivery team tailors during delivery and hands to the customer as their source of truth. Full repo governance, branch model, and clone mechanics are in the Internal Playbook and Accelerator Technical Design companion documents (currently work in progress).

Inside TEKsystems

Internal engineers and solutions teams apply the accelerator as soon as it is released — not only during customer engagements. Typical internal activities:

  • Onboarding — engineers and solutions teams review the Internal Playbook (master repo governance, local dev workflow, playground AgentCore usage, release cadence, LLM key management) before working with the accelerator
  • POCs and exploration — create a sandbox repo from the master template, run the platform locally with docker-compose up, scaffold agents through the Agent Factory UI in minutes
  • Pre-sales demos and RFP responses — use the pre-built reference agents and working demo script to show capability without building from scratch for every conversation
  • Design and prototype work ahead of engagements — validate customer-specific agent designs against the platform before engagement kickoff
  • Real-runtime validation — optional deploy to the practice's playground AgentCore sandbox to test against real AWS managed services (AgentCore Runtime, Memory, Identity, Gateway, Observability, and Bedrock) before customer deployment
  • Feedback and feature requests — engineers suggest features, report issues, and provide delivery feedback; the Platform Team evaluates and incorporates improvements into the master repo in future releases
Internal detail — Platform Team ownership model, master repo governance, local development workflow, playground AgentCore usage, release cadence, LLM key management for internal use — is in the Internal Playbook companion document.

At the Customer — Four Modes

Not every customer will accept every part of the accelerator into their environment. Healthcare payers have strict policies on what runs in their AWS accounts, what code they accept from outside, and which open-source dependencies they approve. The accelerator is structured so it delivers value across a spectrum — from full deployment down to reference-only use — without losing its benefit to the delivery team.

Mode Deployed to Customer Customer AI COE Post-Handoff
1. Full deployment Entire accelerator (all four containers, IaC, agents, admin UI) in customer AWS; engagement repo handed over as their master copy Owns platform code and templates. Scaffolds new agents independently. Pulls our upstream releases or freezes on the version they received.
2. Partial / hybrid Selected components accepted; others rebuilt or swapped for customer-standard equivalents Owns the deployed portion plus the custom integration layer. Upstream release adoption depends on which components they accepted.
3. Reference only Nothing deployed; customer builds their own version using our architecture as blueprint Owns what they built. No ongoing upstream relationship with us on platform code.
4. Enablement only No deployment, no code sharing; used for design, demo, and enablement No code ownership. Customer uses the enablement to inform their own build-vs-buy decisions.
Why this matters: the value of building this accelerator does not depend on any single customer accepting full deployment. Every engagement captures some subset of the value — and in the worst case (mode 4), the practice still skips the whiteboard phase and delivers faster.
Per-mode detail — what exactly the customer receives, deployment shape, AI COE role, and when each mode fits — is in the Customer Delivery Playbook companion document.
Section 6

How the Agentic Enterprise Accelerator Gets Built

Building the accelerator is what this proposal is asking to fund. The build is not starting from zero — a working prototype is already in place. The four-container platform, the core shared library (memory, RAG, HITL, tool gateway, observability, prompt management), one end-to-end reference agent (pre-call assessment for care management), and a capability UI for care management are running today. Major architectural decisions — runtime split, agent-type × reasoning-strategy model, policy-as-code approach, HIPAA posture — are locked and documented. Full detail of what exists today is in the Accelerator Technical Design companion document.

The remaining work is to complete the accelerator for delivery — produce the Prototype Current-State Documentation and complete a codebase audit as a Phase 1 pre-requisite, establish the enterprise master repo with governance (branch protection, CODEOWNERS, release tagging), add the missing backend adapters, implement remaining reasoning strategies, expand from one to three reference agent templates (PA Denial Remediation, Pre-Call Assessment for Care Management, Claims Triage — each payer-tuned), build the pre-built payer tool library (FHIR, entity-level payer APIs, InterQual, MCG, Availity, Change Healthcare, EHR FHIR), wire Amazon Connect integration pattern and IDP pipelines via Bedrock Data Automation, add Infrastructure as Code and CI/CD, implement PHI masking as an enforced layer (pre-response interceptor architecture), produce the platform-level security evidence package, and author the delivery playbook and demo script.

With a planned team of one offshore Delivery Lead, two full-time offshore developers, plus a part-time Platform Architect / Team Lead, a part-time Security Engineer, and a part-time DevOps Engineer, the build runs approximately 22 weeks across eight phases. Final scope, team, and duration are subject to sizing during Phase 2.

The accelerator's core architectural choices match reference architectures published by major agentic AI platform vendors, and multiple major SI competitors are productizing agent offerings in the same market. TEKsystems' differentiation is what it covers end-to-end for its specific niche.

Platform & Pattern Alignment

Each agent deploys as its own containerized runtime with unique ARN and versions; Memory, Gateway, Identity, Observability are shared account services. Where the Agentic Enterprise Accelerator Adds on Top

The core architecture is the right bet — converged across AWS reference implementations, LangChain, PwC's public case, and conceptually across the Big 4 SI category. TEKsystems' value is the layer above that common foundation, which competitor materials do not describe.

  • Vertical packaging for healthcare payer — reference agents aligned to payer workflows (UM, CM, claims, prior-auth), HIPAA-eligible compliance evidence, PHI masking, payer-specific source system adapters.
  • Explicit four-container architecture — C1 Agent Runtime, C2 Platform Services, C3 Tool Gateway, C4 Control Plane with defined boundaries and deployment topology.
  • Agent-type × reasoning-strategy overlay matrix — productized combinations each with strategy implementation, prompt scaffolding, and YAML config; discoverable in the Agent Factory UI.
  • Agent Factory UI + pre-built templates — create, configure, and govern new agents through a web console using pre-built templates; no code for standard agents.
  • Agent Registry — single source of truth for every agent deployed across AWS accounts, indexed by Agent ID + LOB namespace; one place for audit, discovery, version tracking, and governance.
  • Policy-as-code governance — prompts, HITL gates, memory scopes, RAG config (Stage × Source × Extraction), tool allow/deny — all in YAML, no redeploys.
  • Multi-LOB isolation — per-LOB AWS accounts and Container 1 / Container 3 pairs; shared Container 2 + Container 4 with Agent ID + LOB namespace enforcement.
  • Delivery-integrated — engagement playbook, repo governance, CODEOWNERS-enforced platform/app split, template + scaffolded-instance pattern.
