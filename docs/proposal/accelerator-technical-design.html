What the Accelerator Includes — Capabilities in Detail

The capabilities below are produced by this offering as code, infrastructure, and configuration. They ship pre-wired so delivery teams and developers extend them rather than rebuild the foundations.

Capability What It Does
Agent Factory UI Admin UI to scaffold, configure, govern, and observe agents — no code required for config changes. Single surface for all agent operations across capabilities.
Agent Types 7 pre-built agent templates: chat, summarization, extraction, triage, monitoring, workflow, multi-agent — each configured per use case during engagement.
Reasoning Strategies 5 strategies per agent: simple, ReAct, plan-execute, multi-hop, reflection — selected in config, no code change.
Multi-Agent & Workflow Orchestration Supervisor pattern for multi-agent workflows where one agent orchestrates sub-agents operating at different domain scope levels. New workflow = new YAML config, zero platform code change.
Memory System 4 memory types: short-term (session), episodic (case), semantic (member facts), summary — scoped to healthcare domain with per-type read/write policies, scope-aware write enforcement (member / case / assessment), memory pruning (TTL, max-size eviction), and intelligent retrieval (rule-based pre-filter + optional LLM decision).
Context Engineering Token budgeting per context type (memory, tools, history, retrieved docs), priority-based truncation, and context quality scoring — protects the LLM context window from silent overflow as agents handle long sessions and large tool outputs at real member-data scale.
RAG — Multi-Dimensional Three dimensions combined per agent: strategy (semantic / keyword / hybrid / graph), pattern (naive, self-corrective, multi-hop, HyDE, agentic), and stage (pre-graph, in-graph, post-graph). Configurable per agent. Supports pgvector, OpenSearch, Bedrock Knowledge Bases.
Tool Gateway Secure HTTP-native layer connecting agents to claims, member systems, care management platforms, prior-auth systems, and external APIs — tag filtering and schema validation per agent. Pluggable backend (wraps AgentCore Gateway by default).
MCP Server Optional Model Context Protocol facade — standard protocol interface on top of the Tool Gateway for third-party agent interoperability.
Human-in-the-Loop (HITL) Approval gates at any agent step; per-tool allow/deny; durable session rehydration after pause/resume; adapter pattern supporting internal UI, email, Slack, Teams, and payer enterprise workflow systems (Pega Care Management, ServiceNow) on the roadmap.
Event-Driven Agents Event bus / trigger service connecting webhooks, Kafka, SQS, CDC feeds, and cron schedules to agent /invocations endpoints. Unlocks production workflow automation — agents triggered by business events rather than user requests. Pluggable service, zero agent code change to adopt.
Prompt Management Prompt versioning, lifecycle (draft → active → retired), A/B evaluation, per-agent prompt governance — no redeploy needed to ship a prompt change.
Guardrails Input, output, and tool-call policy intercept. Blocks prompt injection, PII leakage in responses, hallucinated clinical facts, out-of-scope requests, high-risk tool arguments. Adapter-compatible with AWS Bedrock Guardrails, Azure Content Safety, Guardrails AI. Complements HITL — guardrails fire automatically; HITL fires for what passes guardrails but still needs human review.
Resilience & Exception Handling Retry with backoff, circuit breakers, timeouts, and fallback patterns across all external calls (tools, LLM APIs, memory, RAG). Graceful degradation so single failures do not cascade. Platform-core library plus optional network-level sidecar (Envoy / Istio) for infrastructure-layer resilience.
Observability Per-agent trace events, reasoning-step decisions, tool call detail, memory read/write events, HITL gate outcomes; lineage graph tying a final response back to every upstream dependency; analytics dashboard per capability and per agent.
Evaluation & Testing Framework LLM-as-judge, RAG evaluation (Ragas), prompt regression detection, per-strategy eval metrics, score dashboards. Offline scoring loop integrated with prompt lifecycle so every prompt change can be evaluated against a test set before promotion to active.
LLM Ops & Cost Tracking Per-agent and per-capability cost tracking, token usage dashboards, latency monitoring, model swap A/B testing, per-tenant budget caps, kill-switch for runaway agents. Model selection per-agent in agent.yaml enables cost optimization (move predictable steps to Haiku, keep hard reasoning on Opus) without redeploy.
PII / PHI Masking Pre-response PHI filter in the agent runtime and before trace/log emission — prevents sensitive data from reaching the UI or observability layer. Day-one requirement for healthcare deployments.
Authentication & Authorization User authentication (SSO / Cognito / OIDC) for admin UIs, service-to-service auth across the four containers (IAM / mTLS), tool authentication to external payer systems (claims, care management, prior-auth), agent-level authorization (allow-listed tools per agent, scoped memory access, tag filtering), per-tenant scope enforcement for multi-LOB deployments, and IAM-based Bedrock access with no static keys.
Infrastructure as Code Full IaC — AWS CDK for AgentCore resources, Terraform for VPC, networking, IAM, Helm charts for container deployments. All in Git, auditable and repeatable. iac/ folder has its own CODEOWNERS and branch protection for infra-review discipline.
Section 2

Reference Architecture — Request Flow Through Four Containers

The platform is structured as four independent containers, each with a distinct responsibility. A single agent request moves through them in sequence — from entry point, through reasoning, through tool execution, with configuration and governance served by the control plane. This separation is what makes the accelerator deployable per line of business, per AWS account, without duplicating intelligence.

Agent request →
Container 1
Agent Runtime
Thin shell, per LOB, per AWS account
  • Pre-graph — load domain scope, auth/authz, hydrate session
  • In-graph — delegates reasoning to C2 (no LLM logic here)
  • Post-graph — HITL gate evaluation, PHI masking, response delivery
  • Registers with C4 on boot using LOB + AWS account ID
AWS AgentCore Runtime · VPC · IAM
Container 2
Platform Services
Shared backend service — one per tenant
  • Reasoning services — simple, ReAct, plan-execute, multi-hop, reflection
  • Memory services — short-term, episodic, semantic, summary read and write
  • RAG services — semantic, keyword, hybrid, graph per agent config
  • HITL, prompt, observability services — shared across all agents
ECS / Fargate · Bedrock · RDS · ElastiCache · OpenSearch
Container 3
Tool Gateway
Shared tool execution — one per tenant
  • HTTP-native tool serving — schema-validated per agent
  • Tag filtering — each agent sees only its allowed tools
  • Connects to claims, member systems, care management platforms, prior-auth systems, external APIs
  • Optional MCP facade and AgentCore Gateway wrapper for interop
ECS / Fargate · Secrets Manager · PrivateLink
Container 4
Control Plane
Configuration, governance, observability
  • Agent Factory UI — scaffold, configure, govern agents
  • Prompt Management — versioning, lifecycle, A/B eval
  • Agent registry — all deployed agents across LOBs
  • Observability — traces, evals, lineage, analytics
ECS / Fargate · RDS · S3
▼
▼
▼
▼

What flows through: C1 receives the request, hydrates session and policy, and calls C2. C2 runs the reasoning graph — invoking C3 whenever a tool call is required and reading/writing memory along the way. C3 executes the tool, returns structured output, and C2 continues reasoning. When the graph completes, C1 applies HITL gates and PHI masking before the response reaches the UI. C4 does not sit in the request path — it serves configuration to C1/C2/C3 at load time and receives traces and metrics after each run.

Why four containers: C1 is thin so it is cheap to deploy per line of business per AWS account — each LOB gets its own data boundary and blast radius without duplicating reasoning logic. C2 and C3 run on AWS AgentCore so we inherit managed scaling, native memory, and tool integration. C4 is shared across the practice — one control plane manages all agents across all customer deployments. Tool Gateway vs MCP: the Tool Gateway is the execution engine (internal control, per-agent access policy, schema validation). MCP is the interoperability interface — an optional facade for third-party clients. They are not competing; they serve different purposes at different layers.

Section 4

Container Architecture & Repo Design

This section captures the core design decisions that determine how the four containers are scoped, how templates relate to scaffolded agent instances, what gets cloned into engagement and developer repos, and what developers can and cannot modify. These decisions are what let the accelerator scale: adding an agent is a YAML and scaffold operation, not a platform change.

What Gets Cloned

When a developer (internal sandbox) or delivery team (customer engagement) clones the accelerator, they receive a full fork of the master accelerator repo. The repo contains everything:

  • The four-container platform code (C1 / C2 / C3 / C4) — runnable locally with docker-compose up
  • The platform-core shared library and supporting services (prompt management, tool gateway)
  • The templates/ folder — source material the Agent Factory UI reads to scaffold new agents
  • Pre-built reference agents — already scaffolded, configured, and running under agents/<domain>/<agent>/
  • The Agent Factory UI — the interface for creating new agents
  • IaC, CI/CD, and documentation — everything required to run and deploy

What gets modified vs. what does not is enforced by policy, not by absence. Every fork contains every file. The separation is enforced through CODEOWNERS files on platform directories (Platform Team approval required), branch protection rules (PRs required, platform-directory changes need additional reviewers), and team discipline (engagement teams operate only in agents/, capabilities/, and config files).

Read-Only by Policy

  • platform-core/ — shared library
  • platform-services/ — C2 services code
  • shared-infra/ — C3 tool gateway code
  • platform-tools/ — C4 Agent Factory UI + API
  • templates/ — scaffolding source material

Modified Freely

  • agents/<domain>/<agent>/overlays/<variant>/config/ — per-agent YAML
  • agents/<domain>/<agent>/ — custom tool adapters, custom UI
  • capabilities/<domain>/ — per-capability domain.yaml and UI
  • Engagement-specific IaC values (tfvars, CDK context)

Upstream updates flow via git merge. When the practice ships v1.1, each engagement's architect pulls it at a sprint/phase boundary (git fetch upstream && git merge upstream/v1.1). The merge updates platform code and templates; engagement-team-owned folders (agents/, capabilities/) remain untouched.

At customer handoff: the customer receives their engagement repo as their own master copy — full platform code, templates, Agent Factory UI, everything. Their AI COE becomes the equivalent of the practice's Platform Team internally, free to evolve platform code and templates for their own needs or to pull upstream releases from the practice at their cadence.

Container Scoping — How Each Container Scales

The four containers scale differently. Getting this right matters for cost and deployment topology.

Container Scope Count per tenant
C1 — Agent Runtime (thin shell) Per-agent 100 agents = 100 C1 instances
C2 — Platform Services Shared across all agents within a tenant 1 C2 per tenant
C3 — Tool Gateway Shared across all agents within a tenant 1 C3 per tenant
C4 — Control Plane Per-tenant 1 C4 per tenant

Why C1 is per-agent. Each agent has its own graph topology (encoded in build_graph.py), its own endpoint, its own cold-start behavior, and its own deployment lifecycle. Keeping C1 thin means 100 instances cost almost nothing in engineering effort or runtime — no intelligence is duplicated.

Why C2 and C3 are shared within a tenant. Reasoning strategies, memory services, RAG logic, HITL service, tool gateway code — these are identical across agents. Running 100 copies of the same reasoning service would waste resources. One C2 and one C3 serve all agents in the tenant. Per-agent customization happens through the agent's YAML config, which C2 and C3 read per request.

Why C4 is per-tenant. C4 administers Agent Factory UI, agent registry, observability, and prompt management for a given tenant. Internally at the practice, the practice itself is the tenant; developers for each capability access one C4 instance. At a customer, each customer is a tenant; C4 deploys per customer (or per LOB within a customer). C4 is not a single shared instance across all customers — it is one per tenant.

Shell Code Structure

The C1 shell is deliberately minimal. Structure for a scaffolded agent:

agents/<domain>/<agent>/ ├── overlays/<variant>/ ← per-variant YAML config ├── domain.yaml ← capability-level domain config └── services/ └── agent-runtime/ ← the C1 scaffolded instance ├── Dockerfile ├── pyproject.toml ├── config/ ← base/dev/staging/prod.yaml └── src/ ← the shell code ├── clients/ ← HTTP clients to C2, C3, C4 │ ├── strategy_client.py │ ├── memory_client.py │ ├── rag_client.py │ ├── tools_client.py │ └── config_client.py ├── graph/ │ └── build_graph.py ← agent-specific graph topology └── platform/ ├── app.py ← FastAPI entry point └── langgraph_runner.py

graph/build_graph.py is the main per-agent file. It encodes the agent's LangGraph topology — nodes, edges, state transitions. Every other file in the shell is identical plumbing across all agents.

Generated, not hand-edited. build_graph.py is generated by the Agent Factory UI from the agent's YAML config (reasoning strategy, tools, RAG stages, HITL gates). When YAML config changes, build_graph.py is regenerated. Developers treat it as generated code; they do not edit it by hand.

What Developers Modify

Developers working on an agent modify configuration and per-agent extensions — not platform code.

Developer Modifies

  • agent.yaml — reasoning strategy, tools, RAG, model
  • memory.yaml — memory types, scopes, backends
  • prompt-defaults.yaml — prompt seeds
  • domain.yaml — capability-level config
  • Prompts via Agent Factory UI / Prompt Management service
  • Custom tool adapters when a new integration is required
  • Custom agent UI for per-agent UI customizations

Read-Only for Developers

  • C1 shell code (generated from template + YAML)
  • C2 platform services code
  • C3 tool gateway code
  • C4 control plane code
  • platform-core shared library
  • Template source in templates/

Platform code is owned by the Platform Team. New reasoning strategies, memory backends, RAG patterns, tool adapters, and compliance controls ship as versioned platform releases (v1.0, v1.1, v1.2) — not per-engagement code edits. Developers interact with the platform the way application developers interact with a framework: through configuration and their own code, not by modifying the framework itself.

Summary of the design:
  • Cloning gives a working system. Every fork contains everything — platform code, templates, reference agents, Agent Factory UI.
  • Separation is enforced by policy (CODEOWNERS, branch protection), not by absence of code.
  • C1 multiplies with agents; C2 and C3 are shared within a tenant; C4 is per-tenant.
  • Scaffolded instances, not templates, build into containers. Templates in the repo are source material the Agent Factory UI reads.
  • build_graph.py is the one agent-specific file — generated from YAML.
  • Developers work in YAML, prompts, custom tool adapters, and custom UI. Everything else is read-only platform code.
  • At customer handoff, the customer receives the full repo. Their AI COE owns platform code and templates going forward.

What We Plan to Build vs. What AgentCore Provides

The accelerator sits on top of Amazon Bedrock AgentCore. AgentCore ships five managed services — Runtime, Memory, Identity, Gateway, Observability — which the accelerator consumes directly. Container 2 in our design is a thin layer of our own code on ECS/Fargate that orchestrates those managed services and adds the pieces AgentCore does not ship: reasoning strategies, HITL service, RAG orchestration, and domain-level trace enrichment. The table below makes each decision explicit.

Container 2 is a hybrid. It runs on ECS/Fargate (our LangGraph runner, reasoning strategies, HITL service, RAG orchestration, domain-level observability emission — all our code) and consumes AWS-managed services (AgentCore Memory, AgentCore Identity, AgentCore Observability, Bedrock for LLM inference, Bedrock Knowledge Bases for RAG backends, Bedrock Prompt Management) over HTTPS. C1 Agent Runtime is the only container that runs on AgentCore Runtime; C2, C3, and C4 are our own ECS/Fargate containers that call AgentCore managed services as AWS services.

Why C1 runs on AgentCore Runtime but C2, C3, and C4 do not

AgentCore Runtime is purpose-built for hosting agents — session-isolated microVMs per invocation, per-agent IAM, per-agent ARN and versioning. C1 matches that model exactly: one shell, one agent, isolated session. C2, C3, and C4 are shared tenant-level services that serve many agents concurrently. They need warm state (connection pools, cached config, prompt cache), high fan-in traffic, and long-lived server processes — characteristics that fit regular ECS/Fargate, not a per-session microVM model. This split mirrors how AWS itself organizes AgentCore: AgentCore Runtime hosts agents; AgentCore Memory, Identity, Gateway, and Observability are separate managed services, not workloads on AgentCore Runtime.

The positioning in plain language: AgentCore is our foundation. This accelerator is the healthcare-payer vertical layer built on top of it — pre-built reference agents, compliance evidence, delivery assets, scaffolding UI, and the orchestration wrappers that give operators a domain-ready platform. AgentCore primitives are consumed where they fit; wrapped where flexibility is needed; built new only where AgentCore does not ship the capability.

Accelerator Differentiation

The architectural patterns are shared with Amazon Bedrock AgentCore. The accelerator's differentiation is what sits on top of them — the vertical packaging, the delivery assets, and the agent matrix:

  1. Vertical packaging — healthcare payer-specific patterns, workflows, and compliance posture. AgentCore is domain-agnostic; this accelerator ships with payer-context baked in.
  2. Delivery-integrated — comes with a Delivery Playbook, demo + demo script, security evidence package, and pre-built reference agents. Not code alone; a full engagement-ready asset.
  3. Pre-built agent matrix — six working agents across agent-type × reasoning-strategy combinations, each with its own UI. Ready to demo, ready to tailor to customer workflows during engagement.

Architectural Classification — Build-time, Config-time, Runtime

The accelerator is policy-driven, not runtime-dynamic for foundational elements. Reasoning topology, agent type, container split, and scope structure are committed once at scaffold time and cannot change without rescaffolding — this is intentional for auditability and compliance. Most governance-layer decisions are config-time updatable (YAML push, no redeploy) — prompts, HITL gates, tool allow/deny, RAG source list, guardrail policies. Runtime-routable decisions are limited to per-request data and model dispatch — which source to query, which LLM to invoke, which tool to use, whether a HITL gate trips.

The build-time / config-time / runtime split is a conscious trade-off between flexibility and governability. Enterprise and regulated domains — healthcare payers included — value governability over maximal runtime flexibility.

Known Design Constraints

The design matches industry patterns, but like every architectural choice it carries specific constraints that should be acknowledged and planned around. These are neither unique to this accelerator nor fatal — they are predictable consequences of the decisions above.

Constraint Impact Mitigation
Per-agent C1 runtime cost at scale At roughly 1000+ agents per tenant, running one dedicated Fargate task per agent becomes cost-noticeable even idle. The per-agent container model is not built for very high agent counts without evolution. Plan a runtime-multiplexed mode as a config option — multiple low-traffic agents share a C1 process. Salesforce Agentforce (agents as metadata on a shared runtime — docs) and Microsoft Foundry's Prompt and Workflow agent tiers show this is production-viable. Keep per-agent as default; add multiplexed mode if agent counts cross the threshold.
C2 monolithic process as shared blast radius C2 hosts reasoning, memory, RAG, HITL, prompt, observability, policy, and tool-governance in one FastAPI process. An OOM or runaway query in any one service can affect all agents in the tenant. Apply internal bulkheads within the same container — separate asyncio thread pools, separate connection pools, and circuit breakers around each service module. Do not split C2 into microservices; the on-premises deployability constraint makes a single container important. Peel specific services (memory and observability are the first candidates) into separate containers only if real traffic shows it is needed.
HIPAA data-plane hardening Infrastructure-layer posture (VPC endpoints, KMS, CloudTrail, PHI masking layer) is sound. The data-plane layer — memory-at-rest encryption, prompt and trace retention, member-scoped deletion, access audit separate from agent trace, cross-tenant isolation tests — needs explicit commitments before customer InfoSec engagement. Document each control before Phase 5 (Security & Compliance) closes. Microsoft Foundry's "standard agent setup" — customer brings their own Azure resources for all customer data, including files, conversations, and vector stores (architecture) — is a useful reference posture. AWS AgentCore's HIPAA eligibility evidence (compliance) is the AWS-side corollary.
One cautionary data point worth acknowledging: Microsoft Copilot Studio's tenant isolation is off by default — admins must explicitly turn it on (guidance). For a healthcare payer accelerator, this reinforces that tenant isolation must be explicit by design, not assumed to be inherited from any underlying platform. Every deployment of this accelerator should include cross-tenant isolation tests in CI.

Deployment Topology Options

Production deployment supports three topologies depending on customer preference — chosen during Phase 1 based on data residency, existing AWS estate, and LOB structure.

Per-LOB Multi-Account (Default)

  • C1 Agent Runtime deployed once per line of business, each in its own AWS account
  • C2 and C3 run on ECS/Fargate in the tenant's VPC; consume AWS managed services (Bedrock, AgentCore Memory, Secrets Manager) over PrivateLink
  • C4 Control Plane is a single instance — registers all LOB deployments
  • Blast radius contained to a single LOB; data never crosses LOB boundary
  • Preferred for large health systems with multiple business units

Single-Account & Air-Gapped Variants

  • Single-account — all four containers in one AWS account. Simpler for smaller deployments or single-LOB customers.

  • Both variants packaged as CDK stacks — identical agent code, only the IaC target changes

Repo Governance & Developer Onboarding

The accelerator is distributed through a three-tier repository model. One master repo owned by the practice, one engagement repo per customer, and individual exploration repos for developers who want to learn or prototype. Each tier has a different governance posture — deliberately.

For a walkthrough of what exists today as a working prototype — the reference agent running end-to-end — see Section 9 at the end of this document.

Repo Type Access Purpose
Master / Template Platform team writes; practice reads Source of truth. Versioned releases (v1.0, v1.1, …). Marked as template repo on the Git platform.
Engagement Assigned engagement team only (10–50 developers) Customer delivery. Branch-protected, forking disabled, PR-enforced. Handed to customer at end.
Exploration Individual developer POCs, learning, experimentation. Self-service via "Use this template." No governance.

Who Enforces Access

  • Platform Team (3–5 people) — owns the master repo, controls write access, publishes releases
  • Practice PMO / Engagement Lead — requests engagement repo at kickoff, defines team, owns access lifecycle
  • Git platform permissions — Teams / Groups enforce access; branch protection enforces PR workflow
  • Developer self-service — exploration repos via "Use this template," no approval needed

How Master Updates Propagate

  • Platform team tags a new release (e.g. v1.1) and publishes release notes
  • Engagement architect decides when to pull — typically at sprint / phase boundary, not mid-sprint
  • Merge happens via git fetch upstream && git merge upstream/v1.1 → PR into engagement main
  • Engagements may stay on their starting version for stability / compliance freeze
Why this model works: The master repo is the version-controlled source of truth; branch protection enforces review before merge to main. Engagement repos are access-controlled team workspaces — 50 developers on a customer engagement share one repo, not 50 independent copies. Exploration repos are disposable sandboxes for developers to learn the platform without engagement approval. Same source, three different creation and governance paths.

Infrastructure as Code Gets Its Own Review Discipline

Platform code, agent code, and Infrastructure as Code all live in the same repository — one clone gives a team everything it needs to stand up an environment. But IaC changes (VPC, IAM, Bedrock policies, security groups, KMS keys) carry a different risk profile than application code, so the iac/ folder is governed differently inside the same repo:

  • iac/ has its own CODEOWNERS file — infrastructure changes require review by designated infra/SRE owners, not just any engagement engineer
  • Its own branch protection rules applied on top of the repo-wide protections — for example, two approvals required for production-facing changes
  • Dedicated CI pipeline for IaC: plan on PR, apply on merge after approval, with drift detection
  • Customer-specific IaC values (AWS account IDs, VPC CIDRs, DNS names, resource tags) live in engagement-specific tfvars / CDK context — same repo, different files, clearly separated from reusable modules

Single repository, but with review discipline that matches the risk profile of each folder.

Three Consumption Flows

The same master accelerator repo is consumed three different ways. Each flow below answers one simple question: "What does this actor actually do, in order?"

Flow 1 — Individual Developer (Sandbox / POC)

  1. Goes to master accelerator repo URL in the practice's Git
  2. Clicks "Use this template" → new sandbox repo in their own namespace
  3. git clone + docker-compose up — platform running locally in ~5 minutes
  4. Uses Agent Factory UI on localhost to scaffold, configure, and test agents
  5. (Optional) deploys to practice playground AgentCore — namespace-scoped, budget-capped
  6. Pulls v1.x updates when they want; sandbox repos are disposable

Flow 2 — Customer Delivery Engagement

  1. Engagement kicks off — PMO requests an engagement repo from Platform Team
  2. Platform Team forks master into engagements/customer-x-agents, adds the assigned team, enables branch protection, disables personal forks
  3. All assigned developers clone the same engagement repo — one shared workspace
  4. Team tailors platform-core/, shared-infra/, templates; adds customer-specific agents under agents/<domain>/
  5. CI/CD builds and deploys — dev → staging (practice) → prod (customer AWS account)
  6. At end of engagement, codebase is handed to customer with full IP transfer

Flow 3 — Accelerator Version Upgrade

  1. Platform Team releases v1.1 — new backends, agent templates, compliance patterns, bug fixes
  2. Release notes announced to practice and all engagement architects
  3. Each engagement architect decides when to pull — typically at a sprint or phase boundary, not mid-sprint
  4. git fetch upstream && git merge upstream/v1.1 → PR into engagement main → CI passes → deployed
  5. Customers who have received handoff pull the same release on their own cadence
Why three flows, one master: Sandbox exploration is self-service. Customer delivery is access-controlled. Version upgrade is a standard Git merge. All three operate against the same versioned master repo — no side channels, no special tooling. When we ship a new version, every engagement and every developer can pull it with one merge. A zip-based download cannot offer upgrades once extracted — the artifact is frozen. Distributing through Git keeps every engagement and every developer on a living, improvable asset.
Section 7

LLM Keys, Models & Cost Control

LLM access is structured around two principles: no static API keys on developer laptops for production workloads, and cost attributed to the engagement, agent, and developer that incurred it. The accelerator is Bedrock-native for production, with a controlled OpenAI proxy available for dev exploration only.

Primary — AWS Bedrock (Production)

  • No static keys — ECS/Fargate tasks assume IAM roles, call Bedrock directly
  • HIPAA-eligible — Claude models accessed via VPC endpoint
  • Cost isolation — per-engagement AWS account = natural cost boundary, tagged in AWS Cost Explorer
  • Auth audit — every Bedrock call logged to CloudTrail

Secondary — OpenAI Proxy (Dev / Exploration)

  • Practice-run proxy (LiteLLM or similar) — developers never see the raw OpenAI key
  • Scoped proxy tokens — per-developer, short-lived, revocable
  • Daily spend caps enforced at proxy — developer can't exceed their quota
  • Non-PHI only — proxy enforced on exploration data, blocked from engagement repos

Cost Control — Four Layers

Layer Mechanism Owner
Account AWS Budgets + SNS alerts per engagement account — monthly cap with 50% / 80% / 100% alerts Engagement Lead
Model Bedrock per-model token quota — Opus limited, Sonnet standard, Haiku high-volume Platform Team
Agent Per-agent cost metrics in Control Plane — dashboard in Agent Factory UI, traces include token count + cost per invocation Engagement Architect
Emergency Kill switch — engagement lead disables a specific agent from Agent Factory UI without redeploy Engagement Lead

Available Models (chosen per-agent in agent.yaml)

Specific models are drawn from two approved lists:

  • TEKsystems-approved model list for internal practice use (sandbox development, playground AgentCore testing, pre-sales demos, POCs).
  • Customer-approved model list for a given engagement's production deployment — subject to the customer's InfoSec and compliance review.

In both contexts, models are selected per-agent in agent.yaml based on three criteria: reasoning complexity (how hard the step is), latency budget (how fast the response needs to be), and cost envelope (how many times the step runs). A complex multi-hop reasoning step uses a higher-capability model; a routine classification or extraction step uses a faster, lower-cost model. The platform consumes Amazon Bedrock by default (HIPAA-eligible), so the available models are whichever Claude or Titan variants are on both the approved list and the Bedrock catalog for the deployment region.

Model choice is configuration, not code: Each agent specifies its model in agent.yaml. Swapping between models — or between providers — is a config change, not a redeploy. This allows cost and quality optimization after an agent is live: observe traces, move predictable steps to a lighter model, keep the hard reasoning step on a higher-capability model, without touching code.
Section 8

Technical Limitations & Design Guardrails

The architecture sits on top of cloud services (AWS AgentCore, Amazon Bedrock), a graph orchestration framework (LangGraph), large language models, and a regulated industry (HIPAA). Each comes with documented limits. The table below lists the constraints that shape the design and the guardrails the accelerator uses to work within them.

Limitation Impact If Ignored How the Design Mitigates It
AgentCore Runtime timeout — 15-minute synchronous invocation limit Long-running tools (bulk claims scans, batch summarization, external-system calls that queue) would time out mid-execution and fail the agent run Any Tool Gateway tool that may exceed 2 min runs async: tool returns a job ID immediately, the agent polls a status endpoint, and the workflow resumes on completion. Short tool calls stay synchronous for latency.
AgentCore Gateway / MCP timeout — 5-minute invocation limit on MCP tools/call Agents using the optional MCP facade hit a tighter ceiling than direct Tool Gateway calls MCP is reserved for short, discovery-style tool calls. Long-running work is routed through the direct Tool Gateway async pattern. Same async job-ID mechanism.
MCP schema constraint — JSON Schema $ref / $defs not supported by AgentCore Gateway target sync Tools registered with referenced schemas fail to sync and become invisible to MCP clients Schemas are fully resolved at registration time inside the Tool Gateway before exposure. No runtime reference lookup.
Bedrock model availability by region Not every Claude model is in every AWS region; a deployment could miss the model an agent is configured for Region selection is part of Phase 1 discovery. Per-agent agent.yaml declares a primary model plus a fallback list. Config loader validates availability at deployment.
LLM context window limits Memory, RAG results, prompt scaffolding, and tool schemas all compete for finite context; long sessions blow through budget silently Context engineering trims and prioritizes before the LLM call: summarization of older episodic memory, configurable top-K for RAG, token budgets per section. Summary memory is explicitly compressive.
LLM non-determinism Same input can produce different outputs; evaluation frameworks that assume exact match are unreliable Evaluation uses semantic matching (LLM-as-judge, structural checks, policy rubrics) — not string equality. Temperature and seed are controlled per agent in agent.yaml.
HIPAA PHI boundary — no PHI may leave the customer's VPC or reach external logs A single PHI leak in a trace, log line, or LLM prompt constitutes a compliance incident A PHI masking layer intercepts on the way out of Container 1 (responses) and on the way into Container 2 (LLM prompts). Bedrock is called through VPC endpoints only. Traces, logs, and audit entries apply the same mask before emission.
LangGraph state serialization — graph state must be JSON-serializable to cross the Container 1 ↔ Container 2 HTTP boundary Pydantic objects, callables, open connections, or tool instances in state would break the runtime split Graph state is constrained to primitives only. Rich objects are rebuilt on each side from IDs. Tool clients and LLM clients are created per-request, not held in state.
HITL pause/resume lives in Container 1 only — graph state cannot round-trip to Container 2 during approval windows Approvals can take hours or days; graph state must survive container restarts A durable HITL store in Container 1 (RDS / DynamoDB) persists the full session snapshot on every gate. Resume on approval restores state exactly; Container 2 is stateless across the gate.
Multi-tenant isolation — shared services (C2 platform services, C3 tool gateway) serve multiple customers or multiple lines of business Data or configuration from one tenant visible to another is an immediate compliance failure Per-request tenant context is required on every inbound call. Memory scopes, tool registry entries, and configuration are all keyed by tenant. No shared global state can be written across tenants.
Tool action serialization — write-class tools (note creation, status updates, external submissions) cannot be parallelized Two writes firing concurrently from the same agent can create conflicting state in downstream systems The Tool Gateway tags tools as read or write. Write tools serialize even when routing matches multiple; read tools run in parallel.
LLM cost per invocation — complex multi-step reasoning can accumulate significant cost per request Without guardrails, a single agent going wrong (reasoning loop, retries) can spike spend Per-agent cost tracking, per-tenant budgets, and a kill switch in the Control Plane. Model selection in agent.yaml allows moving predictable steps to Haiku without redeploy. See Section 7 for full cost-control mechanics.

Each item above is a documented constraint. The accelerator's architecture reflects these constraints directly.

Section 9

Current Prototype — What's Built Today

A working prototype exists and runs the architecture end-to-end. The prototype is not the final accelerator this offering will produce — it is one working reference agent (pre-call assessment for care management) exercising the platform's core patterns against realistic data. What follows is a plain-language description of what the prototype demonstrates today.

Config-Driven by Design

The platform is built so that an agent's behavior is declared, not coded. Every choice that defines how an agent thinks and acts — which reasoning strategy it follows, which memory types it uses, which tools it can call, which prompts it loads, which LLM model it runs on, how it retrieves knowledge — lives in YAML configuration. This is the core design choice that lets operators tune, govern, and evolve agents through a UI rather than through code, and that lets behavior changes ship without redeploying services.

Four Containers, One Coordinated System

The prototype runs as four independent containers that a developer can bring up on a laptop with a single command. Each container has a clear, narrow responsibility: the agent runtime acts as the thin entry point, the orchestration engine runs the reasoning graph, the tool gateway executes tools, and the control plane serves configuration, governance, and observability. The four are wired together by well-defined interfaces, not by shared state, so each can evolve independently and scale differently in production. The same four containers run locally under docker-compose and in production on AWS AgentCore and ECS — the substrate changes, the boundaries do not.

Shared Core Services

A lot of what agents need is not agent-specific — memory, retrieval, human-in-the-loop approval, observability, tool invocation. Rather than rebuilding these in every agent, the prototype places them in a shared core library that every agent links against. Memory comes as four coordinated types — short-term session state, episodic case history, semantic member facts, and running summary — behind a single interface, with swappable storage backends. Retrieval is multi-dimensional: the platform supports different retrieval strategies, different retrieval patterns, and different stages in the agent flow where retrieval runs. HITL is built with an adapter pattern so approval surfaces can plug in over time. Observability wraps every reasoning step, tool call, and memory operation. All of this is available to every agent by default.

Agents Organized as Type × Reasoning Strategy

The prototype organizes agents as a combination of an agent type and a reasoning strategy. Agent types describe what kind of work the agent does — conversational chat, structured summarization, classification and triage, long-running workflow, monitoring, multi-agent coordination. Reasoning strategies describe how the agent thinks — a simple linear plan, a ReAct loop, plan-then-execute, multi-hop reasoning, reflection. These combine: a chat agent using the simple strategy behaves differently from a chat agent using ReAct, and the framework supports every sensible pairing. The prototype ships with two fully wired combinations; the full matrix of combinations is part of what this offering will produce.

Shared Infrastructure for Every Agent

Beyond the core library, the prototype includes shared infrastructure that every agent depends on. The tool gateway is the central enforcement point for secure tool execution — schema validation, per-agent access policy, audit logging, and an optional MCP facade for third-party interoperability. The prompt management service stores prompts with full lifecycle — draft through active through retired — so operators can version and roll back without redeploys. These are deployable services, not libraries; every agent in a deployment uses the same instances.

A Template Offering for New Agents

When a new agent is needed — whether for a new healthcare workflow inside a customer engagement or for a developer building a proof of concept — the prototype does not start from a blank file. The platform ships with a set of templates that capture the patterns above in pre-wired form. An operator or developer picks the template that matches the target agent type and reasoning strategy, adjusts the configuration for the workflow, and gets a working agent. This is how we envision new agents being built across every engagement: through configuration, not greenfield code. The Agent Factory UI is the surface that drives this — an operator scaffolds, configures, and governs agents through a UI, and the platform writes the right templates and YAMLs behind the scenes.

Capability-Level UI for End Users

Agents do not exist in isolation — a care manager, a nurse, a claims reviewer works on a whole domain, not a single agent. The prototype demonstrates a capability-level UI pattern: a domain (for example, care management) gets its own unified interface that spans the agents within it. A nurse uses one console to navigate members, cases, and assessments; multiple agents power different panels inside that console. This is how we envision the end-user experience: capability-first, with agents composed underneath. The prototype includes a working capability UI for care management as evidence of the pattern.

Governance and Compliance in the Platform Design

Healthcare's compliance posture is part of the platform's design. PHI masking runs before any response reaches a UI. HITL gates can be placed at any step an agent takes. Every reasoning step, tool call, and memory operation is traced. Prompts are versioned and lifecycle-managed. Tool access is filtered per-agent. Access control and authorization run before reasoning begins. The prototype exercises all of this together in a single agent flow.

What the Prototype Demonstrates

The reference agent — pre-call assessment for care management — runs end-to-end using every pattern above: the four containers, YAML-driven configuration, the agent-type-by-reasoning-strategy composition, multi-type memory, multi-dimensional retrieval, the tool gateway, prompt management, HITL gates, observability, and a capability-level nurse-facing console. What this offering will produce extends the prototype into a shippable accelerator — hardened backends, complete Infrastructure as Code, production CI/CD, and the full catalogue of agent-type and reasoning-strategy combinations.

Where the Agentic Enterprise Accelerator Adds Over the Common Pattern

The core architecture is the right bet — converged across AWS (AgentCore), LangChain (LangGraph), and PwC's public implementation. The accelerator's value is the layer above that common foundation, which PwC's publicly disclosed materials do not describe.

  • Vertical packaging for healthcare payer — reference agents aligned to payer workflows (UM, CM, claims, prior-auth), HIPAA-eligible compliance evidence, PHI masking, payer-specific source system adapters.
  • Explicit four-container architecture — named C1 (Agent Runtime), C2 (Platform Services), C3 (Tool Gateway), C4 (Control Plane) with defined boundaries and per-container deployment topology.
  • Agent-type × reasoning-strategy overlay matrix — productized combinations (chat × ReAct, workflow × plan-execute, multi-agent × supervisor, summarization × reflection, etc.) each with strategy implementation, prompt scaffolding, and YAML config, discoverable in the Agent Factory UI.
  • Agent Factory UI + template/scaffold pattern — operators scaffold, configure, and govern agents through a UI; platform writes the right templates and YAMLs behind the scenes.
  • Policy-as-code governance from Container 4 — prompts (versioned, lifecycle), HITL gates, memory scopes, RAG configuration (Stage × Source × Extraction), tool allow/deny, and LOB-namespaced tool catalogs — all in YAML, no redeploys.
  • Multi-LOB isolation model — per-LOB AWS accounts, per-LOB Container 1 and Container 3 pairs, shared Container 2 and Container 4 with Agent ID + LOB namespace enforcement.
  • Delivery-integrated — engagement playbook, customer handoff model, repo governance, CODEOWNERS-enforced platform/app split, template + scaffolded-instance pattern for reusable delivery.
