About This Document

This is the Internal Playbook for the Agentic Enterprise Accelerator for Healthcare. It describes how the practice's Platform Team manages the master accelerator repo and how internal developers work with the accelerator — locally for development and on the practice's playground AgentCore for real-runtime testing.

This document is separate from the Customer Delivery Playbook, which describes how delivery teams use the accelerator in customer engagements. Read both together when necessary; they share the same underlying accelerator but address different audiences.

Who this is for: the practice's Platform Team (3–5 people owning the master accelerator), internal developers (anyone in the practice who builds, tests, or demos agents using the accelerator), and solution architects who need to understand how the practice operates the platform internally.
Section 1

Master Repo & Platform Team

The master accelerator repo is the practice's source of truth. Everything — platform code, templates, reference agents, IaC, CI/CD, documentation — lives in one repo. The Platform Team owns it, and every release originates here.

Master Repo Location & Access

  • Repo location: practice's Git organization — e.g. practice/accelerator
  • Platform Team (3–5 people) — the only team with write access to platform directories
  • Practice read-only — every developer in the practice has read access by default; they can clone for sandbox exploration
  • Template repository flag — master repo marked as a template on the Git platform so engagement and sandbox repos can be created from it via "Use this template"

Platform Team Responsibilities

  • Own architecture, design decisions, and runtime-split conventions
  • Approve all changes to packages/platform-core/, platform-services/, services/, services/, templates/, and iac/
  • Publish versioned releases (v1.0, v1.1, …) and release notes
  • Coordinate security scans, SBOM generation, and penetration tests per release
  • Maintain the compliance evidence package (HIPAA controls, architecture security review, HIPAA-eligible service list)
  • Review and merge pull requests from engagement teams that suggest platform improvements

Branch Protection & CODEOWNERS

The master repo uses branch protection and CODEOWNERS to enforce separation between platform-team-owned directories and developer-owned directories.

# Example CODEOWNERS file (simplified) platform-core/ @practice/platform-team platform-services/ @practice/platform-team services/ @practice/platform-team services/ @practice/platform-team templates/ @practice/platform-team iac/ @practice/platform-team @practice/sre agents/ @practice/agent-developers capabilities/ @practice/capability-owners docs/ @practice/platform-team @practice/docs-team

Main branch protection rules on the master repo:

  • No direct pushes to main
  • All changes via PR with at least one required review
  • CI (tests, security scans, eval suite) must pass before merge
  • Platform-directory PRs require Platform Team approval
  • iac/ PRs require additional SRE review

Release Cadence

  • Major versions (v1.0, v2.0, …) — significant architecture or API changes; planned release cadence, typically every 6 months
  • Minor versions (v1.1, v1.2, …) — new backend adapters, new agent types, new reasoning strategies, new compliance patterns; typically monthly
  • Patch versions (v1.1.1, …) — bug fixes and security patches; as needed
  • Each release tagged in Git, release notes published to docs/releases/
  • SBOM and security scan reports versioned with each release
Section 2

Internal Developer Workflows

Internal developers — practice engineers, architects, or capability owners — use the accelerator for exploration, proof-of-concepts, demos, learning, and sales enablement. They do not wait for a customer engagement to get hands-on.

Getting Started — Sandbox Repo

Any developer in the practice can create their own sandbox repo from the master template in seconds:

  1. Go to the master accelerator repo URL in the practice's Git
  2. Click "Use this template"
  3. Name the new repo (e.g. sandbox/jane-doe-chat-agent-poc)
  4. Choose visibility — typically "Internal" (visible to the practice)
  5. Clone the new repo locally

Sandbox repos are disposable. Developers can create as many as they need, discard them when done, and start fresh from the latest master at any time.

Running Locally

The accelerator runs end-to-end on a developer laptop. All four containers come up with a single command:

git clone git@practice:sandbox/jane-doe-chat-agent-poc.git cd jane-doe-chat-agent-poc docker-compose up

After docker-compose up:

  • Agent Factory UI available at http://localhost:3000
  • Reference agents already running at their respective ports (pre-call assessment chat at localhost:5173, etc.)
  • C2, C3, C4 running as docker services with their APIs exposed internally
  • Mock claims / care-management fixtures loaded so agents can call "real" tools against fake payer data
  • Local LLM access — configured to use the practice-approved LLM provider (via practice-issued IAM role) or the practice exploration proxy (practice-issued scoped token) for non-PHI exploration

Scaffolding a New Agent

The typical internal development loop: scaffold a new agent, configure it, test it, tear it down.

  1. Open Agent Factory UI at localhost:3000
  2. Click New Agent
  3. Pick an agent type (chat, summarization, extraction, triage, monitoring, workflow, multi-agent)
  4. Pick a reasoning strategy (simple, ReAct, plan-execute, multi-hop, reflection)
  5. Configure memory types, RAG dimensions, tool allow-list, HITL placement, LLM model
  6. Click Generate — Agent Factory reads from templates/ and scaffolds the agent into agents/<domain>/<agent-name>/
  7. Hot reload picks up the new agent; it becomes callable immediately
  8. Developer iterates — edit YAML, adjust prompts, add tools, rerun

What Developers Can and Cannot Modify Locally

Modify Freely

  • agents/<domain>/<agent>/ — their own agents
  • agents/<domain>/<agent>/overlays/<variant>/config/ — YAML configs
  • capabilities/<domain>/ — capability-level config and UI
  • Prompts via Agent Factory UI

Locally Modifiable, but PR Back to Master

  • packages/platform-core/, platform-services/, services/, services/ — if a developer discovers a platform improvement during exploration, they submit it as a PR to the master repo for Platform Team review, not merged directly into their sandbox
  • templates/ — same pattern; template improvements go through Platform Team

Pulling Master Updates Into a Sandbox

When the Platform Team ships a new release, sandbox developers can pull it if they want the latest:

git remote add upstream git@practice:practice/accelerator.git git fetch upstream git merge upstream/v1.1

Sandbox repos are disposable, so developers also have the option to discard their sandbox and create a fresh one from the latest master template.

Section 3

Deploying to Practice Playground AgentCore

Local docker-compose covers most internal development. When a developer needs to test against real managed services — verify managed runtime behavior, long-running session durability, real LLM latency, real managed memory behavior — they deploy to the practice's playground AgentCore environment.

What the Playground Is

  • One shared AWS sandbox account owned by the practice
  • One shared AgentCore instance in that account — developers deploy their agents as namespaced resources alongside each other
  • Shared backing services — one RDS, one ElastiCache, one OpenSearch — with namespace isolation per developer
  • Per-developer budget cap — AWS Budgets alert at $50/month per developer, auto-disable at $100
  • No PHI, no real customer data — mock fixtures only

Deploying to the Playground

Deployment from a sandbox repo uses the accelerator's standard IaC plus a playground profile:

# From the sandbox repo make deploy-playground AGENT_NAME=jane-doe-chat-agent # What this does: # - Builds the agent's C1 container image # - Pushes to the practice's sandbox ECR # - Registers the agent with the shared playground AgentCore # - Namespaces the resource with the developer's username # - Deploys IaC additions (specific to this agent) into the sandbox account # - Returns the agent endpoint URL for testing

What the Playground Is For

  • Verify the agent runs on AgentCore as expected (not just docker-compose)
  • Test real LLM latency and model behavior against production-like managed infrastructure
  • Exercise memory persistence across sessions using the real managed memory service
  • Validate observability wiring — traces, metrics, logs in the expected shape
  • Demo to internal stakeholders or sales teams in an environment that isn't a laptop
  • Practice deployment and IaC patterns before handling a real customer engagement

What the Playground Is Not For

  • Hosting production customer workloads (those go into customer AWS accounts)
  • PHI or any real patient / member data
  • Long-lived agents that should persist beyond a sandbox lifecycle
  • Large-scale load testing (use a dedicated load-test environment)

Cleaning Up

Playground resources are expected to be short-lived. Developers should clean up after experiments:

make teardown-playground AGENT_NAME=jane-doe-chat-agent

The Platform Team also runs a weekly cleanup job that tears down playground resources older than 30 days.

Section 4

Adding New Platform Features

When the practice decides to extend the platform itself — a new reasoning strategy, a new memory backend, a new RAG pattern, a new compliance control, a new tool adapter — the work happens in the master repo under Platform Team ownership.

Typical Paths

Addition Where the work happens
New reasoning strategy Add implementation in packages/platform-core/ and services/platform-services/src/services/reasoning/strategies/; create overlay variants under templates/overlay-templates/overlays/ for each agent-type × strategy combination; update Agent Factory UI to offer the new option
New memory backend Implement adapter in packages/platform-core/src/platform_core/memory/; add backend option to memory.yaml schema; update Agent Factory UI
New RAG pattern Implement in packages/platform-core/src/platform_core/rag/patterns/; add to Agent Factory UI pattern selector
New agent-type template Add under templates/overlay-templates/overlays/<new-type>_<strategy>/; update Agent Factory UI
New tool adapter Shared adapter → services/tool-policy-gateway/; document in OpenAPI schema
New compliance control Add module to packages/platform-core/ or relevant service; update HIPAA controls matrix; update security evidence package

Feature-Addition Workflow

  1. Design proposal — short RFC or design note reviewed by Platform Team
  2. Implementation on a feature branch of the master repo
  3. CI runs — unit tests, integration tests, security scans, eval regressions
  4. Platform Team review + approval
  5. Merge to main behind the branch protection rules
  6. Ride along in the next versioned release (v1.x) with release notes

Promoting Engagement Improvements Back to Master

Engagement teams often discover platform improvements during real customer delivery — a new backend pattern, a better HITL adapter, a more efficient memory scope strategy. These flow back to the practice:

  • Engagement team submits a PR from their engagement repo to the master repo
  • Platform Team reviews, adapts if necessary to be generic (not customer-specific)
  • Merges into master, ships in the next release
  • Every subsequent engagement benefits
Section 5

LLM Keys & Model Access for Internal Use

Internal developers need LLM access for exploration and testing. The practice manages access centrally so developers never handle raw API keys.

Access Pattern

  • Practice-approved LLM providers — each developer receives a practice-issued IAM role (for primary-provider access) or a scoped proxy token (for non-PHI exploration through the practice-run proxy). No static API keys, no raw vendor keys on developer laptops.
  • Provider selection — providers available for internal use are on the TEKsystems-approved LLM provider list and may change as the list evolves.

Model Availability

Specific models are drawn from the TEKsystems-approved model list. Developers choose per-agent in agent.yaml based on reasoning complexity, latency budget, and cost envelope. Consult the current approved model list before scaffolding a new agent; the practice primary-provider path is used for production-aligned development and the practice proxy is used for non-PHI exploration only.

Cost Controls

  • Per-developer daily spend caps enforced at the practice proxy
  • Cloud account budget alerts on the playground account (50% / 80% / 100% of monthly cap)
  • Per-agent cost metrics visible in the Agent Factory UI (even in local dev)
  • Platform Team reviews monthly spend and adjusts caps as needed
